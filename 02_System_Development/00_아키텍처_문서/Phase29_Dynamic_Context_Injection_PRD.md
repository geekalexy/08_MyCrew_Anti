# Phase 29: Dynamic Context Injection (동적 컨텍스트 참조 아키텍처)

## 1. 개요 (Overview)
- **목적**: 프로젝트 생성 시 설정된 **격리 수준(Isolation Scope - A, B, C)**에 따라, AI 에이전트(Nova, Luca 등)가 작업 수행 시 수집하는 백그라운드 컨텍스트의 반경을 데이터베이스 및 프롬프트 레이어에서 동적으로 조절합니다.
- **핵심 목표**: 타 프로젝트의 유용한 지식(디버깅, 기획 의도 등)은 흡수하되, 에이전트의 현재 작업 목적이 흔들리거나 환각(Hallucination)이 발생하는 현상인 **'컨텍스트 오염(Context Pollution)'**을 원천 차단하는 것입니다.

---

## 2. 격리 수준별 참조 모델 및 데이터 접근 전략

### A. 독립적 공간 (Strict Isolation)
* **데이터 접근**: `WHERE project_id = '현재_프로젝트_ID'`
* **특징**: 현재 동작하는 기본 방식으로, 다른 프로젝트의 데이터가 프롬프트에 절대 섞이지 않습니다.
* **적용 대상**: 코어 시스템 개발, 독립적인 파일럿 테스트, 기밀 프로젝트.

### B. 부분적 독립 공간 (Cross-Project Link / 단방향 참조)
* **데이터 접근**: `WHERE project_id IN ('현재_프로젝트_ID', '참조_프로젝트_ID')`
* **특징 (마케팅팀이 미니앱 개발팀을 참조하는 경우)**:
  * 마케팅 에이전트는 미니앱 개발 프로젝트의 과거 산출물과 고군분투한 로그를 모두 읽어들여 퀄리티 높은 스토리를 뽑아냅니다.
  * **(중요)** 단방향 참조(Read-only)이므로 마케팅팀이 쏟아내는 수많은 카피라이팅 데이터는 개발 프로젝트 쪽에 전혀 저장되지 않으며 영향을 주지 않습니다.

### C. 상호 공유 공간 (Global Knowledge)
* **데이터 접근**: `WHERE isolation_scope = 'global_knowledge'` (공유를 허락한 모든 글로벌 프로젝트)
* **특징**: 조직 전체의 공통 브랜딩 지침, 사내 규정, 범용 프레임워크 에셋 등 모든 크루가 상시로 읽고 업데이트 내용을 서로 공유해야 하는 오픈형 워크스페이스에 적용됩니다.

---

## 3. 참조 대상 데이터의 종류 (Data Types)
에이전트가 타겟 프로젝트에서 가져올 수 있는 데이터는 다음 3가지로 분류됩니다. 무조건 전체를 가져오지 않고, 목적에 맞는 데이터를 필터링합니다.

1. **Tasks & Outcomes (산출물 및 태스크 결과)**
   * **내용**: 완료된 태스크의 결과물 텍스트, 코드 스니펫, 기획안.
   * **활용**: 타 프로젝트의 최종 성과물이나 아키텍처를 파악할 때 우선적으로 로드됩니다.
2. **Logs & Episodes (비하인드/디버깅 에피소드)**
   * **내용**: 에이전트가 작업 중 발생시킨 `THINKING`, `WORKED`, 실패 사례 및 해결 과정.
   * **활용**: 마케팅 팀이 "이 앱이 어떤 에러를 극복하며 탄생했는가"와 같은 스토리텔링 소스를 발굴할 때 핵심이 되는 데이터입니다.
3. **Agent Memory (지식 요약본)**
   * **내용**: 특정 프로젝트를 진행하며 크루가 자체적으로 축적한 '도메인 지식 요약본'.

---

## 4. 컨텍스트 오염 방지 아키텍처 (Anti-Pollution System)
타 프로젝트의 데이터를 무분별하게 읽어오면 AI가 현재 자신의 역할과 남의 데이터를 혼동하는 현상이 발생합니다. 이를 막기 위해 다음 3중 방어막을 구축합니다.

### ① Prompt Partitioning (프롬프트 구획화 및 태깅)
참조 데이터를 LLM에 던질 때, 단순히 이어 붙이지 않고 XML 태그를 이용해 물리적인 구획을 나눕니다.
```xml
<current_project_context>
  <!-- 에이전트가 현재 집중해야 할 메인 목표와 본인 데이터 -->
</current_project_context>

<reference_project_data source="미니앱 개발 프로젝트" mode="read-only">
  <!-- 참조용으로만 읽어야 할 타 부서의 데이터. 절대 이 데이터를 수정하려 들지 말 것 -->
  [로그 1] ...
</reference_project_data>
```

### ② Read-Only 제약 (ORM 레벨 블록)
데이터베이스 쿼리 레이어(`taskRequester.js` 등)에서, 참조 프로젝트의 ID로 `INSERT`, `UPDATE`, `DELETE` 명령이 발생하는 것을 시스템 코드로 원천 차단합니다. AI가 실수로 타 프로젝트의 상태를 변경하라고 지시해도 백엔드가 방어합니다.

### ③ Token Limit & Weight Management (토큰 비율 조절)
* 참조 프로젝트의 데이터가 너무 길어지면 원래 프로젝트의 컨텍스트(초심)가 컨텍스트 윈도우 밖으로 밀려나는 현상이 발생합니다.
* **해결책**: 참조 데이터(Reference)는 전체 토큰 제한의 최대 **30%**까지만 할당되도록 제한합니다. 초과 시 가장 최신 로그나 Task 결과물 순서대로 잘라내어(Truncation) 본래 프로젝트의 컨텍스트(70%)를 보호합니다.

## 5. 개발 순서 계획 (Development Sequence Plan)

본 Phase 29는 데이터베이스 조회 로직부터 프롬프트 주입 로직까지 엔진의 코어 데이터 흐름을 건드리므로 다음의 4단계 순서로 안전하게 개발을 진행합니다.

### 📍 Step 1: Database Layer 동적 쿼리 확장
* **목표**: `isolation_scope`에 따라 읽기 가능한 `project_id` 목록을 반환하는 유틸리티 개발.
* **작업 내역**:
  1. `database.js` 내에 프로젝트의 `isolation_scope` JSON을 파싱하여 접근 가능한 프로젝트 ID 배열(`[target_project_ids]`)을 반환하는 헬퍼 함수 구현.
  2. `getAllTasks`, `getLogs`, `getMemory` 함수들의 쿼리를 수정하여, 단일 `project_id`가 아닌 `WHERE project_id IN (?, ?, ...)` 형태로 다중 조회가 가능하도록 확장.

### 📍 Step 2: Context Gathering Layer (Read-Only 블록)
* **목표**: 에이전트 데몬(`ariDaemon.js`, `taskRequester.js`)이 컨텍스트를 수집할 때 참조 프로젝트 데이터를 구분하고 쓰기를 차단.
* **작업 내역**:
  1. 컨텍스트 수집 시, 가져온 데이터(Task/Log) 배열 요소에 `is_reference: true/false` 플래그를 부착.
  2. 에이전트가 도구를 호출하여 Task 상태를 업데이트하거나 새 로그를 기록할 때(`createKanbanTask`, `appendLog` 등), 타겟 `project_id`가 현재 프로젝트 ID와 일치하는지 검증하는 로직(Read-Only 제약) 추가. 일치하지 않으면 에러 반환.

### 📍 Step 3: Prompt Partitioning & Truncation (어댑터 레이어)
* **목표**: LLM으로 전송되기 직전의 프롬프트 텍스트를 포매팅하고 토큰 한도를 제어.
* **작업 내역**:
  1. `antigravityAdapter.js` (또는 `geminiAdapter.js`)에서 컨텍스트 데이터를 문자열로 조합할 때, `is_reference === true`인 데이터는 `<reference_project_data>` 태그로 별도 구획화.
  2. 참조 데이터의 텍스트 길이가 시스템 프롬프트 전체 토큰 제한의 30%를 초과하는지 계산.
  3. 초과 시, 참조 데이터 중 오래된 순서대로 Truncation(자르기) 수행하여 핵심 컨텍스트 보존.

### 📍 Step 4: 에이전트 시스템 프롬프트 업데이트 및 통합 테스트
* **목표**: 에이전트가 `<reference_project_data>` 태그의 의미를 완벽히 이해하도록 규칙 추가 및 테스트.
* **작업 내역**:
  1. `zeroConfigService.js` 및 공통 시스템 프롬프트 템플릿에 "참조 데이터는 오직 배경 지식으로만 사용하며, 이를 바탕으로 새 프로젝트의 목표에 맞는 결과물만 생성하라"는 지침 강제 주입.
  2. **테스트 시나리오**: '미니앱 개발' 프로젝트(A 설정) 생성 후, '미니앱 마케팅' 프로젝트(B 설정, 미니앱 개발 참조)를 생성하여 마케팅 에이전트가 개발 로그를 바탕으로 블로그 포스팅을 성공적으로(에러 없이, 개발 DB 오염 없이) 작성하는지 검증.

## 6. 프로젝트 설정 변경 (Edit) 및 전환 정책

사용자는 생성된 프로젝트의 격리 수준을 업무 변경에 따라 유연하게 변경할 수 있어야 합니다. 하지만 데이터 오염과 에이전트 환각을 방지하기 위해 특정한 제약이 존재합니다.

### 6.1. 격리 수준 전환 정책 (Transition Rules)
* **A (독립) ➡️ B (참조) 또는 C (공유)**: 🟢 **자유롭게 가능**
  * 이유: 막혀 있던 시야를 넓혀 타 프로젝트 데이터를 참조하는 것은 기존 데이터 무결성에 악영향을 주지 않습니다.
* **B (참조) ➡️ A (독립) 또는 C (공유)**: 🟢 **자유롭게 가능**
  * 이유: 단방향 참조 연결고리를 끊고 다시 폐쇄적으로 가거나, 아예 전역으로 푸는 조치는 안전합니다.
* **C (공유) ➡️ A (독립) 또는 B (참조)**: 🔴 **전환 불가 (시스템 강제)**
  * 이유: C 상태에서 생산된 데이터는 '공용 지식'으로 취급되어 이미 타 프로젝트의 에이전트들이 읽어다 썼을 확률이 높습니다. 갑자기 이를 비공개(A, B)로 전환하면 타 프로젝트 에이전트들이 "의존하던 지식의 증발"을 겪게 되어 심각한 환각(Hallucination) 에러를 일으킵니다.
  * 조치: 한 번 C(Global)로 오픈된 프로젝트는 다시 닫을 수 없도록 UI 상에서 제어합니다.

### 6.2. UI/UX 구현 방안
* **진입점**: 사이드바의 프로젝트 이름 우측 또는 보드 상단 헤더에 **`⚙️ (설정)`** 아이콘 배치.
* **모달 UI**: 기존의 `NewProjectModal` 컴포넌트를 `EditProjectModal`로 재활용(혹은 확장)하여 사용.
  * 타이틀: "프로젝트 설정"
  * 액션 버튼: "변경사항 저장"
* **상태 제어 로직**: 현재 프로젝트의 `isolation_scope`가 `global_knowledge(C)`일 경우, 라디오 버튼 조작을 비활성화(Disabled) 시키고 툴팁으로 전환 불가 사유 안내.
