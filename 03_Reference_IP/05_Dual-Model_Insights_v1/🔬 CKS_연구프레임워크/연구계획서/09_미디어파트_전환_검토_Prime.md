## 🚨 핵심 문제: 이 변경은 CKS 실험의 Mirror Design을 파괴합니다

### 현재 확정된 크루 배치 (v3.1)

```
Team A: NOVA(Flash) + REX/LILY(Sonnet) + OLLIE(Opus)
Team B: LUMI(Flash) + PICO(Sonnet)       + SAGE(Opus)
         ↕ 대칭 ↕                           ↕ 대칭 ↕
```

### 이 문서가 제안하는 변경

```
영상 크루 2명: Sonnet → Gemini Pro로 교체
```

이것을 적용하면:

```
Team A: NOVA(Flash) + LILY(Pro) + OLLIE(Opus)
Team B: LUMI(Flash) + PICO(Pro) + SAGE(Opus)
```

**양 팀 대칭이 유지되므로 실험 공정성 자체는 깨지지 않습니다.** 그러나 이 문서에는 이 맥락이 전혀 반영되어 있지 않으며, CKS 실험에 미치는 영향을 분석하지 않았습니다.

## 📋 문서 품질 이슈 — 5건

### ⛔ 이슈 1: 사실 오류 — 모델명 혼동 (L5)

> *"제안: 동일 인원(2명)으로 **재미나이(Claude Opus)** 모델을"*

**재미나이(Gemini)는 Google 모델이고, Claude Opus는 Anthropic 모델입니다.** 이 문장은 두 모델을 혼동하고 있습니다. 이 정도의 기본 사실 오류가 있으면 문서 전체의 신뢰도가 떨어집니다.

### ⛔ 이슈 2: 비용 수치가 비현실적 (L46-49)

```
Pro 모델 비용:    0.0004 USD/토큰 → $800/month
Sonnet 모델 비용: 0.0005 USD/토큰 → $1,000/month
```

**이 수치는 현실과 크게 다릅니다.** 실제 가격:

| 모델 | Input 토큰 | Output 토큰 | 비고 |
|------|-----------|------------|------|
| Gemini 3.1 Pro | ~$0.00125/1K | ~$0.005/1K | Google 공식 |
| Claude Sonnet 4.6 | ~$0.003/1K | ~$0.015/1K | Anthropic 공식 |

- Gemini Pro가 Sonnet보다 **저렴한 것은 맞지만**, 토큰당 단가가 문서에 적힌 것과 **자릿수가 다릅니다**
- 2M 토큰에 $800이면 토큰당 $0.0004인데, 이는 어떤 모델 가격과도 맞지 않음
- **정확한 비용 시뮬레이션을 다시 해야** 합니다

### ⚠️ 이슈 3: "영상 품질 1.5배 향상" 근거 부재 (L14)

> *"Gemini 3.1 Pro는 최신 이미지·영상 모델로, 기존 Sonnet 대비 1.5배 이상 해상도·프레임 레이트 향상"*

이 주장에 **근거(출처, 벤치마크)가 없습니다.** 실제로:

- Gemini Pro와 Sonnet은 모두 **텍스트 → 코드를 생성하는 LLM**이지, 직접 영상을 렌더링하는 모델이 아닙니다
- 영상 품질은 **Remotion 렌더러의 설정**(해상도, FPS)에 의존하지, LLM 모델 선택과는 무관합니다
- LLM이 영향을 미치는 것은 **Remotion 코드의 품질**(장면 구성, 애니메이션 로직, 색상 선택 등)이며, 이는 "해상도 1.5배"로 표현할 수 없습니다

### ⚠️ 이슈 4: CKS 실험 영향 분석 없음

이 문서는 **제품 운영 관점**에서만 작성되었고, CKS 실험에 미치는 영향을 전혀 분석하지 않았습니다:

- 모델 변경 시 **TEI 베이스라인이 달라짐** — 기존 Sprint 데이터와 비교 불가
- KSI-S 측정에서 **모델 특성이 바뀌면** 의미론적 유사도의 기준선이 바뀜
- **실험 도중 모델을 바꾸면 데이터 오염** — 변경 전/후를 분리해야 함

### ⚠️ 이슈 5: Anti-Bridge 아키텍처와의 관계 미언급

현재 설계에서 영상 크루(PICO/LILY)가 Sonnet인 이유 중 하나는 **코딩 능력이 중요하기 때문**(대표님 원문)입니다. Gemini Pro가 Remotion 코드 생성에서 Sonnet만큼 우수한지 **검증이 선행되어야** 합니다.

## ✅ 변경 자체에 대한 Prime의 견해

### 방향은 맞습니다

영상 크루를 **Sonnet → Gemini Pro**로 변경하면:

| 장점 | 상세 |
|------|------|
| **비용 절감** | Gemini Pro < Sonnet (실제로 맞음, 단 정확한 수치 재계산 필요) |
| **API 일원화** | MyCrew 서버가 이미 Gemini API만 가동 → **Anti-Bridge 없이 직접 호출 가능** |
| **속도 향상** | Gemini Pro 응답 속도 ≤ Sonnet (일반적으로) |
| **Mirror Design 유지** | 양 팀 모두 Pro로 변경하면 대칭 유지 |

### 가장 큰 이득: Anti-Bridge 우회 없이 완전 자동화

```
Before: PICO(Sonnet) → Anti-Bridge → 대표님 수동 트리거 → 응답 (반자동)
After:  PICO(Pro)    → geminiAdapter → 직접 API 호출 (완전 자동)
```

**이것이 이 변경의 진짜 가치입니다.** 문서에서 이 점이 충분히 강조되지 않았습니다.

단, OLLIE/SAGE(Opus)는 여전히 Anti-Bridge가 필요합니다 — Opus API가 없으므로.

### 변경을 진행하되, 다음 조건을 충족해야 합니다:

| # | 조건 | 이유 |
|---|------|------|
| **1** | 문서의 사실 오류 수정 (모델명, 비용 수치) | 의사결정 문서의 기본 신뢰도 |
| **2** | Gemini Pro vs Sonnet의 **Remotion 코드 품질 벤치마크** | "코딩 능력이 중요"한 역할이므로 |
| **3** | CKS 실험에서 **변경 전/후 Sprint를 분리** | 데이터 오염 방지 |
| **4** | 양 팀 동시 변경 (Mirror Design 유지) | 실험 공정성 |
| **5** | 비용 시뮬레이션 재계산 (실제 API 가격 기반) | 예산 의사결정 정확도 |

### Remotion 코드 품질 벤치마크 방법

```
테스트 과제: "소시안 Plan C 30초 소개 릴스" 동일 프롬프트
   ├── Gemini Pro로 Remotion 코드 생성 → 렌더링 → 품질 평가
   └── Sonnet으로 Remotion 코드 생성 → 렌더링 → 품질 평가

평가 항목:
   1. 코드 실행 성공률 (에러 없이 빌드되는가)
   2. 장면 구성 창의성 (1~5점)
   3. 애니메이션 복잡도 (요소 수, 전환 효과)
   4. 코드 길이 / 토큰 효율
```

> Week 1 사전 검증에 이 벤치마크를 포함하면 별도 일정 추가 없이 가능합니다.
