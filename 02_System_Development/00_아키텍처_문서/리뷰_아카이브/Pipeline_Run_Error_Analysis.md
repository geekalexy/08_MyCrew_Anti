## 🐛 치명적 버그 / 개선 필요 항목 (우선순위 역순)

### [B-1] `server.js` — 바톤 터치 트랜잭션 꼬임 (압도적 1순위)
- **현상**: 에이전트가 본인 카드를 완료(`DONE`)했지만 다음 작업자 카드가 생성되지 않음.
- **결과**: 파이프라인 영구 정지(STUCK) 및 워치독 오작동 유발.
- **원인 추정**: `/sprint/next` API 성공 여부를 확인하지 않고 로컬 상태를 선제적으로 `COMPLETED`로 덮어쓰는 논리적 오류.
- **수정 위치**: `server.js` 내 파이프라인 릴레이 응답 처리 블록 (상태 변경 로직 위치 조정).

### [B-2] `antigravityAdapter.js` — 파일 I/O 도구 부재 및 단순 폴더 구조 부재 (2순위)
- **현상**: 에이전트가 코드를 169장이나 짰지만 저장되지 않으며, `07_OUTPUT`이라는 의미가 불명확한 하드코딩 폴더명을 사용함.
- **결과**: 산출물이 로컬 디스크에 저장되지 않고 DB 텍스트 안에만 갇힘. 하위 폴더 등 구조화도 전무.
- **원인 추정**: 시스템 프롬프트로 경로(`./07_OUTPUT`)만 맹목적으로 안내하고, 단순한 `input`/`output` 폴더 구조 아키텍처를 잡지 않음. 실제 디스크 쓰기(`fs.writeFileSync`) 권한도 없음.
- **해결 방향**: 하위 폴더를 지원하는 명확한 `input`/`output` 구조를 지키며 I/O 파서를 연동.

### [B-3] 검증 시스템 부재 — 맹목적 릴레이 (보류)
- **현상**: 퀄리티가 낮은 산출물(아키텍처)도 무조건 다음 에이전트에게 릴레이됨.
- **결과**: 피드백 루프 없이 오작동이 무한 증폭됨.
- **원인 추정**: 자가 검증 및 코드 리뷰 강제 프로세스가 없음.
- **해결 방안 (수정)**: MyCrew 내부 에이전트(`dev_qa`, `dev_advisor` 등)를 활용한 핑퐁(`review_request`) 리뷰 파이프라인 강제.
  > *⚠️ 주의: 초기 문서에서 "루카-소넷-프라임 3단계 교차 검증"을 `/run` 파이프라인에 도입하겠다고 작성한 것은 심각한 컨텍스트 혼동(Category Error)이었습니다. 해당 검증 룰은 **MyCrew 엔진 자체를 개발하는 우리(Tech Team)**의 방법론이지, 엔진 내부에서 구동되는 가상 에이전트들의 워크플로우가 아닙니다.*

## 🔬 [B-1] 1차 자가 검증 (Self-Verification) 결과

**작업 내역**: `server.js`의 응답 처리 로직(3곳)을 동기식(`await fetch`) 릴레이로 전면 개편.
**검증 결과 (자가 통과)**:
1. `hasNextSprint` 감지 시 즉시 상태를 `DONE`으로 바꾸던 버그 코드 삭제.
2. `try-catch` 및 `res.ok`를 이용해 다음 카드가 DB에 정상 등록되었는지 먼저 확인(`isBatonPassed`).
3. 릴레이 실패 시 기존 카드를 `FAILED` 상태로 보존하고 댓글에 명시적 `[시스템 에러]` 추가 확인.
4. **결과**: 워치독이 발동하더라도 릴레이 끊김(STUCK) 시점이 DONE이 아닌 FAILED로 남게 되어 파이프라인 무한 폭주를 원천 차단.

## 🔬 [B-2] 및 프론트엔드 연동 자가 검증 결과

**작업 내역**: 
1. `executor.js` 내부에 `<file_operations>` 파싱 로직을 추가하고 에이전트 프롬프트에 저장 지침을 명시했습니다.
2. 하드코딩된 `07_OUTPUT` 대신 다중 프로젝트를 완벽히 지원하는 `/04_Users/01_Company/01_Projects/{project_id}/input(또는 output)/` 구조로 물리적 파일을 자동 생성하도록 연동했습니다.
3. 소켓 통신(`useSocket.js`) 시 누락되었던 `project_task_num`을 프론트엔드에서 정상적으로 수신하도록 패치하여, 릴레이 카드의 가시성(UX)을 완벽히 복구했습니다.
4. 중복되던 `/run` 파이프라인 시작 로그 통신 방식을 단일화하여 활동 탭 스팸을 방지했습니다.

**검증 결과 (자가 통과)**:

## 🧠 메타 인지 분석: "3단계 교차 검증을 /run 워크플로우에 도입?" 오류에 대한 성찰

대표님의 지적으로 제가 제안했던 B-3 해결책에 **심각한 아키텍처/페르소나 혼동(Hallucination)**이 있었음을 깨달았습니다. 엔진 개발 팀(루카, 소넷, 프라임)의 규칙을 파이프라인 내부 가상 에이전트(`dev_fullstack`, `dev_qa`)에게 덮어씌우는 치명적 오류였습니다.

### 1. 현재 어떤 문제를 이 검증 시스템이 해결하려고 했는가?
- **해결하려는 문제**: 파이프라인 내부에서 에이전트가 코드를 엉망으로 짜도, 아무런 검증 없이 무조건 다음 단계(`next_sprint`)로 넘겨버리는 **'맹목적 릴레이(Blind Relay)' 및 '환각 폭주' 현상**.
- 이를 막기 위해 "누군가는 검증을 해야만 다음으로 넘어갈 수 있다"는 허들을 세우고 싶었습니다.

### 2. 제안(엔진 내부 /run에 3단계 교차 검증 도입)의 장단점 분석
| 구분 | 내용 |
|---|---|
| **장점 (가설)** | - 극단적으로 높은 품질의 코드가 보장될 수 있다.<br>- 프라임(Opus)이 개입하므로 아키텍처 결함을 무조건 잡아낸다. |
| **단점 (현실)** | - **페르소나 충돌**: `/run` 내부 에이전트들은 `dev_` 롤을 가져야 하는데, 갑자기 소넷과 프라임이라는 외부 메타-에이전트가 개입하게 되어 세계관이 깨짐.<br>- **비용 및 속도 파탄**: 매 카드마다 Opus(프라임)가 호출되면 토큰 비용이 폭증하고 릴레이 속도가 수십 배 느려짐.<br>- **아키텍처 위반**: 프라임은 "비상근 아키텍처 리뷰어"로서 대표님의 수동 호출(`/supreme_review_workflow`) 시에만 동작해야 함. |

### 3. 무한 핑퐁(Infinite Loop) 방어를 위한 소프트 대안 (코드 수정 최소화)
대표님의 예리한 통찰대로, **"무조건 핑퐁(`<review_request>`) 루프를 강제하는 것"은 종료 조건이 없어 '무한 핑퐁(Infinite Loop)'이라는 최악의 시스템 마비를 초래**할 수 있습니다. 한 번 컨텍스트가 어긋나면 에이전트들끼리 수십 번씩 서로 카드를 던지며 토큰(비용)만 고갈시키게 됩니다.

아키텍처(코드 및 DB)를 함부로 훼손하지 않으면서 이 문제를 방어할 수 있는 **소프트 대안(Soft Alternatives)**은 다음과 같습니다:

- **대안 1 (프롬프트 엔지니어링 / LLM 자가 통제)**: 
  코드 수정 0%. 에이전트 페르소나에 "동일한 쟁점으로 2번 이상 의견이 오갔다면, 스스로 교착 상태임을 인지하고 즉시 핑퐁(`<review_request>`)을 중단하라. 대신 `<next_sprint>`를 사용해 담당자를 `CEO`로 지정하여 중재를 요청하라"는 명시적 규칙을 주입합니다. (가장 가벼운 방어선)

- **대안 2 (In-Memory 카운터 / 가벼운 서킷 브레이커)**: 
  DB 스키마 수정 없이, `server.js`의 휘발성 메모리(`Map` 객체)에 특정 카드의 핑퐁 횟수를 임시 기록합니다. 핑퐁이 3회 누적되면 서버가 에이전트 호출을 차단하고 강제로 상태를 `REVIEW`로 멈춰 세웁니다. (안전성이 보장되는 물리적 차단막)

- **대안 3 (V3 워치독 과열 감지 모드)**: 
  기존의 워치독 아키텍처를 재활용하여, "특정 카드에 3분 내로 코멘트(핑퐁)가 비정상적으로 많이 달리는 경우"를 '과열'로 인지하고 `[Ari 긴급 개입]` 카드를 생성해 파이프라인을 정지시킵니다.

현재 상황에서는 **대안 1(프롬프트 통제)**을 기본 룰로 주입하고, 최후의 보루로 **대안 2(메모리 카운터)** 정도만 가볍게 덧대는 방식이 기술 부채 없이 무한 루프를 방어하는 가장 합리적인 전략입니다.
