# Phase 28: 전사적 아키텍처 영향도 분석 (Impact Analysis)

우측 패널(타임라인/채팅)을 프로젝트 단위로 분리하는 기획을 시스템 전체(루카 엔진, 대시보드, 팀/프로필 페이지)로 확장했을 때 추가로 조치되어야 하는 핵심 아키텍처 변경점들을 깊이 있게 도출한 보고서입니다.

---

## 1. 🧠 AI 엔진 및 지식 베이스 (Ari Engine & Knowledge)

현재 MyCrew의 AI 자가학습(Self-Learning)과 지식 저장 체계는 '글로벌 물리 파일(SKILL.md)' 구조입니다. 프로젝트가 분리될 경우 가장 먼저 충돌하는 영역입니다.

### 1.1. SKILL.md의 가상화 및 네임스페이스 분리
*   **문제점**: A 프로젝트(마케팅)에서 실패한 패턴이 카테고리(예: `02_marketing/SKILL.md`)에 전역 기록되어, B 프로젝트(B2B 영업)의 에이전트 행동 지침까지 오염시킵니다.
*   **조치 사항**: 물리적인 `SKILL.md`를 프로젝트 단위로 복제(`SKILL_{project_id}.md`)하거나, **데이터베이스(DB) 기반의 동적 프롬프트 시스템**으로 마이그레이션해야 합니다.
*   **자가학습(Auto-Digest) 격리**: `executor.js`의 자가 진화 로직이 글로벌이 아닌, 현재 실행 중인 태스크의 `project_id` 스코프 내에서만 규칙(Rule)을 업데이트하도록 변경해야 합니다.

### 1.2. M-FDS (MyCrew Folder & Document Standard) 참조 권한
*   **조치 사항**: RAG(검색 증강 생성) 또는 Context Injector 구동 시, 해당 에이전트가 현재 활성화된 프로젝트의 M-FDS 폴더와 '전역 공유 폴더(Shared)'에만 접근할 수 있도록 경로 접근 제어(Access Control) 로직이 추가되어야 합니다.

---

## 2. 👥 팀 & 에이전트 프로필 페이지 (Team & Profile Pages)

에이전트가 여러 프로젝트에 겹치기 투입될 수 있는 매트릭스(Matrix) 조직 구조를 지원해야 합니다.

### 2.1. 익명 역할 기반(Anonymous Role-Based) 크루 생성 및 닉네임 객체화
*   **초기 제공 형태 (이름 없는 에이전트)**: 팀 빌딩 시 'Luna', 'Ollie' 같은 고정된 이름을 처음부터 부여하지 않습니다. 오직 **역할명과 탑재 모델**(예: `카피라이터_Sonnet 4.6`)로만 초기 세팅되며, 좌측 메뉴(LNB)에도 역할명만 직관적으로 표기됩니다.
*   **닉네임 부여 및 객체화 (DB 설계)**: 사용자가 애착을 가질 수 있도록 언제든 '닉네임(예: 루나)'을 붙일 수 있습니다.
*   **스키마 구조**: DB 설계 시 에이전트를 고정된 ID가 아닌, `[project_id, role_name, base_model, nickname]` 형태의 **복합 객체(Composite Entity)**로 인식하도록 구성해야 합니다. 이렇게 하면 하나의 프로젝트 안에서 이 속성 조합 자체가 고유한 에이전트 자아(Persona)로 완벽히 동작하게 됩니다.

### 2.2. 평가 지표(CKS Metrics)의 다차원화
*   **조치 사항**: 에이전트 개인 프로필 페이지의 성과 그래프(TEI, KSI 등)가 "전체 누적"뿐만 아니라 "프로젝트별 기여도" 탭으로 나뉘어 렌더링되어야 합니다. 이를 위해 `CksMetrics` 테이블 쿼리 시 `project_id` 기반 GROUP BY가 필요합니다.

## 3. 🖥️ 메인 대시보드 및 칸반 보드 (Dashboard & Kanban)

우측 패널만이 아니라, 메인 화면의 워크스페이스 구조 자체가 다중 프로젝트를 수용하도록 뼈대가 바뀌어야 합니다.

### 3.1. 전역 네비게이션(LNB) 및 컨텍스트 스위칭
*   **조치 사항**: 좌측 사이드바 또는 상단 헤더에 **'프로젝트 스위처(Project Switcher)'** 컴포넌트가 신설되어야 합니다.
*   **스왑 범위**: 프로젝트 변경 시 우측 패널만 바뀌는 것이 아니라, 중앙의 **칸반 보드(Task Board)** 자체도 해당 프로젝트의 태스크로 완전히 교체(Hydration)되어야 합니다.

### 3.2. 크로스 프로젝트 알림 (Cross-Project Notifications)
*   **문제점**: 대표님이 A 프로젝트 화면을 보고 있을 때, B 프로젝트에서 딥워크가 끝났을 경우 이를 인지할 방법이 없습니다.
*   **조치 사항**: 소켓을 2-Track으로 연결해야 합니다.
    *   `project_A_room`: 상세 화면, 실시간 로그 스트림 수신.
    *   `global_notification_room`: 뱃지 카운트용. "B 프로젝트에서 Task #12 완료됨" 형태의 토스트 알림 수신 기능.

## 4. 💸 리소스 및 시스템 운영 (Resource & Ops)

### 4.1. 비용(Token)의 프로젝트별 청구서 (Billing)
*   **조치 사항**: 여러 프로젝트를 동시에 운영할 경우, API 토큰 소비량(Gemini Pro, Opus 등)을 프로젝트 단위로 집계하는 기능이 필요합니다. 어떤 프로젝트가 가장 비효율적으로 리소스를 소모하는지 파악하는 '프로젝트 비용 대시보드' 기획이 수반되어야 합니다.

### 4.2. 프로젝트 아카이빙 (Cold Storage)
*   **조치 사항**: 완료된 프로젝트를 Soft Delete 하는 것을 넘어, 완전히 '아카이브(Archive)' 처리하여 읽기 전용(Read-only) 상태로 동결하는 기능이 필요합니다. 아카이브된 프로젝트는 칸반 이동이나 채팅이 불가능해지며, 템플릿(지식 베이스)용으로만 활용됩니다.
