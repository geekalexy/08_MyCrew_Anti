# SaaS 확장 검토안

## 1. 서비스 모델

내 PC가 서버 역할, 웹사이트를 통해 고객에게 서비스 제공.

### 아키텍처

```
고객 (모바일/PC 브라우저)
    ↓
웹사이트 (대시보드)          ← 고객이 쉽게 쓰는 인터페이스
    ↓ 서류 업로드 / 지시 / 승인
내 PC (서버)
    ├── Paperclip (에이전트 엔진)
    ├── Bridge (텔레그램 연동)
    ├── Claude Code (C1/C2)
    └── 고객 파일 저장소
```

### 고객 경험

1. 웹사이트 가입 → 회사 생성 (내가 초기세팅)
2. 텔레그램 봇 연결 (/start)
3. 웹에서 서류 업로드 → 서버에 저장
4. 텔레그램으로 지시/승인
5. 에이전트가 서류 참조하며 작업

---

## 2. Paperclip 분양 방식

Paperclip에서 Add company로 여러 회사를 만들어두고 분양.

```
내 PC (Paperclip 서버)
├── 페이퍼클립소시안 (내 회사)
├── 고객A 회사 (초기세팅 완료) — 분양
├── 고객B 회사 (초기세팅 완료) — 분양
└── ...
```

- 내가 에이전트 구성 + 초기 이슈 세팅
- 고객에게 companyId + 텔레그램 봇 제공
- 고객은 Paperclip 설치 없이 텔레그램만으로 사용

---

## 3. 장점

| 항목 | 장점 |
|---|---|
| 고객 진입장벽 | 웹사이트 + 텔레그램만 — 설치 없음 |
| 서류 접근 | 웹에서 업로드 → 서버 저장 → 에이전트 참조 |
| 관리 | 한 곳에서 전체 운영 |
| 과금 | 월 구독 모델 가능 |

---

## 4. 현실적 과제

| 과제 | 대응 |
|---|---|
| PC 1대 한계 | 5~8개 회사까지. 이후 PC 추가 or 클라우드 |
| 웹사이트 개발 | FastAPI + React 또는 Notion/Google Forms로 MVP |
| 파일 저장 | 업로드 → 프로젝트 폴더, C2가 참조 |
| 보안 | 고객간 데이터 격리, Tailscale 접근 제어 |
| PC 꺼짐 | 항시 가동 필요 — Mac mini 서버 추천 |

## 5. MVP 로드맵

```
1단계 (현재): 내 회사(소시안) 운영 안정화
2단계: 지인 1명에게 분양 테스트 (Add company + 텔레그램)
3단계: 간단 웹 대시보드 (파일 업로드 + 상태 확인)
4단계: Mac mini 서버 전환 → 다수 고객 운영
5단계: 클라우드 이전 (AWS/GCP) → SaaS 확장
```

### 2단계 선행 조건
- 왕짱플젝 운영 안정화
- 텔레그램 봇 Chat ID 분리 (A안) 검증
- Bootstrap 스크립트 완성

## 6. 클라우드 전환 상세 검토

### cmux 플랫폼 의존성

| 구성요소 | macOS | Linux (AWS) | 대체 필요 |
|---|---|---|---|
| cmux | ✅ macOS 전용 | ❌ 불가 | tmux로 교체 |
| Claude Code | ✅ | ✅ | 그대로 |
| Paperclip | ✅ | ✅ (Node.js) | 그대로 |
| Bridge 스크립트 | ✅ | ✅ (Python) | cmux→tmux 명령 교체 |
| Tailscale | ✅ | ✅ | 그대로 |

### AWS 이전 선택지

| 방식 | 월비용 | 비고 |
|---|---|---|
| AWS Mac 인스턴스 | ~15~25만원 | cmux 그대로, 비쌈 |
| AWS Linux + tmux | ~4~10만원 | cmux→tmux 리팩토링 1~2주 |
| Mac mini (물리) | ~2만원 (전기) | 초기비용 ~100만원 |
| 하이브리드 (Mac mini + AWS) | ~3~5만원 | Mac mini 고장 리스크 |

## 7. 최종 운영 전략 (대표 피드백 반영, 2026-04-04)

### 핵심 판단
- **Mac mini 구매는 초기비용 대비 사업성 낮음** → 물리 서버 투자 보류
- **3~4명까지는 지인 위주로 내 PC에서 관리** → 초기 비용 0원
- **유료 전환 의사 확인 후 클라우드로 전체 이전** → 수요 검증 후 투자

### 확정 로드맵

```
1단계 (현재): 내 회사(소시안) 운영 안정화 — 비용 0원
    ↓
2단계: 지인 3~4명에게 무료 분양 — 내 PC에서 운영
    · Add company로 회사 세팅 → 텔레그램 봇 연결
    · 피드백 수집, 서비스 검증
    ↓
3단계: 유료 전환 의사 확인
    · "돈 내고 계속 쓰겠다" → 클라우드 이전 결정
    · 안 되면 → 사업 피봇 or 중단
    ↓
4단계: 클라우드 서버(AWS Linux)로 전체 이전
    · cmux → tmux 리팩토링
    · 웹 대시보드 구축
    · 월 과금 모델 적용
    ↓
5단계: SaaS 확장 — 다수 고객 운영
```

### 단계별 비용

| 단계 | 월 비용 | 고객 수 |
|---|---|---|
| 1~2단계 | 0원 (내 PC) | 1~4명 |
| 4단계 | ~4~10만원 (AWS Linux) | 5~15명 |
| 5단계 | ~10~30만원 (AWS 확장) | 15~50명 |

### Mac mini를 건너뛰는 이유
- 초기비용 ~100만원 → 수요 미검증 상태에서 리스크
- AWS Linux가 월 4만원~로 시작 가능 → 수요 확인 후 점진 확장
- cmux→tmux 리팩토링은 필요하지만 1~2주로 해결 가능

## 8. 클라우드 서버 브라우저 접근성 검토

Paperclip 조작 빈도가 높아 브라우저 접근성이 중요. cmux는 macOS 전용이므로 클라우드 대체재 필요.

### cmux 대체재 비교

| 도구 | 터미널 | 브라우저 내장 | CLI 브라우저 조작 | 클라우드 | 적합도 |
|---|---|---|---|---|---|
| **cmux** | ✅ | ✅ | ✅ | ❌ macOS만 | 현재 사용 |
| **tmux + Playwright** | ✅ | 헤드리스 | ✅ (코드) | ✅ | ⚠️ 브라우저 눈에 안 보임 |
| **code-server (VS Code 웹)** | ✅ | 프리뷰만 | ❌ | ✅ | ⚠️ |
| **noVNC + cmux** | ✅ | ✅ | ✅ | ✅ | ✅ 유력 |

### A안: AWS Mac + noVNC (리팩토링 0)

```
AWS Mac 인스턴스 (macOS 가상머신)
  └── cmux 그대로 실행
  └── noVNC로 웹에서 원격 접속
  └── 고객은 웹에서 cmux 화면을 그대로 봄
```

- 장점: 리팩토링 0, 지금 코드 그대로
- 단점: 월 ~15~25만원

### B안: AWS Linux + Playwright + 웹 대시보드 (저렴)

```
AWS Linux
  ├── tmux (터미널)
  ├── Playwright (헤드리스 브라우저 — Paperclip 조작)
  ├── Bridge (텔레그램)
  └── 웹 대시보드 (고객용 Paperclip 상태 화면)
```

- 장점: 월 ~4~10만원
- 단점: Playwright 리팩토링 + 웹 대시보드 개발 (1~2주)

### 비용 비교 (12개월 기준)

| 방식 | 월 비용 | 12개월 총비용 | 브라우저 접근성 |
|---|---|---|---|
| AWS Mac + cmux | ~20만원 | ~240만원 | 완벽 (cmux 그대로) |
| AWS Linux + Playwright | ~7만원 | ~84만원 + 개발비 | 코드로만 조작 |

### 단계별 추천 (수정)

```
고객 3~4명 (2단계): 내 PC — 비용 0원
유료 전환 시 (4단계): AWS Linux + Playwright — 자체 웹 UI 제공
고객 확장 (5단계): 서버 스케일링
```

## 9. 제품화 방향 — 자체 웹 인터페이스 (대표 피드백 반영, 2026-04-04)

### 핵심 판단
- 클라우드 전환 시 고객은 **우리 브랜드의 웹 대시보드**를 사용
- Paperclip은 **백엔드 엔진** 역할 — 고객에게 노출되지 않음
- Playwright가 헤드리스(눈에 안 보임)여도 **전혀 문제 없음**
- 따라서 AWS Mac(비쌈) 없이 **AWS Linux + Playwright로 직행 가능**

### 고객이 보는 화면

```
현재 (개발 단계):
  cmux → Paperclip 브라우저 → 직접 조작

미래 (제품):
  고객 → 자체 웹 대시보드 → 에이전트 팀 현황
         ↕ (백엔드)
       Playwright → Paperclip API/UI 조작
```

### 자체 웹 대시보드 주요 기능

| 기능 | 설명 |
|---|---|
| 에이전트 현황 | CEO, CMO 등 누가 무엇을 하고 있는지 |
| 이슈 목록/상세 | Paperclip 이슈를 보기 좋게 포장 |
| 댓글/지시 | 웹에서 바로 에이전트에게 지시 |
| 승인 처리 | 승인 요청 목록 + 원클릭 승인 |
| 서류 업로드 | 파일 올리면 에이전트가 참조 |
| 알림 | 텔레그램 + 웹 푸시 알림 |

### 최종 아키텍처 (AWS Linux)

```
고객 (웹/모바일/텔레그램)
    ↓
자체 웹 대시보드 (FastAPI + React)
    ↓
Playwright (헤드리스) → Paperclip 조작
    ↓
Paperclip (백엔드 엔진) → 에이전트 실행
    ↓
Claude Code (C2) → 분석/요약/지시 처리
    ↓
Bridge → 텔레그램 알림
```

### AWS Mac 불필요 판단 근거
- cmux browser 기능 → Playwright가 동등 이상 성능으로 대체
- 고객은 Paperclip UI를 직접 볼 일이 없음
- 자체 웹 UI를 씌우므로 헤드리스 브라우저로 충분
- 월 비용 ~7만원 vs ~20만원 → **연간 ~156만원 절감**

### 수정된 로드맵

```
1단계 (현재): 내 PC에서 소시안 운영 — cmux + Paperclip 직접 사용
2단계: 지인 3~4명 무료 분양 — 내 PC, 텔레그램만 제공
3단계: 유료 전환 확인 → AWS Linux 전환 결정
4단계: AWS Linux + Playwright + 자체 웹 대시보드 구축
5단계: SaaS 확장 — 다수 고객, 서버 스케일링
```

## 10. 서비스 모델 및 가격 구조 (대표 피드백 반영, 2026-04-05)

### 두 가지 서비스 모델

| 모델 | 설명 | 대상 |
|---|---|---|
| **모델 1: 설치형** | 사용자 Mac에 직접 설치, 무료 오픈소스 | Mac 있는 얼리어답터/개발자 |
| **모델 2: 대행형** | 내 서버에서 대신 운영, 텔레그램만 제공 | Mac 없는 전문직/대표 |

### 가격 구조

| 티어 | 대상 | 월 가격 | 제공 내용 |
|---|---|---|---|
| **Free** | Mac 있는 사용자 | 무료 | 설치형 오픈소스 (bootstrap.sh) |
| **Pro** | Mac 없는 사용자 (대행) | 월 80~90만원 | AI 회사 세팅 + 운영대행 + 텔레그램 제어 |
| **Enterprise** | 맞춤 대규모 구성 | 협의 | 전담 세팅 + 컨설팅 + SLA |

### Pro 티어 수익 분석

| 항목 | 금액 |
|---|---|
| 고객 월 구독료 | 80~90만원 |
| 서버 비용 (AWS Linux) | ~7만원 |
| Claude API 비용 | ~5~20만원 (사용량 비례) |
| **마진** | **~53~78만원/고객** |

### Pro 대상 고객
- IT에 약한 쇼핑몰 사장님
- 나이 든 회사 대표
- Mac이 없는 전문직 (변호사, 회계사 등)
- "설치고 뭐고 모르겠고, 텔레그램만 쓸래" 수요

> 클라우드 확장(4~5단계) 이후에도 Pro 대행은 **프리미엄 서비스로 영구 유지**.
> 설치형 무료 사용자 → 대행 전환 수요도 기대.

### 수정된 로드맵

```
1단계 (현재): 모델 1 — 설치형, 내 PC에서 소시안 운영
2단계: 모델 1 — 지인 3~4명 무료 분양 (Mac 있는 사람)
3단계: 모델 2 — Mac 없는 사용자 대행 시작 (내 PC, Pro 월 80~90만원)
4단계: AWS Linux + tmux + Playwright로 대행 확장
5단계: 자체 웹 UI + SaaS (Free + Pro 병행)
```
