## 1. 개요 및 배경 (Background)
최근 `/run` 명령어를 통한 자율 주행 릴레이 파이프라인 테스트 과정에서, 에이전트들이 구현 결과를 제대로 확인하지 않은 채 성급하게 다음 태스크로 핑퐁(Ping-pong)을 진행하다가 **169개의 환각(Hallucination) 태스크가 폭증하는 치명적인 사태**가 발생했습니다. (상세 내용은 `Pipeline_Run_Error_Analysis.md` 참조)

"빠르다고 좋은 것이 아니다. 실수를 방지하기 위해 팀제로 운영해야 한다"는 대표님(CEO)의 핵심 피드백에 따라, 한 번의 구현을 맹신하지 않고 단계별로 코드를 점검하는 **3단계 교차 검증 시스템(3-Stage Verification System)**을 파이프라인 워크플로우에 공식 도입하고자 합니다.

## 2. 3단계 교차 검증 시스템 설계 (Architecture Proposal)

### 1차 검증: 자가 검증 시스템 (Self-Verification Loop)
- **주체**: 코드 구현 담당 에이전트 (예: `dev_fullstack` 또는 `Luca`)
- **방법**: 아키텍처 설계 후 본 코드를 작성하기 전에, 혹은 작성 직후에 **테스트/검증 스크립트(Unit Test, Console Log 등)를 먼저 작성**하여 로컬 샌드박스에서 돌려봄.
- **목적**: 문법 오류(Syntax Error), 경로 문제, 오타 등 사소하지만 치명적인 버그를 자신이 배포하기 전에 스스로 걸러냄.

### 2차 검증: 동료 코드 리뷰 (Peer Review)
- **주체**: 동료 에이전트 (예: `dev_senior`, `Sonnet`)
- **방법**: 1차 자가 검증이 끝난 코드를 `<review_request>` 태스크로 전환하여 동료에게 전달. 코드의 아키텍처 정합성, 보안 로직, 중복 코드 여부를 크로스 체크함.
- **목적**: 팀 단위의 집단 지성을 활용하여 개인의 좁은 시야(환각)를 탈피하고, 서로의 실수를 보완.

### 3차 검증: 프라임 최종 리뷰 (Supreme Review)
- **주체**: Prime (Supreme Advisor)
- **방법**: 기능이 완벽하게 구현되고 2차 검증까지 마친 PR(Pull Request) 및 최종 산출물에 대해 프라임 어드바이저가 구조적/전략적 최종 승인을 진행.
- **목적**: 코드가 단순 동작을 넘어, MyCrew 전체의 시스템 설계 원칙(M-FDS, STRICT 정책 등)에 부합하는지 보증.

## 3. 구현 방안 (Action Plan)
1. **프롬프트 갱신 (`antigravityAdapter.js`)**: 에이전트 시스템 프롬프트에 "코드를 작성한 후 반드시 테스트 코드를 돌려 자가 검증을 수행하라"는 지시어 추가.
2. **워크플로우 체인 제어 (`executor.js`)**: 에이전트가 `<next_sprint>`를 호출하기 전에, 자신이 작성한 결과물을 검증하는 단계를 필수로 거치도록 파이프라인 제어.
3. **오류 감지 및 정지 연동**: 자가 검증 중 `Error`가 2회 이상 발생 시, 파이프라인을 일시 정지(Pause)하고 인간(CEO)의 판단을 기다리도록 워치독 강화.

## 4. 리뷰 요청 사항 (Request for Comments)
- **To Sonnet**: 동료 리뷰어로서 2차 검증(Peer Review) 단계에서 에이전트 간 어떤 데이터를 주고받아야 컨텍스트 유실 없이 효과적인 리뷰가 가능할지 피드백 부탁드립니다.
- **To Prime**: 3단계 교차 검증 시스템이 자율 주행의 속도를 저하시킬 수 있다는 우려가 있습니다. '안정성'과 '속도'의 트레이드오프에서 적절한 임계점을 어디로 설정해야 할지 고견을 요청합니다.
