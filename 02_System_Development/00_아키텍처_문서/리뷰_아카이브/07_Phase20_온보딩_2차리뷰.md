## 🟡 신규 발견 1: AnthropicAdapter의 Eager Initialization 문제

기획서에서 "모든 통신 모듈이 `keyProvider.getKey()`를 바라보도록 동적 마이그레이션"이라고 했는데, 현재 AnthropicAdapter를 보면:

```javascript
// anthropicAdapter.js L3-11 — 현재 코드
class AnthropicAdapter {
    constructor() {
        if (!process.env.ANTHROPIC_API_KEY) {
            console.warn("⚠️ ANTHROPIC_API_KEY가 등록되지 않았습니다.");
        }
        this.client = new Anthropic({
            apiKey: process.env.ANTHROPIC_API_KEY,  // ← 생성자에서 즉시 초기화
        });
    }
}
export default new AnthropicAdapter();  // ← 모듈 로드 시점에 인스턴스 생성
```

**문제:** 이 파일이 `import`되는 순간 생성자가 실행되어 `process.env.ANTHROPIC_API_KEY`를 읽습니다. KeyProvider로 마이그레이션하면 **`await keyProvider.getKey()`는 비동기**인데, 생성자는 `async`일 수 없습니다.

### Prime 권고: Lazy Init 패턴 (GeminiAdapter와 동일하게)

```javascript
// anthropicAdapter.js — KeyProvider 호환 리팩토링
import keyProvider from '../tools/keyProvider.js';

class AnthropicAdapter {
    constructor() {
        this.client = null;  // Lazy: 첫 호출 시 초기화
    }

    async _ensureClient() {
        if (this.client) return;
        const apiKey = await keyProvider.getKey('ANTHROPIC_API_KEY');
        if (!apiKey) {
            throw new Error("ANTHROPIC_API_KEY를 찾을 수 없습니다.");
        }
        const Anthropic = (await import('@anthropic-ai/sdk')).default;
        this.client = new Anthropic({ apiKey });
    }

    async generateResponse(systemPrompt, userPrompt) {
        await this._ensureClient();
        // ... 기존 로직
    }
}

export default new AnthropicAdapter();
```

**GeminiAdapter는 이미 `_initClient()` 패턴을 갖고 있으므로 비교적 쉽게 마이그레이션 가능**하지만, AnthropicAdapter는 생성자 구조 자체를 바꿔야 합니다. 기획서에 이 차이를 명시하세요.

## 🟢 잘 설계된 부분

### ✅ KeyProvider 3단계 우선순위

```
1순위: 메모리 캐시 → 2순위: DB(온보딩) → 3순위: .env(레거시)
```

이 설계의 핵심 가치:
- **하위 호환성 100%**: 기존 `.env` 사용자는 아무것도 바꾸지 않아도 됨
- **온보딩 사용자 즉시 반영**: DB에 키 저장 즉시 캐시 갱신 → 서버 재시작 불필요
- **테스트 용이성**: 유닛 테스트에서 캐시만 주입하면 외부 의존 없이 테스트 가능

### ✅ 스마트 분기 로직

```javascript
!hasEnvKey && !hasDbKey && !onboardingCompleted
```

이 3중 조건이 커버하는 시나리오:

| `.env` 키 | DB 키 | 온보딩 완료 | 위저드 표시 |
|:---|:---|:---|:---|
| ✅ 있음 | — | — | ❌ 표시 안 함 (기존 사용자) |
| ❌ 없음 | ✅ 있음 | — | ❌ 표시 안 함 (이전에 온보딩 완료) |
| ❌ 없음 | ❌ 없음 | ✅ 완료 | ❌ 표시 안 함 (스킵한 사용자) |
| ❌ 없음 | ❌ 없음 | ❌ 미완료 | ✅ **위저드 표시** |

4개 시나리오 모두 올바르게 처리됩니다.

팀 유형 선택 → AgentSkill 자동 인서트를 별도 모듈로 분리한 것은 SRP(단일 책임 원칙)에 부합합니다. 다만 구체적인 매핑 예시가 없으므로 참고용 레퍼런스를 제안합니다:

```javascript
// teamActivator.js 예시 구현
const TEAM_PRESETS = {
  marketing: {
    ari:  ['routing'],
    nova: ['marketing', 'content'],
    lumi: ['design'],
    pico: ['content'],
    ollie: ['analysis'],
  },
  development: {
    ari:  ['routing', 'analysis'],
    nova: ['marketing'],
    lumi: ['design'],
    pico: ['content'],
    ollie: ['analysis', 'research'],
  },
  fullstack: {
    ari:  ['routing'],
    nova: ['marketing', 'content'],
    lumi: ['design'],
    pico: ['content'],
    ollie: ['analysis', 'research'],
  },
};

export async function activateTeam(teamType, dbManager) {
  const preset = TEAM_PRESETS[teamType];
  if (!preset) throw new Error(`Unknown team type: ${teamType}`);
  
  for (const [agentId, skills] of Object.entries(preset)) {
    for (const skillId of skills) {
      await dbManager.upsertAgentSkill(agentId, skillId, true);
    }
  }
}
```
