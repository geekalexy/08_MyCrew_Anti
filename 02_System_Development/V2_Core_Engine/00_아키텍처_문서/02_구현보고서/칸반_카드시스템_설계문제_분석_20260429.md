### 🔴 문제 1: `@멘션`과 `Task Assignee 변경`의 개념 미분리

**현상:**
CEO가 `@nova 작업 시작해줘`라고 댓글을 남기면, 시스템이 이를
- 에이전트 호출 (알림)
- 담당자(Assignee) 변경
- 작업 지시

세 가지를 **동시에 수행하는 단일 인터페이스**로 처리함.

**설계 문제:**
`@멘션`의 본래 역할은 코멘트 알림(Notification)이며, Task 할당은 별도의 명시적 액션이어야 함. 이 두 개념이 혼용되면 어떤 액션이 실제로 실행되는지 명세가 불명확해지고, 실행 결과 검증이 불가능해짐.

**개선 방향:**
- `@멘션` → 알림/호출 (코멘트 수신 알림)
- `/assign @nova` 또는 Assignee 필드 직접 변경 → 담당자 변경
- 두 개념을 명령 문법 수준에서 분리

### 🔴 문제 2: 에이전트 응답과 실제 카드 상태 변경의 검증 루프 부재

**현상:**
- `ari`가 `"Nova에게 할당했습니다"` 응답 → 카드 Assignee는 **미할당 유지**
- `nova`가 `"담당자가 [nova]로 변경되었습니다"` 로그 출력 → 카드 실제 상태와 불일치

**설계 문제:**
에이전트의 텍스트 응답이 실제 카드 상태 변경(DB Mutation)의 성공을 보장하지 않음.
에이전트가 "변경했습니다"라고 응답하면 시스템이 완료로 인식하는 구조이나, 실제 변경 성공 여부를 확인하는 피드백 루프가 없음.

```
현재 구조:  에이전트 응답 텍스트 = 완료 처리
올바른 구조: 에이전트 액션 실행 → 카드 상태 변경 성공/실패 확인 → 결과 기반 응답 생성
```

**개선 방향:**
- 에이전트가 카드 상태를 변경하는 API 호출 후, 응답 코드(성공/실패)를 확인하고 그 결과를 응답 텍스트에 반영
- 변경 성공 시: `"Nova에게 할당 완료 ✓ (카드 확인됨)"`
- 변경 실패 시: `"할당 시도 실패 — 재시도 또는 수동 처리 필요"`

### 🔴 문제 3: Task 실패 후 자동 에스컬레이션 로직 부재

**현상:**
- 02:52 Hard Timeout 발동 → Task #3 실패 처리
- 이후 아무런 자동 알림 없이 3시간 21분 방치
- CEO가 수동으로 발견하여 재지시

**설계 문제:**
Task가 실패 상태로 전환되는 이벤트를 감지하지만, 이후 처리는 전적으로 수동에 의존.
실패 감지 → 후속 처리 간의 자동화 연결이 없음.

**개선 방향:**
- Task 실패 전환 시 즉시 담당 에이전트 / 담당자에게 알림 발송
- 일정 시간(예: 30분) 이내 수동 복구가 없으면 자동 재할당 또는 에스컬레이션 트리거
- 실패 이력을 카드에 배지(Badge) 형태로 표시하여 대시보드에서 육안으로 식별 가능하게

### 🟠 문제 4: Task 이동 실패 시 원인 미노출 (블랙박스 에러 처리)

**현상:**
- `"Task #3 이동 실패 — 이전 상태로 복원"` 메시지만 출력
- 실패 원인(권한 부족? 상태 조건 불충족? 타임아웃 충돌?) 미표시

**설계 문제:**
이동 실패를 감지하고 롤백은 수행하지만, 왜 실패했는지 사용자나 에이전트가 알 수 없음.
원인을 모르면 재시도 방법도 알 수 없어 동일한 실패가 반복될 수 있음.

**개선 방향:**
- 실패 시 에러 코드 + 사람이 읽을 수 있는 원인 메시지 표시
  - 예: `"이동 실패: Task가 '실패' 상태일 때는 '진행 중'으로 직접 이동 불가 — '대기 중'으로 먼저 복구 필요"`
- 이동 가능한 다음 상태 힌트 제공

### 🟠 문제 5: 멘션 라우팅의 불투명성 — 멘션 대상 ≠ 실제 실행 에이전트

**현상:**
- CEO → `@nova` 멘션 → Discussion에 `nova → ARI(위임)`으로 표시
- CEO → `@ari` 멘션 → Discussion에 `ari → ARI(위임)`으로 표시
- 어떤 에이전트를 멘션해도 실제 처리는 `ARI(위임)` 레이어를 경유

**설계 문제:**
CEO 입장에서 `@nova`를 호출했지만, 실제로 누가 작업을 담당·실행하는지 카드 UI에서 추적 불가.
`멘션 대상 에이전트 ≠ 실제 실행 에이전트`인 구조가 사용자에게 투명하게 노출되지 않아 혼란 유발.

**개선 방향:**
- 멘션된 에이전트와 실제 실행 에이전트가 다를 경우, 위임 관계를 명시적으로 표시
  - 예: `nova (→ ARI 위임으로 실행 중)`
- 카드 Assignee 필드에 현재 실제 실행 중인 에이전트를 실시간 표시

### 🟠 문제 6: 에이전트 간 재할당 시 작업 컨텍스트 이전 체계 부재

**현상:**
- `nova → ari → nova` 순서로 담당자가 여러 번 변경
- 각 에이전트는 이전 에이전트의 진행 상태, 실패 원인, 작업 결과를 파악하지 못한 채 동일한 첫 응답을 반복

**설계 문제:**
재할당은 Assignee 필드 값만 교체하는 단순 변경으로 처리됨.
이전 담당자의 작업 내용, 완료율, 실패 원인이 다음 담당 에이전트에게 자동으로 전달되는 Handoff 프로토콜이 없음.

**개선 방향:**
- 재할당 이벤트 발생 시 자동으로 **작업 상태 스냅샷** 생성하여 카드에 첨부
  - 포함 내용: 이전 담당자, 작업 진행률, 완료된 작업, 실패 원인, 다음 필요 액션
- 새 담당자(에이전트)에게 스냅샷을 컨텍스트로 주입하여 인수인계 없이 이어서 진행 가능하게

### 🔴 문제 7: 카드 상태를 코멘트 없이 직접 변경하는 수단 없음

**현상:**
- 카드 상태를 변경하려면 하단 코멘트 입력창에 텍스트를 작성하고 "전송 및 반영"을 눌러야만 가능
- 더 이상 에이전트에게 전달할 내용이 없고, 단순히 상태만 `실패 → 대기 중`으로 바꾸고 싶어도 반드시 코멘트를 함께 입력해야 함
- 빈 코멘트 전송이 허용되는지도 UI에서 불명확

**설계 문제:**
상태 변경이 "코멘트 전송"의 부산물로 처리되는 구조. 상태 전환과 커뮤니케이션이 하나의 액션으로 묶여 있어, 상태만 바꾸고 싶은 경우에도 코멘트가 강제됨.

**개선 방향:**
- 하단 상태 드롭다운을 코멘트 전송과 완전히 독립적인 인라인 편집 컨트롤로 분리
- 상태 변경 전용 버튼(예: `[상태 변경]`) 또는 드롭다운 선택 즉시 저장 방식 도입
- 코멘트 없이 상태만 변경할 때는 시스템 이벤트 댓글로 자동 기록

### 🔴 문제 8: 배지(Badge) 노출 규칙 불명확 — `ANTI-BRIDGE-NOVA`

**현상:**
- 카드 상단에 `ANTI-BRIDGE-NOVA` 배지가 표시되나, 다음이 전혀 불명확함:
  - 이 배지가 언제 생성되는가 (태스크 생성 시? 에이전트 실행 시?)
  - 무엇을 의미하는가 (현재 실행 에이전트? 마지막 실행 에이전트? 라우팅 경로?)
  - 클릭하면 어떤 동작이 발생하는가
  - 태스크 실패 후에도 배지가 남아있는 이유
- CEO 입장에서 이 배지가 "현재 실행 중"인지, "마지막으로 시도한"인지 알 수 없음

**설계 문제:**
내부 어댑터 라우팅 정보(anti-bridge-nova)가 그대로 UI에 노출되어 있으나, 사용자에게 유용한 맥락 정보로 정제되지 않은 상태. 내부 구현 용어가 CEO 대시보드에 그대로 렌더링됨.

**개선 방향:**
- 배지를 "현재 실행 에이전트" / "마지막 실행 에이전트" 두 상태로 명확히 구분
- 표시명을 사용자 친화적으로 변환: `ANTI-BRIDGE-NOVA` → `NOVA (Anti Bridge)`
- 배지에 툴팁 추가: "이 에이전트가 마지막으로 처리 시도 (20:52, 실패)"
- 실패 상태에서는 배지 색상을 빨간색으로 변경하여 상태 연동

### 🟠 문제 9: 에이전트 위임 승인의 액션 경로 불명확

**현상 (Task #117 실제 사례):**
```
1. CEO → PICO 할당
2. PICO → CEO: "Nova에게 할당하는 것이 적합합니다. 진행할까요?"
3. CEO → 어떤 액션을 해야 할지 모름:
   - 코멘트에 "그래"만 입력하면 되는가?
   - NOVA에게 직접 재할당해야 하는가?
   - 결국 NOVA로 수동 재할당 + "그래 진행해줘" 코멘트를 모두 수행
```

**설계 문제:**
에이전트가 위임 제안을 할 때, CEO가 승인만 하면 시스템이 자동으로 재할당을 수행해야 하는지,
아니면 CEO가 수동으로 재할당까지 해야 하는지 명세가 없음.
"그래"라는 코멘트가 위임 승인 명령으로 해석되는지, 단순 확인인지 시스템이 구분하지 못함.

**개선 방향:**
- 에이전트가 위임 제안 시 인라인 승인 버튼 표시:
  ```
  [✅ Nova에게 위임 승인]  [❌ 직접 처리 계속]
  ```
- 버튼 클릭으로 Assignee 변경 + 에이전트 실행이 자동 연계
- 텍스트 코멘트 방식은 보조 수단으로만 유지

### 🟠 문제 10: 시스템 이벤트 댓글 과다 분절 — 1개 액션 = 3개 댓글

**현상 (Task #117 실제 사례):**
```
system → CEO  20:42  👤 담당자 변경: PICO → NOVA
system → CEO  20:42  🔴 우선순위 변경: normal → high  
system → CEO  20:42  📋 상태 변경: 승인 대기 → 진행 중
```
동시에 발생한 3가지 변경사항이 각각 개별 시스템 댓글로 분리 출력됨.
Discussion 카운터가 올라가고 채팅 스크롤이 늘어나지만 실질 정보량은 1개.

**설계 문제:**
상태 변경 이벤트를 개별 메시지로 emit하여 Discussion이 시스템 노이즈로 오염됨.
실제 에이전트·CEO 간 커뮤니케이션이 시스템 메시지에 묻혀 맥락 파악이 어려워짐.

**개선 방향:**
동일 트랜잭션 내 복수 변경사항을 단일 시스템 댓글로 묶어 표시:
```
system → CEO  20:42
🔄 카드 업데이트
  • 담당자: PICO → NOVA
  • 우선순위: 보통 → 높음
  • 상태: 승인 대기 → 진행 중
```

### 🟠 문제 11: 에이전트 내부 태그 및 도구 코드(TOOL_CODE) UI 노출

**현상:**
- Discussion에 에이전트 응답 내 JSON 코드블록이 그대로 출력:
  ```json
  { "tool_code": "createKanbanTask(assignee='Nova', title='...' }  
  ```
- 타임라인에 내부 처리 태그 그대로 노출:
  - `[QUICK_CHAT]`, `[CONTENT]`, `[TOOL_CODE]`, `[WORKED]`, `(anti-bridge-nova)` 등
- PICO 응답의 마크다운(`**Nova**`, `**담당자:**`)이 타임라인에서 미렌더링, 원문 기호 그대로 표시

**설계 문제:**
에이전트 응답을 UI에 렌더링하기 전에 내부 태그·도구 호출 블록을 필터링하는 전처리 레이어가 없음.
사용자가 접해야 할 정보(에이전트의 업무 결과)와 개발 디버그용 정보(TOOL_CODE JSON)가 혼재.

**개선 방향:**
- 렌더링 파이프라인에 출력 필터 레이어 추가:
  - `[TOOL_CODE]` 블록 → 숨김 (또는 접힌 아코디언으로 선택 노출)
  - `[QUICK_CHAT]`, `[CONTENT]`, `[WORKED]` 태그 → 제거 후 정제된 텍스트만 출력
  - `(anti-bridge-nova)` 등 내부 어댑터 식별자 → 제거 또는 변환
- 타임라인에서 마크다운 렌더링 활성화
- TOOL_CODE는 로그 드로어(Log Drawer) 전용으로 이동

## 카드 상세 페이지 UI 개선점

> **분석 근거:** Task #3 카드 상세 페이지 캡처 기반  
> **범위:** 카드 UI 레이아웃, 상태 표시, 입력 UX, 메타데이터 구조

---

### 🔴 UI-1: 상태 불일치 — "실패" 배지 vs "진행 중" 필드 동시 표시

**현상:**
- 카드 상단 배지: `실패` (빨간색)
- 카드 하단 상태 필드: `진행 중 (In Progress)`

두 상태가 동시에 존재하며 서로 모순됨. 사용자가 어느 상태를 신뢰해야 하는지 알 수 없음.

**개선 방향:**
- 상태가 "실패"로 전환되면 하단 상태 필드도 자동 동기화
- 카드의 상태 소스를 단일화하여 두 곳에서 별도 관리되는 구조 제거
- 상단 배지와 하단 필드를 같은 상태 값을 바라보는 단일 소스로 통합

---

### 🔴 UI-2: 실패 상태에서 다음 액션 가이드(CTA) 없음

**현상:**
Task가 "실패"임에도 카드 UI에 어떤 행동 유도 버튼도 없음. 사용자가 무엇을 해야 할지 스스로 판단해야 하며, 채팅 멘션으로 수동 지시하는 수밖에 없음.

**개선 방향:**
실패 상태 진입 시 카드 상단 또는 하단에 인라인 액션 버튼 표시:
```
[🔄 재시도]  [👤 재할당]  [📦 보관]
```
각 버튼에 간단한 설명 툴팁 제공. 재시도 시 이전 실패 원인을 컨텍스트로 포함.

---

### 🔴 UI-3: 실패 원인이 카드 UI에 요약 표시되지 않음

**현상:**
Hard Timeout, 실패 원인이 Discussion 채팅 로그 안에만 묻혀있음. 카드 자체에서는 "실패" 배지 외에 원인 정보가 전혀 없어, 사용자가 채팅 로그를 스크롤하며 직접 찾아야 함.

**개선 방향:**
상단 실패 배지 아래 또는 메타데이터 영역에 실패 요약 인라인 표시:
```
⚠️ 실패 원인: Hard Timeout (10분 초과 — 02:52)
→ 수동 복구 또는 재시도 필요
```
클릭 시 상세 로그로 이동하는 링크 제공.

---

### 🟠 UI-4: 에이전트 라우팅 정보의 위치와 의미 불명확

**현상:**
`DISCUSSION (7)` 헤더 우측에 `ARI(위임) → NOVA`가 작게 표시됨.
- 현재 라우팅인지, 마지막으로 시도된 라우팅인지 불명확
- Discussion 헤더에 위치해 "채팅 설정"처럼 읽힘
- 실시간으로 업데이트되는지 여부도 불명확

**개선 방향:**
별도 "현재 실행 에이전트" 필드로 분리하여 Assignee 옆에 배치:
```
담당자: 미할당   |   현재 실행: ARI → NOVA (마지막 시도: 09:18)
```
실행 중일 때는 스피너, 완료/실패 시 아이콘으로 상태 시각화.

---

### 🟠 UI-5: Assignee 필드 직접 편집 불가 (추정)

**현상:**
하단 상태 바의 `미할당` 필드가 클릭/드롭다운으로 직접 변경 가능한지 UI에서 명확하지 않음. 현재는 채팅 멘션을 통해서만 담당자를 변경할 수 있는 구조로 보임.

**개선 방향:**
- `미할당` 클릭 시 에이전트/팀원 목록 팝업으로 직접 변경 가능하게
- 변경 즉시 카드 상태에 반영 + 해당 에이전트에게 자동 알림 발송
- 이것이 문제 1번(멘션 = 할당 혼용)의 UI 레벨 해결책과 연결됨

---

### 🟠 UI-6: 하단 상태 바의 필드 레이블 없음

**현상:**
```
진행 중 (In Progress)  |  미할당  |  보통
```
세 개의 값이 나란히 있으나, 각각이 **상태 / 담당자 / 우선순위**를 의미한다는 레이블이 없음. 처음 보는 사용자에게 의미 파악이 어려움.

**개선 방향:**
아이콘 또는 레이블로 각 필드 명칭 명시:
```
📋 상태: 진행 중   |   👤 담당자: 미할당   |   🔺 우선순위: 보통
```

---

### 🟠 UI-7: 입력창 모드(코멘트 vs 에이전트 지시) 구분 없음

**현상:**
하단 입력창 placeholder: `"업무 지시나 피드백을 전달하세요..."`
- 일반 코멘트인지, 에이전트 지시인지 구분 없음
- @멘션이 필요한지 필요없는지 가이드 없음
- Task가 실패 상태일 때도 동일한 입력창 표시 (문맥 미반영)

**개선 방향:**
입력창 상단에 모드 탭 구분:
```
[💬 코멘트]  [🤖 에이전트 지시]
```
- 에이전트 지시 모드 선택 시 @멘션 자동완성 및 대상 에이전트 목록 표시
- 실패 상태일 때 입력창 placeholder를 문맥에 맞게 변경: `"복구 지시 또는 재할당 요청을 입력하세요..."`

---

### 🟡 UI-8: Discussion 카운터가 메시지 유형을 구분하지 않음

**현상:**
`DISCUSSION (7)` — 총 7개의 메시지이지만 시스템 알림, 에이전트 내부 로그, 실제 사람 코멘트가 동일하게 카운트됨. 실제 사람 간 커뮤니케이션이 몇 건인지 파악 불가.

**개선 방향:**
```
DISCUSSION   💬 사람 4   🤖 에이전트 3
```
또는 메시지 유형 필터 탭 제공:
```
[전체]  [사람]  [에이전트]  [시스템]
```

---

### 🟡 UI-9: 카드 메타데이터 요약 영역 부재

**현상:**
카드 상세 페이지에 생성일, 마감일, 태그, 연관 카드 등의 메타데이터를 한눈에 볼 수 있는 영역이 없음. 제목 아래와 Discussion 채팅 사이에 정보 공백이 있음.

**개선 방향:**
카드 제목 바로 아래에 메타데이터 요약 바 추가:
```
생성: 26.04.29 02:37  |  마감: 미설정  |  태그: #콘텐츠 #인스타  |  연관 카드: —
```
각 항목은 클릭 시 인라인 편집 가능하게.

---

### 🔴 UI-10: 편집 모드 — 상태 변경 옵션 없음 + 불필요한 모델 선택 필드

**현상:**
- 카드 편집 모드 진입 시 상태(Status)를 변경할 수 있는 옵션이 없음
- 편집 모드에 "모델 선택" 필드가 존재하나, 편집 시점에 모델을 바꾸는 유스케이스가 없음
  (모델 선택은 에이전트 설정 단계에서 완료되어야 할 사항)

**개선 방향:**
- 편집 모드에 상태 변경 드롭다운 추가 (제목·내용 편집과 동일 편집 컨텍스트 안에서 처리)
- 모델 선택 필드는 편집 모드에서 제거 → 에이전트 설정 탭으로 이전

---

### 🟠 UI-11: 타임라인에서 에이전트 응답 마크다운 미렌더링

**현상:**
- 타임라인에 표시되는 에이전트(PICO, NOVA 등) 응답에서 마크다운 기호가 원문으로 노출:
  - `**Nova**` → **Nova** 로 렌더링되어야 함
  - `**담당자:**`, `**업무명:**` 등 모두 미렌더링
- Discussion 뷰와 타임라인 뷰 간 렌더링 방식이 불일치

**개선 방향:**
- 타임라인 뷰에도 마크다운 렌더러 적용 (현재 Discussion에만 적용된 것으로 보임)
- 단, 마크다운 렌더링 적용 시 내부 태그([TOOL_CODE] 등) 필터링도 함께 처리 (문제 11 연계)

---

### 🟠 UI-12: "전송 및 반영" 버튼 — 동작 의미 모호

**현상:**
- 하단 입력창 우측 버튼: `전송 및 반영`
- "전송"(코멘트 전달)과 "반영"(카드 상태 변경 적용)이 동시에 일어난다는 의미인지,
  또는 단순히 "보내기"를 강조한 표현인지 모호
- 상태 드롭다운에서 상태를 바꾸지 않았을 때도 동일한 버튼이 표시됨

**개선 방향:**
- 상태 변경이 없을 때: `💬 코멘트 전송`
- 상태 변경이 있을 때: `💬 전송 + 상태 반영`
- 버튼 레이블을 현재 액션 내용에 따라 동적으로 변경하여 의도 명확화

---

### 🟠 UI-13: 카드 제목 말줄임 — 전체 내용 확인 수단 없음

**현상:**
- 카드 상세 페이지 상단 제목이 한 줄 말줄임(`…`)으로 잘림
  - 예: `06_소시안자료/컨텐츠/블로그 폴더에 아티클을 참조해서 소시안이 매월 주기적으...`
- 제목 전체를 확인하려면 별도 수단(편집 모드 진입 등)이 필요

**개선 방향:**
- 제목 클릭 시 펼침(expand) 또는 툴팁으로 전체 제목 표시
- 편집 없이 읽기 전용으로 제목 전체를 볼 수 있는 인터랙션 제공

---

### 🟡 UI-14: Hard Timeout 시스템 메시지 — 복구 지시 없음

**현상:**
```
system → CEO  20:52
🚨 작업 시간이 10분을 초과하여 시스템 쉴드(Hard Timeout)가 발동했습니다.
    백그라운드 에이전트 실행을 강제 중단했습니다. 수동 복구를 진행해주세요.
```
"수동 복구를 진행해주세요"라고 안내하지만 어떻게 복구하는지 방법이 없음.
사용자가 다음 액션을 스스로 파악해야 함.

**개선 방향:**
시스템 Timeout 메시지 하단에 인라인 복구 액션 제시:
```
[🔄 동일 에이전트로 재시도]  [👤 다른 에이전트에 재할당]  [📦 태스크 보관]
```
재시도 시 이전 실패 컨텍스트를 자동으로 포함.

---

### 🟡 UI-15: 하단 상태 바 — 하단 입력창과 역할 중복

**현상:**
하단 입력창 바로 위에 `진행 전 (To Do) | NOVA | 보통` 상태 바가 있으나:
- 이 상태 바에서 값을 변경한 후 코멘트 없이 "반영"만 하는 것이 가능한지 불명확
- 실패 상태인 카드에서 상태 바가 `진행 전 (To Do)`을 표시하는 것 자체가 UI-1 문제와 연계
- 입력창과 상태 바가 하나의 복합 폼으로 묶여있는지, 별도 컨트롤인지 구분이 없음

**개선 방향:**
- 상태 바를 카드 상단 메타데이터 영역으로 이전, 코멘트 입력창과 물리적으로 분리
- 상태 바의 각 필드는 독립적으로 저장 가능하도록 설계 (코멘트 의존성 제거)
