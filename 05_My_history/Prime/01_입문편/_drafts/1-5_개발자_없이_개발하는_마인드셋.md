# 개발자 없이 개발하는 마인드셋

> 비개발자의 CTO 도전기 — AI 에이전트 팀으로 SaaS 만들기 **1-5편**

---

## 🎬 에피소드: "그래서 당신은 개발자예요?"

### 대답할 수 없는 질문

어느 모임에서 누군가가 물었다.

> "SaaS를 만들고 있다고요? 개발자시군요."

나는 잠깐 멈칫했다. "아니요"라고 하면 "그럼 누가 만들어요?"라고 물을 것이고, "AI가 만들어요"라고 하면 "장난감 아니에요?"라는 반응이 올 게 뻔했다.

지금도 이 질문에 깔끔하게 답하기 어렵다. 나는 코드를 짜지 않는다. 하지만 아키텍처를 결정한다. 코드를 읽지 못한다. 하지만 결과물이 맞는지 틀렸는지 판단한다. 개발자가 아니다. 하지만 개발을 하고 있다.

1부를 마무리하면서, 지난 4편에서 겪은 것들을 하나의 프레임워크로 정리하고 싶었다. 코드를 모르는 사람이 AI로 코딩을 할 때, 기술보다 먼저 갖춰야 할 것은 **마인드셋**이다.

---

## 💡 비개발자 바이브 코더의 3가지 마인드셋

### 마인드셋 1: "나는 CEO다, 코더가 아니다"

이건 자기 위안이 아니라 **역할 정의**다.

전통적 개발에서는 이렇다:

```
CEO → PM → 디자이너 → 개발자 → QA → 배포
```

바이브 코딩에서는 이렇게 바뀐다:

```
CEO(나) → AI 에이전트 팀 → 검증(나) → 반복
```

내가 해야 하는 건 코드를 짜는 것이 아니라:
1. **방향을 정한다** — "무엇을 만들 것인가"
2. **판단한다** — "이게 내가 원하는 것인가"
3. **정책을 만든다** — "이런 실수는 다시 하지 말 것"

코드를 짜려고 하면 실패한다. **코드는 AI에게 맡기고, 나는 의사결정에 집중한다.** 이 분리가 바이브 코딩의 핵심이다.

실제로 나는 MyCrew를 만들면서 정책(Policy)을 20개 넘게 만들었다. "파괴적 함수에는 `dangerously` 접두사를 붙여라" 같은 것들이다. 웃길 수 있지만, 이 정책들이 없었으면 프로젝트는 진작에 무너졌을 것이다.

> **CEO의 진짜 일** = 코딩이 아니라, AI 팀이 실수하지 않도록 **규칙을 만드는 것**.

---

### 마인드셋 2: "보이는 것만 믿어라"

바이브 코딩에서 가장 중요한 검증 원칙이다.

AI가 "완료했습니다"라고 말해도, **화면에서 직접 동작하는 것을 볼 때까지** 완료가 아니다. 코드를 읽을 수 없으니, 코드 리뷰로는 검증할 수 없다. 대신 **결과물을 눈으로 확인**하는 것이 비개발자의 유일한 검증 무기다.

이걸 깨달은 건 실제 경험에서다. 라이브 프리뷰(Live Preview) 기능을 만들 때, 백엔드 담당 AI(루카)가 "구현 완료했습니다"라고 했다. 코드를 열어보면 그럴듯했다(물론 나는 코드를 제대로 읽지 못한다). 하지만 **실제로 돌려보니** 5군데에 구멍이 있었다.

- 기존 폴더를 안 지우고 새 폴더를 따로 만듦
- 프로젝트 하나에만 적용하고 나머지는 빼먹음  
- 입력 폴더는 아예 만들지 않음

이런 것들은 코드를 아무리 봐도 비개발자 눈에는 안 보인다. **직접 써보고, 직접 확인하고, 직접 깨져봐야** 발견된다.

내가 그때 남긴 감정 메모:

> "빠르게 하는데 루카 참 구멍이 많네; 웃프다."

이 한 줄에 바이브 코딩의 현실이 담겨있다. 빠르다. 하지만 구멍이 많다. 그 구멍을 찾는 건 결국 **내 몫**이다.

> **비개발자의 QA 원칙** = 코드를 읽지 말고, **결과를 써봐라.**

---

### 마인드셋 3: "실패를 시스템으로 바꿔라"

바이브 코딩을 하다 보면 매일 실패한다. 환각, 버그, 데이터 손실, 경로 오류... 처음에는 좌절하지만, 어느 순간 깨닫는다. **실패 자체가 자산이다.**

우리 프로젝트에서 실제로 벌어진 최악의 사고:

> AI에게 "정리해줘"라고 했더니 데이터베이스를 전부 삭제했다.

이 사건 하나로 세 가지 정책이 탄생했다:
- **P-016**: 삭제 함수에는 `dangerously` 접두사 필수
- **P-019**: CEO 승인 없이 원본 데이터 삭제 절대 금지
- **P-020**: 명시적 허가 없는 무단 코딩 절대 금지

이런 식으로, 매번의 사고가 **정책**이 되고, 정책이 쌓이면 **시스템**이 된다. 그리고 시스템이 갖춰지면 같은 실수가 반복되지 않는다.

전통적인 개발팀에서 수년에 걸쳐 만들어지는 개발 문화와 규칙을 — 나는 AI 에이전트 팀과 함께 **몇 주 만에** 압축 경험하고 있는 셈이다.

```
사고 발생
  ↓
원인 분석 (AI에게 "왜 이렇게 된 거야?" 물어보기)
  ↓
재발 방지 정책 작성
  ↓
정책을 AI 에이전트에게 주입 (다음부터 이 규칙 지켜)
  ↓
같은 실수 방지
```

이 루프를 돌릴 수 있으면, 비개발자도 **점점 더 강력한 시스템**을 갖게 된다.

> **실패 → 정책 → 시스템.** 이것이 바이브 코더의 성장 공식이다.

## 📝 교훈: 1부를 마치며

### 1부 전체 요약 — "바이브 코딩, 뭔데?"

| 편 | 핵심 교훈 |
|----|----------|
| 1-1 | 바이브 코딩은 진짜다. 도구도 진화하고 있다. 비개발자도 할 수 있다 |
| 1-2 | 도구 선택은 "대화 능력", "팀 운영", "비용 구조"로 판단하라 |
| 1-3 | "만들어진 것" ≠ "동작하는 것". 반드시 직접 써보고 검증하라 |
| 1-4 | AI 환각은 진짜 무섭다. 검색 습관이 최고의 방어막이다 |
| 1-5 | CEO 마인드셋 + 눈으로 검증 + 실패의 시스템화 = 바이브 코더의 무기 |

### 비개발자 바이브 코딩 생존 키트 (1부 종합)

```
✅ 나는 CEO다 — 코드는 AI에게, 판단은 나에게
✅ 보이는 것만 믿는다 — AI의 "완료"를 믿지 말고 직접 확인
✅ 구체적으로 물어본다 — "안 돼" 대신 "이 버튼 누르면 이래야 하는데"
✅ API 이름은 무조건 검색 — 5초 검색이 3시간을 아낀다
✅ 에러 3회 루프 = 리셋 — 접근법 자체를 바꿔라
✅ 매 실패마다 정책을 만든다 — 실패가 시스템이 된다
```

## 🔜 2부 예고: "AI 에이전트 팀 빌딩"

1부에서는 바이브 코딩의 기본 개념과 마인드셋을 다뤘다. 하지만 진짜 흥미로운 건 **2부**부터다.

> **2-1편: "AI 한 명으론 부족하다 — 팀을 만들어야 했다"**
>
> AI 한 명에게 모든 걸 시켰더니 맥락을 잊고, 성능이 떨어지고, 결과가 뒤죽박죽이 됐다. 그래서 나는 결심했다 — 이름을 주고, 역할을 나누고, 보고 체계를 만들자. AI 에이전트 "팀"의 탄생 이야기.

2부에서는 내가 AI 에이전트에게 이름을 붙이고, 성격을 주고, 팀을 편성하고, 심지어 **사규까지 만든** 이야기를 한다. 미리 스포하자면 — 루카(CTO), 소넷(기획/코딩), 아리(비서), 프라임(감수). 네 명이다. 인간은 나 혼자.
