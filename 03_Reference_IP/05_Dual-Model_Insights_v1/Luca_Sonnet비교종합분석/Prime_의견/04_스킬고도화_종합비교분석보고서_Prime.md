## 1. 스킬 레이어 아키텍처 비교

### Luca — 3-Layer 스킬 맵 (L1/L2/L3)

Luca는 마케팅 크루 전용 스킬을 **기술팀과 명확히 분리된 3계층 체계**로 설계:

| 레이어 | 명칭 | 핵심 스킬 | 명령어 |
|--------|------|----------|--------|
| **L1 · CONTEXT** | 맥락 동기화 | 비즈니스 코어 셋업, VOC 카피 추출 | `/비즈니스_코어셋업`, `/VOC_카피추출` |
| **L2 · GENERATION** | 기획·생성 | 캠페인 부트스트랩, 멀티채널 변환기 | `/템플릿_부트스트랩`, `/에셋_멀티_바리에이션` |
| **L3 · PERFORMANCE** | 자동화·측정 | 트리거 세팅, 스노우볼 보고 | `/트리거_자동화세팅`, `/스노우볼_현황보고` |

**핵심 특징**:
- **명령어 인터페이스**: 각 스킬이 `/슬래시_커맨드` 형태로 즉시 호출 가능하도록 설계
- **기술팀 ↔ 마케팅팀 권한 분리**: "마케팅 크루는 소스코드 수정 불가", "컨텍스트/템플릿 폴더(SSOT)에만 의존"
- **Luca의 역할 명시**: "스킬 로직 자체를 노드 서버에 개발해 주는 역할"

### Sonnet — 역할 기반 암묵적 스킬

Sonnet은 별도 스킬 맵 문서 없이, `콘텐츠_발행스케줄_워크플로우_크루역할.md` 내에서 **역할(Role) 중심으로 스킬을 암묵적 정의**:

| 크루 역할 | MyCrew 에이전트 | 주요 업무 (암묵적 스킬) |
|----------|----------------|----------------------|
| 캠페인 기획자 | CMO | 전체 일정 조율, KPI 설정 |
| 카피라이터 | 마케팅 에이전트 | 블로그 초안, 캡션 작성 |
| 비주얼 디렉터 | 디자인 에이전트 | 캐러셀 기획, 릴스 구성안 |
| 영상 스크립터 | 영상 에이전트 | 릴스/쇼츠 스크립트 |
| SEO 전문가 | SEO 에이전트 | 키워드 최적화, 해시태그 |
| 데이터 분석가 | 분석 에이전트 | 성과 모니터링, 리포트 |

**핵심 특징**:
- **명령어 인터페이스 없음**: 역할 기반 서술이지만 시스템에서 호출 가능한 형태가 아님
- **SEO 전담 역할**: Luca에는 없는 SEO 에이전트를 별도 배치
- **Phase 연결**: 각 역할이 7-Phase 워크플로우의 어느 단계에 투입되는지 명확

## 2. C-Level 거버넌스 (Luca 독점 영역)

Luca는 실무 크루 **위**에 **중간 관리자급(C-Level) 에이전트** 아키텍처를 설계:

### C-Level 5대 임무
1. **Project Manager** — 태스크를 분해하여 크루에게 병렬 할당
2. **Gatekeeper** — 결과물 1차 리뷰 (Red Teaming)
3. **Time-Tracker** — 마감 스케줄 감독
4. **System Auditor** — 컨텍스트 읽기·스킬 사용 준수 감사
5. **HR Advisor** — 크루 역량 한계 시 신규 스킬/채용 건의

### C-Level 전용 스킬 맵
| 레이어 | 스킬 | 발동 조건 |
|--------|------|----------|
| **L1 · MANAGEMENT** | `/태스크_병렬할당` | 기획안 접수 시 |
| | `/1차_실무비평` | 작업 완료칸 이동 시 (Reject/Approve) |
| | `/스케줄_감리` | 마감 설정 시 |
| **L2 · AUDIT & HR** | `/스킬_미아_추적` | 결과물 제출 시 (건별 감사) |
| | `/크루_컴플라이언스_정기평가` | 일간/주간 자동 (누적 감사) |
| | `/스킬_큐레이션_보고` | CEO 반려 2회 이상 시 |
| | `/채용_건의명세` | 병목 발생 시 |

### 칸반 보드 결재 워크플로우
```
[To-Do] → [In Progress] → [CMO Review] → [CEO Review]
  아리      CMO 병렬 지시    매의 눈 심사     마스터피스만 도달
```

> **Prime 평가**: 이 C-Level 거버넌스 아키텍처는 MyCrew의 **핵심 지적 재산(IP)**이며, 사용자(CEO)의 반려율을 "제로"로 만들겠다는 설계 의도가 명확. Sonnet에는 이에 해당하는 어떤 문서도 없음.

## 3. 품질 통제(Hallucination 방지) 비교

### Luca — 체계적 강제 장치
| 장치 | 설명 |
|------|------|
| `/스킬_미아_추적` | 크루가 컨텍스트 폴더를 읽지 않고 뇌피셜로 작업했는지 로그 추적 |
| `/크루_컴플라이언스_정기평가` | 읽기 누락율, 스킬 미사용 빈도를 일간/주간 집계 → 미달 시 재교정 |
| **SSOT 강제 참조** | "마케팅 크루의 모든 스킬은 컨텍스트/템플릿 폴더에 의존" 원칙 |
| **1차_실무비평** | CMO가 Red Teaming으로 ROI/톤앤매너 관점 사전 심사 |

### Sonnet — 구조적 약점
- 품질 통제에 대한 **별도 메커니즘 부재**
- 크루 역할 정의에 "검토" Phase가 있으나 **구체적인 심사 기준이나 반려 조건 미정의**
- 컨텍스트 폴더 강제 참조 규칙 없음

## 4. 기술팀 ↔ 실무팀 분리 비교

### Luca — 3원칙 명문화
1. 마케팅 크루는 소스코드/시스템 인프라 수정 불가 (`run_command` 등 없음)
2. 마케팅 크루의 모든 스킬은 컨텍스트/템플릿 폴더(SSOT)에 의존
3. Luca(기술팀)는 스킬 로직을 브릿지 서버에 개발해 주는 역할

### Sonnet — 분리 원칙 없음
- 크루 역할은 정의했으나 **기술팀과의 권한 경계**에 대한 언급 없음
- "개발팀 담당" 행이 스케줄 테이블에 있으나 역할 분리 원칙은 미정의

## 🏆 최종 비교 결론 — 스킬 고도화

### Luca가 압도적인 영역
1. **스킬 레이어 아키텍처**: L1/L2/L3 3계층 + `/슬래시_커맨드` 인터페이스 = 시스템에서 즉시 구현 가능
2. **C-Level 거버넌스**: CPO/CQA/CHRO 역할 + 칸반 결재 워크플로우 = CEO 반려율 제로화 설계
3. **품질 통제 장치**: 뇌피셜 추적 → 정기 감사 → 재교정 → 채용 건의까지의 완전한 루프
4. **권한 분리 3원칙**: 기술팀 vs 마케팅팀의 명확한 경계

### Sonnet이 우세한 영역
1. **실무 역할 세분화**: 6종 크루(SEO 에이전트 포함)의 구체적 업무 정의
2. **Phase 연동**: 각 역할이 워크플로우의 어느 시점에 투입되는지 명확
3. **KPI 연결**: 역할별 성과 지표가 워크플로우 마지막에 통합

### Prime 권장: 최적 병합 전략
> **Luca의 거버넌스 아키텍처(껍데기) + Sonnet의 실무 역할 세분화(속살) = 완전한 스킬 시스템**

1. **L1~L3 레이어** → Luca 구조 유지, Sonnet의 SEO 에이전트를 L2에 추가
2. **C-Level** → Luca의 CMO/CQA/CHRO 거버넌스 그대로 채택 (대체 불가)
3. **명령어** → Luca의 `/슬래시_커맨드`를 전사 표준으로, Sonnet의 워크플로우를 명령어 실행 시나리오로 매핑
4. **품질 통제** → Luca의 `스킬_미아_추적` + `컴플라이언스_정기평가`를 MyCrew 필수 기능으로 구현
