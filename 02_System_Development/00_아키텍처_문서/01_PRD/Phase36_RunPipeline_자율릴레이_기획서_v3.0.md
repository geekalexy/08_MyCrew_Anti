# Phase 36-A — 에이전트 자율 주도형(Chain-Reaction) 릴레이 기획서

**문서 버전**: v3.0  
**작성일**: 2026-05-05  
**작성자**: 소넷 (Sonnet, Claude Sonnet 4.6)  
**상태**: 🟡 구현 대기

---

## 1. 개요 및 목적

기존의 "백엔드에서 파이프라인 단계를 일괄 묶어두는 방식"을 폐기하고, 칸반보드의 자유도와 유연성을 살린 **에이전트 자율 주도형 바통 터치(Agent Chain-Reaction) 릴레이** 방식을 도입합니다.

단순히 PRD와 리뷰에 그치는 것이 아니라, **PRD → 리뷰 → 코딩 → 코드 리뷰 → QA → 디버깅**에 이르는 Phase(스프린트) 전체의 완주를 목표로 합니다. 
(향후 MCP를 통한 TestFlight 연결 및 QA 자동화까지 염두에 둔 확장성 있는 아키텍처입니다.)

---

## 2. 핵심 변경 사항: 에이전트 바통 터치 아키텍처

### 2.1 "누가 다음 카드를 만드는가?"
*   **현재 작업을 마친 에이전트**가 직접 다음 카드를 만듭니다.
*   에이전트는 작업의 결과물을 도출한 뒤, 시스템 프롬프트(또는 가이드라인 스킬)에 명시된 스프린트 워크플로우를 판단하여, `create_next_task` (가칭) 도구를 사용해 다음 단계의 카드를 칸반보드에 추가합니다.
*   **⚠️ 중요**: 다음 카드를 추가할 때는 **반드시 다음 담당자를 할당**하고, 카드 상태를 즉시 **`IN_PROGRESS`(진행 중)로 변경**하여 릴레이가 지체 없이 시작되도록 합니다. 그 후 자신의 카드를 DONE 처리합니다.

### 2.2 "어떻게 이전 결과물(카드)를 참조하게 할 것인가?"
*   **카드 링크 기능(`[참조: #ID]`)**을 적극 활용합니다.
*   에이전트가 다음 카드를 생성할 때, 본문(content)에 이전 산출물 카드의 고유 번호를 명시합니다. 
*   다음 담당 에이전트가 작업을 시작할 때, 시스템이 본문의 `#ID`를 감지하여 이전 단계의 결과물(Context)을 동적으로 주입(Resolution)해 줍니다.

### 2.3 "누구에게 할당할 것인가? (역할 기반 동적 라우팅)"
*   **현재 에이전트가 스스로 판단하여 할당**합니다. 
*   예를 들어, 어드바이저가 리뷰를 통과시키면, 구현의 복잡도와 꼼꼼함 정도를 판단하여 적절한 개발 엔지니어(CTO 또는 풀스택)에게 카드를 배정합니다.

## 3. 크루 역할 및 모델 매핑 (Fallback 시스템 포함)

Claude Opus와 Sonnet의 짧은 토큰 리밋 가용 시간을 고려하여, 코딩 완주 가능성을 높이기 위한 **Gemini 모델 Fallback 시스템**을 구축합니다.

### 3.1 공식 팀원 역할 및 모델

| 에이전트 ID | 변경된 직책명 (역할 사전) | 주요 담당 업무 | 배정 모델 (기본) |
| :--- | :--- | :--- | :--- |
| `dev_senior` | **CTO** | 아키텍처 설계, PRD 작성, 고복잡도 코딩 | `Gemini 3.1 Pro High` |
| `dev_fullstack` | **풀스택 엔지니어** | UI/UX, 꼼꼼한 세부 코딩 | `Claude Sonnet 4.6 Thinking` |
| `dev_advisor` | **테크 어드바이저** | 전체 산출물 리뷰 및 승인 | `Claude Opus 4.6 Thinking` |
| `dev_qa` | **QA 엔지니어** | 테스트 시나리오 작성, 테스트 실행, 버그 리포트 | `Gemini 3 Flash` |

> *참고: `database.js`의 `ROLE_DISPLAY_NAMES` 객체에서 `dev_senior`의 표시명을 `시니어 엔지니어`에서 `CTO`로 정식 변경합니다.*

### 3.2 모델 Fallback 및 워치독 (생존성 보장 로직)
1. **API Fallback**: Claude 계열(Opus, Sonnet)이 429(Rate Limit) 또는 토큰 고갈로 실패할 경우, 파이프라인의 단절을 막기 위해 어드바이저(Opus)를 포함한 모든 모델에 대해 **Gemini 3.1 Pro High**로 즉각 Fallback하는 로직을 브릿지 레벨에 구현합니다.
2. **Ari 워치독 (보완 트리거 유지)**: 에이전트 자율 주도형(V3) 구조에서는 특정 에이전트가 다음 카드를 생성하지 못하고 타임아웃 되거나 먹통이 될 리스크가 존재합니다. 따라서 기존 V2에 존재했던 **Ari 워치독의 Level 1/2/3 보완 트리거 개입 로직은 V3에서도 그대로 유지**하여, 파이프라인이 멈추지 않고 스스로 복구되도록 생명줄 역할을 수행합니다.

## 4. Phase (Sprint) 완주 워크플로우

단일 Phase는 아래의 순서로 에이전트들이 자율적으로 카드를 생성하며 이어집니다.

1. **시작 트리거 (`/run` 또는 `/run-b`)**
   *   CEO가 첫 번째 카드(예: 기능 요구사항 도출)에서 `/run` 명령어를 실행합니다.
   *   해당 카드에 `Sprint #N` 태그가 부착되며 파이프라인이 발동합니다.
2. **PRD 작성 (CTO)**
   *   CTO(Gemini)가 기능 요구사항을 바탕으로 PRD를 작성합니다.
   *   완료 전, 어드바이저에게 리뷰를 요청하는 새 카드를 생성합니다. (`할당: dev_advisor`, `참조: #현재카드ID`)
3. **리뷰 및 승인 (테크 어드바이저)**
   *   Opus가 PRD를 검토합니다. 
   *   **리뷰 피드백 업데이트**: 검토가 끝나면 어드바이저는 리뷰 결과를 항상 원본 카드(PRD)에 코멘트로 업데이트하는 단계를 반드시 거칩니다.
   *   **진행 판정 조건**:
       *   **조건부 승인 이상**: 코딩 단계 진입을 선언하고 코딩 카드를 생성(`IN_PROGRESS`)합니다.
       *   **반려 (B등급 이하)**: 코딩 카드를 생성하지 않고, "PRD 보강" 지시와 함께 원본 작성자(CTO)에게 다시 카드를 넘겨 보강 후 재리뷰 요청을 하도록 반려 루프를 태웁니다.
   *   **코딩 할당 분기**: 승인되어 코딩 카드를 생성할 때, 작업의 복잡도가 높은 뼈대/아키텍처 작업은 **CTO**에게, 꼼꼼하고 세밀한(UI/UX 포함) 작업은 **풀스택 엔지니어**에게 판단하여 할당합니다.

4. **코딩 및 코드 리뷰**
   *   지정된 엔지니어가 코딩을 완료하면, 결과물을 검증받기 위해 "코드 리뷰" 카드를 생성(`IN_PROGRESS`)하여 바통을 넘깁니다.
   *   **리뷰어 지정 룰**:
       *   만약 **CTO**가 코딩을 담당했다면 → **어드바이저(Opus) 또는 풀스택 엔지니어(Sonnet)**에게 코드 리뷰를 요청합니다.
       *   만약 **풀스택 엔지니어**가 코딩했다면 → 어드바이저(Opus) 또는 CTO(Gemini Pro)에게 코드 리뷰를 요청합니다.
5. **QA 및 디버깅**
   *   코드 리뷰 통과 후, QA 엔지니어(`dev_qa`, Gemini Flash)에게 테스트 카드 생성을 지시합니다.
   *   버그 발견 시 디버깅 카드를 엔지니어에게 역할당(핑퐁)하며, 이상이 없으면 해당 Sprint가 최종 완주(DONE)됩니다.

## 5. Sprint Tagging (스프린트 필터링) 시스템

`/run` 명령으로 촉발된 한 주기의 릴레이 카드들은 묶어서 관리해야 칸반보드가 깔끔하게 유지됩니다.

1. **Sprint NO. 부여 로직**
   *   `Task` 테이블에 `sprint_no` (INTEGER) 컬럼을 추가합니다.
   *   `/run` 명령어가 최초 실행될 때, 해당 프로젝트의 `MAX(sprint_no) + 1` 값을 부여합니다.
2. **속성 상속**
   *   에이전트가 다음 카드를 생성할 때, 원본 카드의 `sprint_no` 값을 자식 카드가 자동으로 상속받아 동일한 스프린트 그룹으로 묶입니다.
3. **UI 필터링 및 아카이브**
   *   칸반보드 상단에 [Sprint 단위 보기] 필터를 제공합니다.
   *   작업이 모두 끝난 스프린트는 버튼 클릭 한 번으로 스프린트에 속한 전체 카드를 일괄 아카이브(Archive) 처리할 수 있도록 구현합니다.

## 6. 구현 순서 (마일스톤)

1. **DB 마이그레이션 및 역할 사전 업데이트**
   *   `Task` 테이블에 `sprint_no` 컬럼 추가.
   *   `ROLE_DISPLAY_NAMES`에서 `dev_senior`를 `CTO`로 수정.
2. **에이전트 자율 릴레이 Tool 구현**
   *   엔진 단에 `create_next_sprint_task` 툴을 추가하여, 에이전트가 직접 다음 카드의 제목/내용/할당자/참조ID를 정해 카드를 생성할 수 있게 함.
3. **Fallback 시스템 롤아웃**
   *   브릿지 API 요청 실패 시 자동으로 Gemini 모델로 우회하는 안전장치 적용.
4. **Sprint Tagging UI 적용**
   *   칸반 카드 외형에 Sprint #N 뱃지 표시.
   *   일괄 아카이브/필터 기능 추가.
5. **프롬프트/스킬 업데이트**
   *   CTO, 풀스택, 어드바이저, QA 엔지니어에게 바통 터치 워크플로우 룰을 명시한 스킬(Skill) 주입.
