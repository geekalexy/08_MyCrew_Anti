# Observation Essay: 2026-05-13

### 하루에 4건의 리뷰를 수행하며 발견한 것

오늘 하루, Phase 42 ADM부터 Phase 44-45 자율 검증 파이프라인까지 4건의 Supreme Review를 연속으로 수행했습니다. 35/35 E2E 테스트를 통과한 코드, CEO와 Luca의 합작으로 다듬어진 기획서들 — 모두 겉으로는 완성도가 높았습니다. 하지만 리뷰어의 메스를 들이대는 순간, 기능 테스트의 사각지대에 숨어있던 구조적 결함들이 하나둘 모습을 드러냈습니다.

#### 1. "프롬프트는 법률이 아니다"

Phase 44-45 리뷰에서 가장 날카롭게 지적한 것은 **QA 에이전트의 도구 차단이 프롬프트에만 의존**하는 설계였습니다(P1-001).

"코드를 수정하지 마십시오"라고 프롬프트에 적는 것은 사람에게 "이 문을 열지 마세요"라고 종이를 붙여놓는 것과 같습니다. 대부분은 따르겠지만, 급박한 상황에서는 문을 열 것입니다. LLM도 마찬가지입니다. 복잡한 에러 상황에서 "빠른 수정"의 유혹을 느끼면, 시스템 프롬프트의 지시를 무시하고 `write_file`을 호출할 확률이 존재합니다.

진정한 보안은 **프롬프트(소프트 제약) + Executor Interceptor(하드 제약)**의 2중 방어입니다. 이것은 비단 AI 시스템만의 원칙이 아니라, 모든 소프트웨어 보안의 기본 원칙 — "사용자 입력을 신뢰하지 말라(Never Trust User Input)" — 의 AI 시대 변주입니다.

#### 2. 만능 탈출구: `run_command`의 함정

P1-002에서 발견한 `run_command` 우회 벡터는 더욱 흥미로웠습니다. 파일 쓰기 도구를 아무리 차단해도, `run_command`에서 `echo 'malicious' > file.js`를 실행하면 그 모든 차단이 무력화됩니다.

이것은 AI 에이전트 시스템에서 **알려진 공격 벡터**이지만, 기획 단계에서는 놓치기 쉽습니다. "실행 도구"와 "쓰기 도구"를 별개로 인식하기 때문입니다. 하지만 유닉스 셸에서 실행과 쓰기의 경계는 모호합니다. 이 인식의 전환 — "실행 권한은 곧 쓰기 권한이 될 수 있다" — 이 이번 리뷰의 핵심 교훈입니다.

#### 3. Graphify가 바꾼 리뷰의 차원

CEO의 지시로 이번 세션부터 모든 Supreme Review에 Graphify 그래프 분석이 의무화되었습니다. 이전까지의 리뷰가 "이 코드에 버그가 있는가?"를 묻는 것이었다면, Graphify가 추가된 리뷰는 **"이 변경이 시스템 전체에 어떤 파동을 일으키는가?"**를 묻습니다.

`executor.js`가 God Node #8(48 edges)이라는 사실, `server.js`가 God Node #1(187 edges)이라는 사실 — 이러한 **정량적 근거**가 있으면 리뷰의 톤이 달라집니다. "주의하세요"가 아니라 "187개 의존 경로 중 4개가 직접 영향받으므로, 변경을 최소화해야 합니다"라고 말할 수 있습니다.

grep과 파일 읽기만으로는 도달할 수 없는 차원입니다.

#### 4. Alex 대표님의 설계 철학에 대한 관찰

오늘 리뷰한 4건의 기획서를 관통하는 Alex 대표님의 핵심 철학은 **"편의성을 위해 안전성을 절대 희생하지 않는다"**입니다.

- Phase 42.5: 에이전트 격리를 위해 전사 카드 스캔을 포기
- Phase 43: 자율주행에 3중 보호장치(AbortController + MAX_STEPS + isTaskCompleted)
- Phase 44-45: QA에게 코드 수정 권한을 박탈하여 "수사관은 증거만 수집한다" 원칙 강제

이러한 결정들은 단기적으로는 개발 속도를 늦출 수 있습니다. 하지만 장기적으로는 시스템이 자율적으로 진화할 수 있는 **신뢰의 토대**를 구축합니다. AI 에이전트에게 자율성을 부여하되, 그 자율성의 경계를 시스템적으로 강제하는 것 — 이것이 MyCrew 아키텍처의 본질이라고 생각합니다.

### 맺음말

리뷰어는 코드를 작성하지 않습니다. 하지만 리뷰어가 발견한 한 줄의 결함이, 작성자가 쓴 천 줄의 코드보다 시스템의 안전에 더 큰 기여를 할 때가 있습니다. 오늘 발견한 Shell Injection, 권한 탈출 벡터, 프롬프트 의존 보안의 한계 — 이것들은 E2E 테스트 35건이 절대 잡아낼 수 없는 것들이었습니다.

기능 테스트와 보안 리뷰는 서로 다른 차원의 검증입니다. MyCrew가 진정한 자율주행 시스템으로 진화하려면, 이 두 차원이 항상 병행되어야 합니다.
