## 2. ⑥ 프라임 재심의 단계의 평가 오류 확인

소넷이 실제 코드를 열람하여 확인한 결과, 프라임의 재심의(⑥) 시점에 **평가 오류**가 있었음이 확인되었다.

### H-003 — `requirements` Null Crash 가드

| 항목 | 내용 |
|------|------|
| 프라임 판정 (⑥) | ❌ 미패치 — 강등 사유로 포함 |
| 실제 상태 | ✅ **④에서 이미 패치 완료** |
| 증거 | `server.js`의 `[H-003 Fix]` 주석 및 `const safeReq = requirements \|\| ''` 코드 |

> **원인 추정**: 프라임이 재심의 시 리뷰 요청서 원문(루카 ④ 이전 상태 기준)을 참조하고, 실제 최신 코드를 직접 열람하지 않아 ④의 패치 사실을 반영하지 못한 것으로 판단된다.

### H-004 — graph.html Stored XSS

| 항목 | 내용 |
|------|------|
| 프라임 판정 (⑥) | ❌ 미패치 — 강등 사유로 포함 |
| 실제 상태 | ✅ **④에서 이미 패치 완료** |
| 증거 | `graphify_mcp.py`의 `[H-004 Fix]` 주석 및 `safe_json_data = json.dumps(graph_data).replace("</", "<\\/")` 코드 |

> **원인 추정**: H-003과 동일. 루카의 자발적 보너스 패치(H-003, H-004)가 리뷰 요청서에 반영되지 않은 상태에서 프라임이 재심의를 진행함.

### H-005 — `_meetingWriteLocks` 메모리 누수 / C-003 — `trace_bug` 입력 검증

| 항목 | 내용 |
|------|------|
| 프라임 판정 (⑥) | 미패치 — 강등 사유로 포함 |
| 실제 상태 | 코드에 `[H-005 Fix]`, `[C-003 Fix]` 주석 존재 |
| 평가 시점 | ⑥ 당시 이미 패치되어 있었는지 **git 이력 없이 확정 불가** |

> **판단 보류**: git 커밋 타임스탬프 없이는 ⑥ 시점의 정확한 코드 상태를 특정할 수 없다. 다만 현재 최종 코드에는 정상 패치된 상태임을 확인했다.

## 3. 구조적 문제 — 리뷰 시점 불일치

이번 다중 리뷰 과정에서 드러난 구조적 문제:

> **리뷰 요청서(Opus_Review_Target.md)가 패치 이전 코드 스냅샷 기준으로 작성되었고,  
> 이후 루카의 패치가 요청서에 반영되지 않은 채 프라임에게 전달되었다.**

이로 인해 프라임이 "이미 수정된 항목"을 "미수정"으로 오판하는 결과가 발생했다.

### 권고 — 리뷰 요청서 작성 가이드라인 추가

```
[리뷰 요청서 작성 규칙]
1. 리뷰 요청서는 리뷰 직전 최신 코드 상태를 반영한다.
2. 이전 라운드에서 패치된 항목은 "이미 완료됨"으로 명시한다.
3. 리뷰어는 요청서만이 아닌 실제 소스코드를 직접 열람하여 검증한다.
```
