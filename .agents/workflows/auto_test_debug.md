# 🕵️ 자율 검증 및 디버깅 (Auto Test & Debug) 워크플로우

본 워크플로우는 Phase 44~45에 정식 아리 엔진에 탑재될 **"2-Track QA & Debug 파이프라인"**을 우리(CEO와 Antigravity)가 먼저 수동으로 시뮬레이션(Dogfooding)해 보기 위해 설계되었습니다.

## 0. 🛠️ 사전 준비 (Pre-flight Check)
- 검증할 코드 혹은 태스크(예: 특정 모듈이나 스크립트)가 준비되어 있어야 합니다.
- Graphify 기반 정적 분석을 위해 최신 상태가 반영되어 있어야 합니다. (필요 시 `// turbo`를 활용해 `graphify update .` 실행)

## 1. 🔍 [QA 에이전트 빙의] Track 1: Graphify 정적 아키텍처 스캔
> 💡 **추천**: 안티그래비티 모델을 **[Claude Sonnet 4.6 (Thinking)]** (dev_qa_auto)으로 변경하여 검증 전문성을 높이세요.

에이전트(QA)는 코드를 무작정 실행하기 전, 다음 항목을 먼저 정적으로 스캔합니다.
- **파급 반경 (Blast Radius)**: 검증 대상 파일들을 Import하고 있거나 의존하는 파일들을 `graph.json`에서 조회합니다.
- **구조적 결함**: 대상 모듈에 순환 참조(Circular Dependency)나 데드 코드(Orphan)가 발생하지 않았는지 구조를 점검합니다.

## 2. 🚀 [QA 에이전트 빙의] Track 2: 동적 실행 및 증거 확보
- 에이전트는 터미널 도구(`run_command`)를 활용해 실제 빌드 또는 테스트 스크립트(`npm test`, `node script.js` 등)를 실행합니다.
- 콘솔 출력(Stdout/Stderr)을 꼼꼼히 확인하고, 에러나 예외(Exception)가 발생하는지 포착합니다.
- 🚨 **[P1-001, P1-002 핵심 규칙]**: 
  - 이 단계에서 에이전트는 **파일 편집기 도구를 절대 사용하지 않습니다.**
  - `run_command` 시에도 `>`, `>>`, `rm`, `mv` 등 **파일 시스템을 훼손하는 명령어를 절대 사용해서는 안 됩니다** (화이트리스트 준수).
  - 오직 `run_command`(실행), `view_file`, `grep_search` 3개의 도구만 활용합니다.

## 3. 📝 QA 리포트 아티팩트 작성 및 보고 (Hard Stop)
에러가 발견되었다면, 에이전트는 아래 양식에 맞추어 `[QA_Report_YYYYMMDD.md]` 아티팩트를 작성합니다. 에러가 없다면 "검증 통과"를 선언하고 종료합니다.
- **실행된 명령어**: (예: `node server.js`)
- **에러 스택 (Log)**: (발견된 스택 트레이스)
- **Graphify 연관 모듈 (Blast Radius)**: (에러 지점과 연결된 이웃 노드들)
- **재현 경로 (Steps to reproduce)**:

🛑 **[에이전트 필수 행동]**: 리포트 작성이 끝나면, 에이전트는 **반드시 작업을 일시 정지(Hard Stop)**하고 대표님께 다음과 같이 명시적으로 보고해야 합니다.
> *"대표님, QA 리포트 작성이 완료되었습니다. 디버깅(Track 4)으로 넘어가시려면, 우측 하단의 모델 선택기에서 **[Gemini 3.1 Pro (High)]**로 변경하신 후 '디버깅 시작해'라고 지시해 주십시오."*

## 4. 🐛 [Debug 에이전트 빙의] Root Cause 분석 및 패치
> 💡 **추천**: 안티그래비티 모델을 **[Gemini 3.1 Pro (High)]** (dev_debug_auto)로 복구하여 아키텍처 수정 전문성을 활용하세요.

CEO의 승인이 떨어지거나 "이어서 디버깅해줘"라는 명령이 주어지면, 에이전트는 디버깅 역할로 전환합니다.
- **역추적(Shortest Path)**: 에러 스택에 찍힌 파일(A)과 핵심 로직 파일(B) 사이의 의존성 경로를 Graphify에서 추적(`shortest_path` 쿼리 또는 JSON 탐색)하여 진짜 원인(Root Cause)을 찾아냅니다.
- **코드 패치**: 추론이 아닌 확실한 증거(로그 + 그래프 경로)를 바탕으로 파일 편집 도구를 사용해 코드를 수정합니다.
- 🚨 **[W-004 핵심 규칙]**: 파괴적 수정(파일/DB 삭제 등)이 불가피할 경우, 반드시 **P-016 정책**에 따라 `dangerously` 접두사를 명시하거나 CEO에게 경고 후 진행해야 합니다.
- **재검증**: 다시 Step 2(동적 실행)로 돌아가 에러가 해결되었는지 확인합니다.
