## 1. 아리의 이중 정체성 문제

현재 설계에서 "아리"라는 이름이 두 가지 다른 역할을 담당하고 있습니다:

| 역할 | 설명 | 필요 모델 | 호출 빈도 | 비용 민감도 |
|------|------|----------|----------|:---------:|
| **① 비서(Secretary)** | 사용자 대화, 태스크 접수, 라우팅 | Flash/Sonnet | **매우 높음** (모든 대화) | 🔴 극도 민감 |
| **② 합성자(Synthesizer)** | 크루 결과물 평가, 통합, 회고 | Opus | **낮음** (Phase 2,4만) | 🟢 둔감 |

이 두 역할이 하나의 "아리"에 합쳐져 있으면:
- 사용자가 "오늘 뭐 해?"라고 물어도 Opus 호출 → 불필요한 비용
- Opus가 비서 역할까지 하면 응답 속도도 느림 → UX 저하

## 2. Prime 권장: "아리 = 비서" + "별도 합성자" 분리

### 제품 관점의 에이전트 레이어

```
┌─────────────────────────────────────────────┐
│  사용자(CEO) 대화 레이어                       │
│                                             │
│  아리 (ARI) — Gemini Flash ← 비용 최소       │
│  · 대화 응대, 태스크 접수, 칸반 라우팅           │
│  · "이 요청은 이미지인가 영상인가?" 판단         │
│  · 크루에게 떠넘기고, 결과를 CEO에게 전달        │
│                                             │
├─────────────────────────────────────────────┤
│  크루 실행 레이어 (사용자에게 안 보임)            │
│                                             │
│  이미지 크루: NOVA/LUMI — Flash              │
│  영상 크루:   REX/PICO  — Sonnet             │
│  합성자:     OLLIE/SAGE — Opus ← 내부 전용    │
│                                             │
└─────────────────────────────────────────────┘
```

> **아리는 "접수 창구"이지 "실력자"가 아닙니다.**  
> 아리는 Flash로 가볍게 돌리고, Opus는 크루 내부에서만 작동합니다.

## 5. 제품에서의 운영 모드

### 일상 운영 (실험 아닐 때)

```
사용자: "블로그 글 써줘"
   ↓
ARI (Flash): "네! 크루에게 전달할게요 ⏱️"
   ↓ 라우팅
크루 내부: LUMI(이미지) + PICO(영상) → SAGE(Opus, 통합) → 결과
   ↓
ARI (Flash): "완성됐어요! 확인해주세요 📋"
```

- 사용자는 **아리하고만 대화** → Flash 비용만 발생
- Opus는 **크루 내부에서 품질 통제** → Task 당 2회만 호출
- 사용자가 체감하는 UX는 동일 (아리가 빠르게 응답)

### 실험 운영 (CKS 연구)

```
동일 구조. 다만:
- Team A: OLLIE(Opus)가 적대적으로 평가
- Team B: SAGE(Opus)가 협력적으로 평가
- 양쪽 다 ARI(Flash)가 접수 → 아리 비용은 실험 변수에 포함 안 됨
```

## 6. 핵심 설계 원칙 정리

| 원칙 | 내용 |
|------|------|
| **아리 = 가벼운 접수 창구** | Flash로 대화, 라우팅, 전달만. 판단/합성 안 함 |
| **Opus = 크루 내부 전용** | 사용자에게 직접 노출되지 않음. Phase 2,4에서만 호출 |
| **비용 격리** | 사용자 대화(고빈도·저비용) ↔ 크루 합성(저빈도·고비용) 분리 |
| **실험 공정성 유지** | 양 팀 모두 Flash + Sonnet + Opus 동일 배치 |

## 🏆 최종 결론

> **아리는 "현관문"이고, Opus는 "안방의 전문가"다.**
>
> 현관문을 금으로 만들 필요 없다.  
> 손님(사용자)은 현관문만 보고,  
> 안방의 전문가가 만든 최종 결과물만 받으면 된다.

### 바뀐 것 요약

| Before (v3) | After (v3.1) |
|-------------|-------------|
| ARI = Opus = 비서+합성자 | ARI(Flash) = 비서, SAGE(Opus) = 합성자 **분리** |
| 사용자 대화마다 Opus 호출 | 사용자 대화는 Flash, Opus는 크루 내부만 |
| ARI가 팀 멤버 겸임 | ARI는 팀 밖 라우터, OLLIE/SAGE가 팀 내 합성자 |
