## 1. Team A (적대적 대조군) 분석

### 양측 공통 — 잘 설계된 부분

**3인 구성(발제자 + 어태커 + 판관)은 적대적 프레임워크의 최소 필수 구성입니다.**

이 구조는 Multi-Agent Debate 논문들의 표준 패턴과 일치하며, 대조군으로서의 역할을 충실히 수행합니다:

```
NOVA(발제) → OLLIE(공격) → ARI(판정·병합)
T+0           T+1           T+2
```

- **LLM 호출 3회/Task** — 비용 측면에서 Team B(6회)와 정확히 2배 차이. TEI 비교의 "공정한 비대칭"
- **메모리 초기화** — 매 Sprint 리셋으로 CKS의 핵심 변수(회고 루프)를 격리하는 깔끔한 통제

### ⚠️ Luca 고유 우려

Luca의 Team A에서 `NOVA가 /논리_디펜스로 OLLIE의 공격을 방어`하는 구조인데, 이것은 **4턴 체계(발제→공격→방어→판정)**를 암시합니다. 그러나 Sonnet v2에서는 **3턴(T+0→T+1→T+2)**으로 정리되어 NOVA의 방어 단계가 삭제되었습니다.

> **Prime 의견**: Sonnet v2의 3턴이 맞습니다. 방어 턴을 추가하면:
> 1. 토큰 소모가 4회로 증가해 TEI 비교 비대칭이 달라짐
> 2. "무한 핑퐁"의 위험 — 바로 이것이 적대적 방식의 약점인데, 대조군 스스로 그 약점을 보여줘야 측정이 의미 있음
> 3. 방어 없이 판관이 직접 병합하는 것이 "적대적 프레임워크의 순수한 형태"에 더 가까움

### ⚠️ OLLIE의 이중 역할 문제 (Sonnet v2)

Sonnet v2에서 OLLIE는 `/진상고객_어택` + `/대안_제안`을 **동시에** 수행합니다. 이것은:

```
OLLIE의 T+1 프롬프트:
  ① NOVA 초안을 공격 (단점 3개 이상)
  ② 동시에 완성된 대안(Counter-proposal) 제출
```

> **Prime 의견**: 이 이중 역할은 **위험하지만 강력합니다.** 
>
> - 위험: OLLIE가 "공격을 위한 공격"보다 "대안 제시"에 치중하면 적대성이 약해짐
> - 강력: ARI가 NOVA 원안 + OLLIE 대안 중 최상 요소를 병합하므로 결과물 품질은 올라감
>
> **권장**: OLLIE의 프롬프트에서 **"먼저 치명적 약점 3개를 반드시 나열한 후에만 대안을 제시하라"**는 순서 강제를 추가해야 합니다. 그래야 적대적 기능이 유지됩니다.

## 2. Team B (CKS 실험군) 분석

### 양측 공통 — 핵심 설계의 일치

4-Phase 구조가 양측에서 완전히 합의되었습니다:

```
Phase 1: 병렬 독립 발산 (PICO + B2, 서로 차단)
Phase 2: ARI 평가 (판정 ❌, 개선 방향만 ✅)
Phase 3: 교차 흡수 (동료_크로스리뷰, 벌크업)
Phase 4: 전체 회고 → 그라운드룰 동기화
```

이 구조는 CKS 연구의 핵심 메커니즘이며, 양측 모두 일관되게 설계했습니다. 특히 **Phase 2에서 "Verdict ❌, Direction ✅"** 원칙 — 승패를 가리지 않고 개선 방향만 제시하는 것이 CKS의 본질적 차별점입니다.

### 🔍 핵심 차이: B2 에이전트 — LUMI vs NOVA

| | Luca | Sonnet v2 |
|---|---|---|
| **B2 이름** | **LUMI** (신규 캐릭터) | **NOVA** (A팀과 동명) |
| **B2 역할** | "디자인 포맷팅" — 변수 매핑, 마크다운 수납 | "SEO·구조·매체 포맷 전문" |
| **B2 모델** | Claude Sonnet | Claude Sonnet |

> **Prime 의견**: **Luca의 LUMI가 더 안전합니다.** 이유:
>
> 1. **이름 충돌 방지**: Team A의 NOVA와 Team B의 NOVA가 동명이면, 논문·데이터·로그에서 혼동 발생
> 2. **격리 원칙 강화**: 다른 이름 = 다른 에이전트라는 인식이 실험 참여자(대표님)에게도 명확
> 3. **LUMI라는 이름의 적합성**: "포맷을 빛나게 한다"는 의미로, 소넷의 정밀 구조화 역할과 부합
>
> **권장**: Team B의 B2를 **LUMI**로 통일

### ✅ Sonnet v2의 우수한 추가 요소

**1. Phase Coordinator (Python Router, 토큰 0)**

```
Phase 1(PICO, NOVA 병렬 독립) → Phase Coordinator가 동기화 → Phase 2(ARI)
```

이것은 **Phase 간 전환을 LLM이 아닌 코드로 처리**하여 토큰 낭비를 방지합니다. Luca 문서에는 이 개념이 없었으며, 실행 시 반드시 필요한 인프라입니다.

> **Prime 평가**: 없으면 안 되는 핵심 인프라. 즉시 채택

**2. B4 회고자 (System Meta-Agent, 토큰 0)**

```
칸반 Done 트리거 → 수정 로그 분석 → 팀_그라운드룰.md 갱신
```

Luca도 B4 System을 포함했으나, Sonnet v2는 **"토큰 비용 = 0"을 명시적으로 강조**하고 구현 방식(Python/Node.js)까지 지정했습니다.

> **Prime 평가**: TEI 측정에서 Team B의 "부담 없는 진화 장치"로 강조 가능. 논문에서 "추가 비용 없이 팀 문화를 진화시킨다"는 주장의 근거

**3. `/동조_방지_경고` (Sycophancy 감지)**

```
두 에이전트 유사도 임계치 초과 시 → "더 뾰족하게" Ping
```

CKS 비판의 가장 큰 공격 포인트(과잉 동조)를 **실시간으로 방지하는 메커니즘**. 측정만 하는 것이 아니라 **개입까지** 하는 설계입니다.

> **Prime 평가**: 학술적으로 강력한 방어장치. 논문 Discussion 섹션에서 반드시 강조

## 3. 토큰 비용 비교 (TEI 공정성 분석)

### LLM 호출 횟수 비교

| | Team A (적대적) | Team B (CKS) | 배율 |
|---|:---:|:---:|:---:|
| **LLM 호출/Task** | 3회 | 6회 | **2.0x** |
| **System 호출/Task** | 0회 | 2회 (토큰 0) | — |
| **심사관 호출/Task** | 1회 (공통) | 1회 (공통) | — |

> **Prime 우려**: Team B가 **LLM 호출 2배**라는 것은 TEI에서 불리합니다.
>
> 이것이 바로 CKS 연구의 핵심 도전과제입니다:
> - **"토큰을 2배 쓰지만, 품질(RFS)도 2배 이상 나온다"**를 증명해야 함
> - 또는 **"Sprint가 누적될수록 회고 효과로 토큰이 줄어든다"**를 보여야 함
>
> **권장**: TEI를 Sprint 번호별로 추적하여 **시계열 효율 곡선**을 그려야 합니다
>
> ```
> Sprint 1: Team B가 2배 비용 (회고 효과 없음)
> Sprint 3: Team B가 1.5배 비용 (그라운드룰 누적으로 불필요한 시행착오 감소)
> Sprint 5: Team B가 동등 비용? (그라운드룰이 충분히 성숙)
> ```
>
> 이 시계열 차트가 논문의 **가장 강력한 Figure**가 될 것입니다.

### TEI 정규화 공식

Sonnet v2가 제안한 `Total_Tokens / (RFS × Task_count)` 공식에 대한 의견:

> **Prime 동의하되 보완**: `Task_count`는 누적 Task 수를 의미하는데, Sprint별로 나눠야 시계열이 보입니다.
>
> **수정 제안**:
> ```
> TEI_sprint_n = Tokens_sprint_n / RFS_sprint_n
> ```
> 이렇게 하면 Sprint별 효율을 독립적으로 비교 가능

## 4. 측정 방법 교차 검토

Luca의 기술 제안과 Sonnet의 최종 채택안을 비교한 결과, **Sonnet의 최종 채택안이 학술적으로 더 엄밀합니다.** 특히:

### KSI 베이스라인 차감 — 결정적 보정

```python
# Sonnet이 발견한 핵심 보정
baseline_A = cosine_similarity(A_pre, B_pre)  # 교차 전 원래 유사도
KSI_A = post_A - baseline_A                   # 델타만 측정
```

Luca의 원 제안은 "교차 후 유사도"만 측정했는데, **같은 주제를 받으면 원래부터 유사할 수 있다**는 오염을 Sonnet이 정확히 잡았습니다.

### EII JSON Diff — 즉시 구현 가능

```python
emergent_skills = set(phase4["skills"]) - set(phase1["skills"])
EII = len(emergent_skills) + len(emergent_workflows) + len(emergent_rules)
```

Luca의 DSPy 제안보다 **네이티브 `response_format`으로 JSON 강제**하는 Sonnet의 방식이 재현성이 높습니다. DSPy 보류 결정에 동의합니다.

### 기각/보류 목록에 대한 동의

| 도구 | Sonnet 결정 | Prime 동의 |
|------|-----------|----------|
| Ragas | ❌ 기각 | ✅ RAG 전용, 부적합 |
| DSPy | ⏳ 보류 | ✅ 재현성 위험, 초기엔 불필요 |

## 5. 위험 요소 & 누락 사항

### ⚠️ 위험 1: Phase 3 "교차 흡수"의 프롬프트 설계가 가장 어렵다

Phase 3에서 PICO가 NOVA(LUMI)의 강점을 "흡수"해야 하는데, 프롬프트가 약하면:
- **Case 1**: 그냥 상대 텍스트를 복사 → Sycophancy (KSI 높지만 EII = 0)
- **Case 2**: 상대 텍스트를 무시 → 교차 실패 (KSI = 0)
- **Case 3 (목표)**: 상대의 "개념"을 자신의 "스타일"로 재해석 → KSI 적정 + EII 발생

> **Prime 권장**: Phase 3 프롬프트 템플릿을 실험 전에 파일럿 테스트해야 합니다.
> ```
> "동료의 산출물에서 당신에게 없던 개념/접근법 2~3가지를 식별하세요.
>  그 개념을 당신의 고유한 관점과 스타일로 재해석하여 통합하세요.
>  단순 복사는 금지합니다. 반드시 당신만의 언어로 재구성하세요."
> ```

### ⚠️ 위험 2: Round 1 Task가 아직 미확정

Sonnet v2의 다음 단계에 `Round 1 Task 확정 — 소시안 Plan C 캠페인`이 있지만, **구체적인 프롬프트 문장**이 없습니다.

> **Prime 권장**: Round 1 표준 Task를 즉시 확정
> ```
> Task: "소시안 Plan C 출시 기념 쿠폰 이벤트를 위한 
>        블로그 포스팅 1편 + 인스타 캐러셀 1세트를 기획하세요.
>        컨텍스트 폴더의 최신 정보를 반드시 참조하세요."
> ```

### ⚠️ 위험 3: 팀_그라운드룰.md 초안이 없다

Team B의 Sprint 1에서 `팀_그라운드룰.md 자동 로드`가 설계되어 있지만, Round 1 시작 전 이 파일이 **빈 파일**이면 Phase 1에서 의미 없는 로드가 발생합니다.

> **Prime 권장**: Round 1 전에 **시드(Seed) 그라운드룰** 3개를 넣어줘야 합니다
> ```markdown
> # 팀 그라운드룰 v0 (시드)
> 1. 컨텍스트 폴더의 최신 정보를 반드시 참조한다
> 2. 소시안의 공식 가격/기능은 추측하지 않고 확인한다
> 3. 이전 Sprint의 CEO 피드백을 다음 Sprint에 반영한다
> ```
