## 🔴 검토 요청 1: 멀티테넌트 메모리 분리 아키텍처

### 1-1. skill-memory 공유 학습의 데이터 유출 엣지 케이스

> [!CAUTION]
> **치명적 위험: "세탁된 기밀(Laundered Confidential)"**

Luca가 설계한 2-Layer 모델의 가장 위험한 엣지 케이스는 직접적 기밀 유출이 아니라, **학습 경험으로 위장된 간접 유출**입니다.

**시나리오:**
```
1. 에이전트가 A사(소시안)에서 일하면서 skill-memory에 기록:
   "FOMO Hook이 저장율을 3배 높인다" ← 이건 범용 지식. 안전.

2. 그런데 다음과 같이 기록되면?
   "B2B SaaS 무료체험 전환율이 23%일 때 FOMO Hook이 최적"
   → 23%는 소시안의 실제 전환율. 이것이 skill-memory에 포함되면
   → B사 에이전트가 "경쟁사 소시안의 전환율은 23%입니다"를 추론 가능.

3. 더 교묘한 케이스:
   "이전 프로젝트에서 키워드 '소시안SaaS'가 CTR 4.7% 달성"
   → 브랜드 키워드 + 실제 성과 수치가 skill-memory에 녹아듦.
```

**근본 원인:** 현재 설계에는 **skill-memory에 무엇이 기록될 때 tenant-data를 자동으로 필터링하는 메커니즘이 없습니다.** skill-memory에 기록하는 주체가 에이전트(LLM)이기 때문에, LLM이 "이건 범용 지식이다"와 "이건 고객사 기밀이다"를 정확히 구분할 수 있다고 가정하는 것 자체가 위험합니다.

**Prime 권고 — Sanitization Layer 필수:**

```
[skill-memory 기록 전 3단계 검증]

Step 1: Entity Scrubbing
  - 회사명, 인명, 프로젝트명, URL, API 키 → 자동 마스킹
  - 정규식 + NER(Named Entity Recognition) 이중 필터

Step 2: Numeric Generalization
  - 구체적 수치(23%, $4,700) → 범위화("20~30%", "수천 달러급")
  - 정확한 숫자는 tenant-data에만 보존

Step 3: Human Review Gate (초기 단계)
  - SaaS 출시 초기에는 skill-memory 업데이트를 자동이 아닌
  - 주 1회 관리자(대표님) 승인 후 반영하는 반자동 모드 권장
```

### 1-2. SOC2 / GDPR 격리 기준 부합 여부

> [!WARNING]
> **현재 설계만으로는 SOC2 Type II / GDPR 인증 요건 미충족**

**이유:**

| 요건 | SOC2/GDPR 기준 | 현재 설계 | Gap |
|:---|:---|:---|:---|
| **데이터 격리** | 논리적 또는 물리적 격리 증명 필요 | 폴더 분리(`tenant-data/[company-id]/`) | 폴더 분리는 **논리적 격리의 최소치**에 불과. 접근 제어 로직(ACL) 미구현 |
| **접근 로그** | 누가 언제 어떤 데이터에 접근했는지 감사 추적 필수 | 미구현 | ❌ **Audit Trail 전무** |
| **데이터 삭제** | GDPR Art.17 "잊혀질 권리" — 고객 요청 시 완전 삭제 | 미설계 | ❌ skill-memory에 녹아든 학습은 삭제 불가 |
| **크로스 테넌시** | 테넌트 간 데이터 절대 공유 불가 증명 | skill-memory가 크로스 테넌트 | ⚠️ skill-memory 자체가 크로스 테넌트 설계이므로 증명 곤란 |

**Prime 권고:**
- **단기(MVP):** 폴더 격리 + 접근 로그(DB에 `AccessLog` 테이블 추가)로 최소 요건 충족 가능
- **중기(SOC2 대비):** skill-memory 업데이트 시 `origin_tenant_id`를 메타데이터로 기록하고, 특정 테넌트 삭제 요청 시 해당 origin의 학습 데이터만 선택적 제거 가능하도록 설계
- **장기:** skill-memory를 **Federated Learning** 방식으로 전환 — 각 테넌트에서 학습된 모델 가중치만 공유하고 원본 데이터는 이동하지 않는 구조

## 🟡 검토 요청 2: 강제 Cold-Start 복구 프로토콜

### 2-1. CLAUDE.md 강제 진입 파일이 칸반 기반 비동기 환경에서 유효한가?

**결론: 조건부 유효 — 단, 현재 설계에는 중대한 전제 오류가 있습니다.**

CLAUDE.md가 작동하는 이유는 **Claude Code가 세션 시작 시 해당 파일을 강제 로드하는 런타임 메커니즘**이 내장되어 있기 때문입니다. 이는 Anthropic이 에이전트 프레임워크 레벨에서 구현한 기능입니다.

> [!IMPORTANT]
> **핵심 질문: MyCrew 에이전트의 런타임에서 MYCREW.md를 "강제로 먼저 읽게" 만드는 메커니즘이 어디에 있습니까?**

현재 코드를 전수 검사한 결과:

```javascript
// executor.js — 스킬 라우팅 후 시스템 프롬프트로 skill.getSystemPrompt()만 사용
const skill = router.route(actualContent, evaluation.category);
const systemPrompt = skill.getSystemPrompt();
// ↑ 여기서 MYCREW.md, IDENTITY.md, recovery-logs를 읽는 코드가 없음

// server.js — 에이전트 재기동 시 복구 로직 부재
// httpServer.listen(PORT) 호출 시 복구 시퀀스 없음
```

**즉, MYCREW.md 파일을 아무리 잘 작성해도 현재 엔진이 그것을 읽지 않습니다.**
문서만 만들어놓고 읽는 코드가 없는 것은 설계서가 아니라 소원 목록입니다.

**Prime 권고 — 2가지 구현 경로:**

```
[경로 A: 시스템 프롬프트 주입 방식 — 권장]
executor.run() 시작 시:
  1. fs.readFileSync('MYCREW.md') 로드
  2. 현재 skill.getSystemPrompt() 앞에 prepend
  3. recovery-logs/latest.md 내용을 context로 추가
  → 매 요청마다 에이전트 정체성 + 최근 맥락 주입됨

[경로 B: Paperclip Agent Configuration 방식]
  Paperclip의 에이전트 설정(Configuration)에
  MYCREW.md 전문을 붙여넣기
  → 별도 코드 수정 없이 SDK 레벨에서 강제 로드
  → 단점: Configuration 토큰 한도(보통 8K~32K)에 의해 제한
```

### 2-2. "30분 스냅샷 + 7일 보관" 정책 평가

**적정 수준이나 개선 필요:**

- **30분 간격:** 칸반 카드 단위 작업에서는 적절. 다만 **카드 상태 변경(status transition) 시점에 즉시 스냅샷**하는 것이 더 중요합니다. 30분 타이머보다 이벤트 드리븐 방식이 우선.
- **7일 보관:** 적정. 다만 `COMPLETED` 상태의 카드는 recovery-logs에서 제거하고 **Obsidian 아카이브로 이관**하는 자동 클린업이 필요합니다.

### 2-3. 가장 취약한 복구 실패 시나리오

> [!CAUTION]
> **최악의 시나리오: "Zombie Context" — 복구는 됐으나 맥락이 틀린 경우**

```
[시나리오]
1. 에이전트가 Task #42 작업 중 (status: IN_PROGRESS)
2. 30분 스냅샷 시점에 기록: "Task #42 마케팅 기획안 50% 작성 중"
3. 스냅샷 직후(t+1분) 대표님이 Task #42를 CANCEL하고 Task #43을 긴급 투입
4. t+15분에 시스템 크래시
5. 재기동 → recovery-logs에서 #42(50% 작성 중) 복구 시도
6. 에이전트가 이미 취소된 #42를 다시 작업 시작 ← ZOMBIE CONTEXT

[근본 원인]
recovery-logs가 DB의 Task 실제 상태와 동기화되지 않음
```

**Prime 권고:**
- recovery-logs는 **보조 참고용**으로만 사용
- 복구의 **Single Source of Truth는 반드시 SQLite DB의 Task 테이블**이어야 합니다
- 재기동 시 `MYCREW.md` → DB 쿼리(`SELECT * FROM Task WHERE status = 'IN_PROGRESS'`) → 실제 진행 중인 작업만 재개

```javascript
// server.js 서버 시작 시 복구 시퀀스 (예시)
async function bootRecovery() {
  const activeTasks = await dbManager.getAllTasks()
    .then(tasks => tasks.filter(t => t.status === 'IN_PROGRESS'));
  
  if (activeTasks.length > 0) {
    const summary = activeTasks.map(t => `#${t.id}: ${t.content}`).join('\n');
    broadcastLog('info', 
      `시스템 재기동 완료. 진행 중이던 ${activeTasks.length}개 작업을 확인했습니다:\n${summary}`, 
      'system');
    // 텔레그램 알림
    if (bot && chatIdEnv) {
      bot.sendMessage(chatIdEnv, 
        `🔄 시스템 재기동 완료.\n진행 중이던 작업 ${activeTasks.length}건:\n${summary}`);
    }
  }
}
```

### 3-1. 독자 개발의 기술적 위험성

> [!WARNING]
> **"NIH 증후군(Not Invented Here)" 경고 — 이 기능은 구현 대비 유지보수 비용이 폭발적입니다**

MKV의 **3가지 커스텀 링크** (`{{SOC-21}}`, `{{skill:marketing-v1.2}}`, `{{company:socian}}`)를 자체 구현한다는 것은 사실상 **미니 파서 + 리졸버 + 접근 제어 엔진**을 0에서 만드는 것입니다.

**구현 난이도 현실 점검:**

| 기능 | 겉보기 난이도 | 실제 난이도 | 이유 |
|:---|:---|:---|:---|
| `{{SOC-21}}` 파싱 | 쉬움 | 중간 | 정규식으로 잡으면 되지만, 본문 내 `{{}}` 가 코드블록 안에 있을 때, 이스케이프 처리, 중첩 링크 등 엣지 케이스 다수 |
| 카드 ID → 컨텍스트 리졸빙 | 중간 | **높음** | DB에서 카드 조회 + 댓글 히스토리 + CCB 블록 추출 + 토큰 한도 내 압축 → 이것이 매 요청마다 실행됨 |
| `{{skill:marketing-v1.2}}` | 중간 | **매우 높음** | **버전 레지스트리가 없으면 불가능**. v1.2가 어떤 파일인지 매핑하는 인덱스 필요. Git tag? 폴더 복사? 별도 DB? |
| `{{company:socian}}` 접근 제어 | 높음 | **극한** | LLM에게 "넌 지금 socian 세션이니까 {{company:socian}} 접근 가능"이라고 알려줘야 함. LLM은 무시할 수 있음 |

### 3-2. 가장 발생하기 쉬운 파싱/캐싱 버그

```
[버그 #1: Greedy Matching]
입력: "{{skill:marketing-v1.2}}와 {{skill:content-v2.0}} 비교"
빠른 정규식: /\{\{(.+?)\}\}/g → 정상 작동
하지만: "{{skill:marketing-v1.2}}의 {{가격}}"
→ "가격"은 MKV 링크가 아닌데 파싱됨 → 리졸버 에러

[버그 #2: Stale Cache]
skill-memory가 업데이트됐는데 캐시된 이전 버전의 SKILL.md를 참조
→ 에이전트가 구버전 규칙으로 작업 → 품질 저하
→ 캐시 무효화(invalidation) 전략이 반드시 필요

[버그 #3: Recursive Resolution]
{{SOC-21}} 카드 안에 "관련 카드: {{SOC-18}}" 텍스트 존재
→ 리졸버가 재귀적으로 {{SOC-18}}도 해석 시도
→ 깊이 제한 없으면 순환 참조(infinite loop) 또는 토큰 폭발
```

### 3-3. MKV에서 가장 먼저 버려야 할 부분

> [!IMPORTANT]
> **즉시 드롭: `{{skill:marketing-v1.2}}` (스킬 버전 참조)**

**이유:**
1. **버전 레지스트리 시스템이 전혀 없는 상태에서 버전 참조는 허상입니다.** Git tag, semantic versioning 인프라, 버전별 스냅샷 저장소가 모두 미구현.
2. 현재 SKILL.md 파일은 **단일 파일이 계속 덮어쓰여지는 구조**. v1.0과 v1.2의 차이를 어디서 보존합니까?
3. 이 기능을 위해 **스킬 레지스트리 서비스**를 별도로 만들어야 하는데, 현재 MyCrew의 인프라(SQLite 단일 파일 + 로컬 런타임)에서는 과잉 엔지니어링입니다.

**대안:** 스킬 버전 관리는 **Git 히스토리**로 대체. SKILL.md의 하단에 `## Changelog` 섹션만 유지하면 됩니다. 나중에 SaaS 규모가 커지면 그때 레지스트리를 도입해도 늦지 않습니다.

## 🔵 추가 발견사항: 기존 코드와의 정합성 검증

기획서를 코드와 대조하면서 발견한 **기획서에서 다루지 않은 기존 결함**들입니다:

### 발견 1: SKILL.md와 `*Skill.js`의 이중 체계 문제

> [!WARNING]
> **현재 스킬 시스템은 "두 곳에 프롬프트가 존재"하는 이중 체계입니다**

```
[A] skill-library/02_marketing/SKILL.md  ← 152줄의 상세 문서
[B] ai-engine/skills/marketingSkill.js   ← 15줄의 getSystemPrompt()
```

기획서에서는 SKILL.md를 정교하게 작성하지만, **실제 런타임은 `marketingSkill.js`의 `getSystemPrompt()`만 읽습니다.** SKILL.md는 현재 어떤 코드에서도 참조되지 않습니다.

이것은 단순히 "아직 연결 안 했다"가 아니라 **아키텍처 결정**이 필요합니다:

```
[옵션 A: SKILL.md를 런타임에 로드]
executor.js에서 fs.readFileSync()로 SKILL.md를 읽어 시스템 프롬프트로 사용
→ 장점: 문서와 실행이 동기화
→ 단점: 매 요청마다 파일 I/O, 200줄 전체를 프롬프트에 넣으면 토큰 비용 급증

[옵션 B: SKILL.md는 인간용 문서, JS는 압축 프롬프트]
SKILL.md는 스킬 설계/관리용 참고 문서로만 유지
getSystemPrompt()는 SKILL.md의 핵심만 압축한 런타임 프롬프트
→ 장점: 토큰 효율적
→ 단점: 문서와 코드가 괴리될 위험 (지금 상태)

[옵션 C: Progressive Injection — Prime 권장]
SKILL.md를 3단계로 분리:
  L1: YAML frontmatter(description) → modelSelector 트리거 판단에 사용
  L2: 핵심 규칙 20줄 → getSystemPrompt()에 하드코딩
  L3: 상세 예시/참조 → 필요 시 동적 로드 (에이전트가 판단)
```

### 발견 2: router.js의 DESIGN 라우팅 로직 결함

```javascript
// router.js:33-36
case 'DESIGN':
    if (taskContent.includes('이미지') || taskContent.includes('영상') || ...) {
        return mediaSkill;  // ← DESIGN으로 분류됐는데 mediaSkill로 빠짐
    }
    return designSkill;
```

modelSelector가 `DESIGN`으로 분류한 후, router가 **키워드 기반으로 다시 재분류**합니다. 이는 modelSelector의 LLM 판단을 런타임에서 하드코딩 키워드가 뒤집는 구조입니다. **두 시스템의 책임 경계가 불명확합니다.**
