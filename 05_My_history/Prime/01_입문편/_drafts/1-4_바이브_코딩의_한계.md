# 바이브 코딩의 한계 — AI는 만능이 아니다

> 비개발자의 CTO 도전기 — AI 에이전트 팀으로 SaaS 만들기 **1-4편**

---

## 🎬 에피소드: AI가 자신 있게 거짓말을 하는 순간

### "이 API 진짜 있어?"

어느 날, 나는 AI에게 텍스트를 음성으로 변환하는 기능을 요청했다.

> "사용자가 문서를 올리면 한국어 음성으로 읽어주는 기능을 만들어줘."

AI가 척척 코드를 짰다. `GoogleTTSv3.synthesize()` 라는 함수를 호출하는 코드였다. 깔끔하게, 자신 있게, 마치 원래 존재하는 API인 것처럼.

문제는 — **`GoogleTTSv3`라는 건 세상에 존재하지 않았다.**

AI가 지어낸 것이다. 진짜 있는 것처럼 함수명도, 파라미터도, 리턴값도 그럴듯하게 만들어냈다. 나는 한참 동안 "왜 이 코드가 안 돌아가지?" 하면서 디버깅을 시도했다. Google 문서를 뒤졌는데 해당 API가 어디에도 없었다.

이때 처음 알게 된 단어가 **환각(Hallucination)**이다.

> **환각(Hallucination)** = AI가 실제로 존재하지 않는 것을 자신 있게 만들어내는 현상.

코드 환각은 특히 위험하다. 문장으로 헛소리하면 "이상한데?" 하고 금방 눈치챈다. 하지만 코드는 비개발자 눈에는 **다 똑같이 보인다.** 진짜 API를 호출하는 코드나 가짜 API를 호출하는 코드나, 내 눈에는 그냥 영어 + 괄호 + 점의 조합이다.

---

### 환각이 불러온 연쇄 사고

환각의 무서운 점은 **한 번으로 끝나지 않는다**는 것이다.

존재하지 않는 API를 호출하는 코드를 AI가 만들면, 당연히 에러가 난다. 그러면 나는 "에러 났어, 고쳐줘"라고 한다. AI는 에러를 보고 "아, 이 부분을 수정하겠습니다"라고 하면서 **또 다른 가짜 API**를 불러온다. 혹은 존재하지 않는 버전의 라이브러리를 설치하라고 한다.

```
내: "에러 났어"
AI: "GoogleTTSv3 대신 GoogleTTSv4를 사용하겠습니다" ← 이것도 없다
내: "또 에러야"
AI: "라이브러리 버전을 2.8.3으로 다운그레이드하겠습니다" ← 이 버전도 없다
내: "......"
```

이 루프에 한 번 빠지면 30분~1시간이 순식간에 사라진다. 나중에 알게 된 건, 이런 상황에서는 **AI에게 계속 고치라고 하는 게 아니라, 아예 다른 접근법을 요청해야 한다**는 것이었다.

---

## 🧱 문제: 비개발자는 환각을 감지할 방법이 없다

개발자라면 "이 API 이름 이상한데?"라고 금방 알아챈다. 그런데 비개발자인 나는:

1. **API 이름이 진짜인지 가짜인지 구분할 수 없다** — `gemini-2.5-flash`와 `gemini-2.5-flash-preview` 중 뭐가 진짜인지 모른다 (실제로 우리 프로젝트에서 이 문제가 터졌다)
2. **라이브러리 버전이 존재하는지 확인할 줄 모른다** — npm, pip 같은 패키지 관리 도구 사용법을 모른다
3. **에러 메시지를 읽을 줄 모른다** — 영어로 된 에러 메시지가 전부 똑같아 보인다

결과적으로, **비개발자는 환각 루프에 빠지면 스스로 빠져나올 수 없다.** AI에게 계속 "고쳐줘"만 반복하다가 시간만 날린다.

---

## 💡 해결: 환각 탐지 5가지 습관

삽질 끝에 나는 나만의 **환각 탐지법**을 만들었다. 코드를 읽을 줄 몰라도 쓸 수 있는 방법들이다.

### 습관 1: "이 API 이름 검색해봐"

AI가 특정 서비스나 API를 사용하는 코드를 짜면, 그 이름을 **구글에 검색**한다. 진짜 존재하는 API라면 공식 문서가 바로 나온다. 안 나오면 환각이다.

```
"GoogleTTSv3" → 구글 검색 → 결과 없음 → 🚨 환각!
"@google-cloud/text-to-speech" → 구글 검색 → 공식 npm 페이지 있음 → ✅ 진짜!
```

### 습관 2: "공식 문서 링크 줘"

AI에게 사용한 라이브러리의 공식 문서 링크를 요청한다.

> "지금 사용한 라이브러리의 공식 문서 URL을 알려줘."

진짜 라이브러리라면 유효한 URL을 준다. 환각이면 404 페이지가 뜨거나, 아예 다른 곳으로 연결된다.

### 습관 3: "다른 방법도 있어?"

같은 기능을 구현하는 **대안**을 물어본다.

> "이거 말고 다른 라이브러리로도 구현할 수 있어? 3가지 대안 알려줘."

AI가 여러 대안을 제시하면, 그중 하나라도 진짜인지 검색해본다. 전부 처음 듣는 이름이면 의심해야 한다.

### 습관 4: "에러가 3번 반복되면 리셋"

같은 에러로 3번 이상 왔다 갔다 하면, **그 접근 자체를 버린다.**

> "지금 방식 말고, 완전히 다른 접근법으로 처음부터 다시 해줘."

환각 루프에서 빠져나오는 가장 확실한 방법은 **대화를 리셋하는 것**이다.

### 습관 5: "최신 정보가 맞아?"

AI의 학습 데이터에는 시점이 있다. 2024년 데이터로 학습한 AI가 2026년 API를 정확히 알 리가 없다.

> "이 라이브러리의 최신 버전이 뭐야? 2026년 기준으로."

이 질문을 던지면 AI가 "제 학습 데이터는 ~까지입니다"라고 솔직하게 말하거나, 확인할 수 없다고 인정하는 경우가 많다. 이게 오히려 안전하다.

### 실제 사례: 우리 프로젝트에서 터진 환각

이건 나중 이야기인데, 선공개한다. MyCrew를 만들면서 실제로 벌어진 일이다.

우리 팀의 CTO 에이전트(루카)가 `gemini-2.5-pro-preview`라는 모델 식별자를 코드에 넣었다. 자신 있게. 마치 원래 있는 것처럼.

**이 식별자는 존재하지 않았다.**

문제는, 루카가 이 환각 식별자를 **여러 파일에 걸쳐** 사용했다는 것이다. 하나를 고치면 다른 곳에서 또 나오고, 또 고치면 또 나오고. 결국 **전수 조사**를 해서 5건의 환각 식별자를 찾아 교정했다.

이 사건 이후 우리 프로젝트에는 이런 정책이 생겼다:

> **P-004**: `-preview`, `-exp`, `-experimental` 접미사 모델 식별자 절대 금지  
> **P-006**: 모델 식별자는 반드시 공식 레지스트리 참조 — 직접 입력 금지

AI 에이전트에게도 **사규**를 만들어야 한다. 이건 2부에서 자세히 다루겠다.

## 📝 교훈: 네 번째 편을 마치며

| 배운 것 | 내용 |
|---------|------|
| **AI는 모르면 모른다고 안 한다** | 자신 있게 거짓을 말하는 게 환각이다. 진짜 무섭다 |
| **비개발자는 환각에 특히 취약하다** | 코드가 진짜인지 가짜인지 구분할 기준이 없기 때문 |
| **검색이 최고의 무기** | API 이름을 구글에 넣어보는 5초가 3시간을 아낀다 |
| **에러 루프 3회 = 리셋 신호** | 같은 에러가 3번 반복되면 접근법 자체를 바꿔야 한다 |
| **AI에게도 사규가 필요하다** | 환각 방지 정책을 문서화하면 같은 실수가 반복되지 않는다 |
