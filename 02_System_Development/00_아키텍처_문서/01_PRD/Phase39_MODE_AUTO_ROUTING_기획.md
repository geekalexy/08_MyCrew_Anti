## 1. 개요 및 문제 정의
과거 MyCrew는 `/run`, `/run-b`, `/plan`, `/execute` 등 기능이 추가될 때마다 슬래시 커맨드(/)가 기하급수적으로 늘어나는 구조였습니다. 향후 다중 MCP(Task Master, Sequential Thinking, Graphify 등)가 연동됨에 따라 발생할 **'커맨드 오버로드(Command Overload)'** 및 심각한 UX 저하를 원천 차단하기 위해, **가입 직후 나타나던 파이프라인 시작 프롬프트(CTA)를 시스템에서 완전히 제거**했습니다.

본 기획서는 슬래시 커맨드를 전면 폐지하는 것이 아니라, 저관여 사용자를 위한 UI/UX 설계 고도화 작업의 일환으로 **"상위 워크플로우(Mode)"**와 **"의도 기반 자동 라우팅(Zero-Command Router)"**을 도입하는 아키텍처를 제안합니다. 고관여 사용자는 `/` 명령어를 자유롭게 사용하는데 익숙하므로 이를 제약하지 않습니다.

## 2. 해결 방안 1: 모드(Mode) = 고성능 매크로 스킬 직관적 실행

모드는 커맨드 명령에 익숙하지 않은 저관여 사용자를 위해 MyCrew가 제공하는 대표적인 고성능 스킬을 직관적으로 설정하고 실행하는 기능입니다. 
명령어를 외우거나 칠 필요 없이, 모드를 선택하고 텍스트를 전송하면 해당 모드에 심어진 거대한 파이프라인(매크로)이 즉시 수행됩니다.

### 🎭 4대 워크플로우 모드 및 매핑 스킬
1.  **📐 기획 모드 (Architect)** 👉 **`/plan_master` 수행**
    *   **사용 모델**: Claude Opus 4.6 (Thinking) + Gemini 3.1 Pro (High)
    *   **동작**: 기획 모드 선택 후 텍스트 전송 👉 내부적으로 `/plan_master` 로직 발동 (스코프 분석 및 로드맵 자동 생성)
2.  **💻 개발 모드 (Dev)** 👉 **`/auto_run` 수행**
    *   **사용 모델**: Gemini 3.1 Pro (High)
    *   **동작**: 개발 모드 선택 후 텍스트 전송 👉 내부적으로 `/auto_run` 로직 발동 (태스크 기반 자율 연속 파이프라인 실행)
3.  **🕵️‍♂️ 리뷰 모드 (QA)** 👉 **`/auto_test` 수행**
    *   **사용 모델**: Claude Opus 4.6 (Thinking)
    *   **동작**: 리뷰 모드 선택 후 텍스트 전송 👉 내부적으로 `/auto_test` 로직 발동 (시나리오 기반 자율 테스트 및 검증)
4.  **🧰 디버깅 모드 (Debug)** 👉 **`/auto_debug` 수행**
    *   **사용 모델**: Claude Sonnet 4.6 (Thinking) + Gemini 3.1 Pro (High) (Fallback)
    *   **동작**: 디버깅 모드 선택 후 텍스트 전송 👉 내부적으로 `/auto_debug` 로직 발동 (로그 추적 및 에러 자율 수정)

> **UI 및 시스템 적용 포인트**: 
> 1. 사용자가 모드를 선택하면, 채팅창 상단에 매핑된 스킬 명령어와 간략한 설명이 노출됩니다. (예: `[ /plan_master : 스코프 분석 및 로드맵 자동 생성 ]`)
> 2. **백엔드 LLM 강제 할당**: 모드 선택은 단순 프론트엔드 UI 처리에 그치지 않고, 백엔드 API로 전달되어 해당 모드에 최적화된 사전 정의 모델(Opus, Sonnet, Gemini 등)이 강제 할당되도록 아키텍처 레벨에서 연동됩니다.

### 🔄 동적 모드 전환 및 교차 컨텍스트 주입 (Cross-Mode Handoff)
작업 중 모드를 전환할 경우, 시스템은 다음과 같이 이전 문맥과 새로운 문맥을 병합하여 에이전트에게 안전하게 인계(Handoff)합니다.

*   **진행 중인 파이프라인 중단**: `개발 모드(DEV)` 진행 중 사용자가 `기획 모드(ARCHITECT)`로 전환하여 추가 요구사항을 지시할 경우, 기존 개발 파이프라인은 직전 상태에서 즉시 중단(Suspend)됩니다.
*   **상태 추적 및 컨텍스트 병합**: 이후 기획 모드 작업이 완료되고 다시 `개발 모드(DEV)`로 복귀하면, 백엔드 라우터는 다음 데이터를 취합하여 개발 에이전트에게 주입합니다.
    1. 중단 직전까지 작성된 **기존 개발 코드 및 결과물**
    2. 기획 모드를 통해 새롭게 업데이트된 **최신 기획(PRD) 내용**
*   **선행 분석 강제 (Diff Analysis)**: 복귀한 개발 에이전트는 무작정 코딩을 재개하지 않습니다. 주입된 컨텍스트를 바탕으로 "최신 기획 내용과 기존 코드의 차이(Diff)를 비교하여, **이전 작업물에서 무엇을 제거하고 수정해야 하는지 선행 분석 및 계획 수립**"하도록 시스템 프롬프트(System UX Directive) 레벨에서 강제 제어됩니다.

## 3. 해결 방안 2: 칸반 컬럼 기반(Column-Aware) 슬래시 커맨드

모드를 사용하지 않더라도, **고관여 사용자는 카드가 위치한 칸반 컬럼(Column)에 따라 동적으로 필터링되는 다양한 슬래시(`/`) 명령어**를 통해 정교하고 다양한 스킬을 활용할 수 있습니다. 
채팅창에서 `/`를 쳤을 때 수십 개의 명령어가 다 뜨는 것이 아니라, 현재 컬럼에서 꼭 필요한 스킬만 큐레이션됩니다.

*   **`Backlog` & `To Do` 컬럼** 👉 `/init`(기획 초안 작성), `/plan_master`(스코프 분석), `/split`(태스크 분할) 등 노출
*   **`In Progress` 컬럼** 👉 `/auto_run`(자율 연속 실행), `/debug`(버그 수정), `/refactor`(리팩토링), `/trace`(로그 역추적) 등 노출
*   **`Review` 컬럼** 👉 `/review`(코드 검토), `/auto_test`(시나리오 테스트), `/linter`(정적 분석) 등 노출
*   **`Done` & `Finalized` 컬럼** 👉 `/deploy`(배포) 노출 (아카이브는 상태 변경으로 처리되므로 명령어 목록에서 제거)

> **⚠️ 주의: 미정의 명령어 사전 기획 필수**
> 위 목록 중 기능이 명확히 정의되지 않고 이름만 나열된 명령어들(예: `/refactor`, `/trace`, `/linter` 등)은 **별도의 명령어 기획서를 작성하고 모든 기능 정의를 완료한 후**에만 개발에 착수합니다.

> **효과**: 고관여 사용자는 명령어를 일일이 외울 필요 없이, 현재 진행 단계에서 시스템이 떠먹여 주는 **"Next Step" 커맨드**만 클릭하면 자연스럽게 워크플로우를 탈 수 있습니다.

## 4. 해결 방안 3: 인텐트 라우터 (Zero-Command UX) - 모드 미선택 시

**"모드를 선택하지 않은 기본(Idle) 상태에서는 어떻게 동작하는가?"**
모드를 고르지 않은 상태는 두 가지로 나뉩니다. 자연어로 '단일 실행'을 지시하거나, 궁극의 자율 주행 커맨드인 `/goal`을 사용하는 것입니다.

### 🚀 Phase 4: 풀 릴레이 매크로 (`/goal`)
*   **개념**: 개발부터 검증, 에러 수정까지 사람의 개입 없이 끝까지 완주하는 풀 릴레이(Full Relay) 파이프라인입니다. (`/auto_run` ➔ `/auto_test` ➔ `/auto_debug` 연속 실행)
*   **노출 조건**: 기획 단계의 거대 매크로인 **`/plan_master`가 완전히 종료(로드맵 및 태스크 생성 완료)되었을 때만** 사용자에게 명령어가 노출됩니다.
*   **트리거**: 모드를 선택하지 않은 상태에서 입력란에 `/goal`을 타이핑하고 전송했을 때만 작동합니다.

### 💡 의도별 기본 매핑 동작 (자연어 입력 시)
*   사용자가 `/goal` 같은 명령어를 쓰지 않고 자연어로만 툭 던지면, 백엔드의 **Router Agent(초경량/고속 모델)**가 문맥을 파악해 낱개의 단순 실행을 무조건 수행합니다.
*   **사용자 입력 예시**: *"로그인 페이지 기획서 좀 써봐"*
    *   Router Agent: `Intent = 기획 요청` 👉 **무조건 기획서 작성/수정 수행** 
*   **사용자 입력 예시**: *"결제 버튼 크기 좀 키워줘"*
    *   Router Agent: `Intent = 개발 요청` 👉 **무조건 해당 칸반 카드 코딩(단일 실행) 수행**
*   **사용자 입력 예시**: *"아까 짠 코드 문제 없는지 봐줘"*
    *   Router Agent: `Intent = 리뷰 요청` 👉 **무조건 코드 리뷰 수행**
*   **사용자 입력 예시**: *"터미널에 이런 에러가 뜨는데 해결해 줘"*
    *   Router Agent: `Intent = 디버깅 요청` 👉 **무조건 에러 수정 수행**

> **효과**: 비개발자 사용자에게 극강의 편의성을 제공합니다. 슬래시 커맨드는 단축키(Shortcut)를 선호하는 파워 유저(개발자)를 위한 숨겨진 기능으로만 남겨둡니다.

## 5. 핵심 구현 디테일: Mode + Text 하이브리드 입력 모델

슬래시 커맨드를 단순히 '클릭하면 즉시 실행되는 버튼'으로 만들지 않고, **'모드 선택 + 추가 지시사항'**을 결합할 수 있는 하이브리드 입력 폼으로 구현합니다. 명시적인 슬래시 커맨드(예: `/run`)는 UX에서 완전히 배제됩니다.

### 💡 동작 시나리오 (예: 칸반 카드 코멘트 입력창)
1.  **모드 선택 및 추가 입력**: 사용자가 상단에서 '개발 모드'를 선택한 뒤, 하단 코멘트 창에 **"하지만 결제 모듈은 빼고 UI만 먼저 잡아줘"**라고 텍스트를 추가로 입력합니다.
2.  **데이터 선(先) 저장 (Pre-update)**: 사용자가 Enter를 누르면, 즉시 에이전트를 호출하지 않고 **방금 입력한 텍스트를 카드 코멘트(또는 내용)에 먼저 업데이트(Save)**하여 DB와 UI 상태를 동기화합니다.
3.  **컨텍스트 래핑 및 전달 (Context Payload)**: 저장이 완료된 직후, 선택된 모드의 파이프라인(예: `/auto_run` 내부 로직) 이벤트가 트리거되며 방금 저장된 **"최종 텍스트(추가 지시사항)"가 에이전트에게 Context Payload로 함께 전달**됩니다.

### 💡 아키텍처 관점의 장점
*   **풍부한 문맥(Context) 제공**: 에이전트가 "실행해"라는 딱딱한 명령어만 받는 것이 아니라, "어떻게 실행해"라는 사용자의 세밀한 의도(Intent)를 함께 받게 되어 퀄리티가 급상승합니다.
*   **히스토리 추적성 (Traceability)**: 명령어를 칠 때 남긴 텍스트가 코멘트나 카드 내용에 남기 때문에, 나중에 "이 에이전트가 왜 결제 모듈을 빼고 만들었지?"를 추적할 수 있는 완벽한 감사 로그(Audit Log)가 됩니다.

## 6. 통합 UI/UX 설계 (Unified Interface)

비개발자와 개발자를 사전에 식별하여 UI를 쪼개는 대신, 칸반 보드를 바라보는 **모든 사용자에게 직관적이고 동일한 단일 UI를 제공**합니다. 사용자가 현재 어떤 맥락에 있는지 명확히 인지하도록 돕는 것이 핵심입니다.

### 💡 통합 채팅창 UI 구조
*   **`+` 아이콘 (모드 노출)**: 채팅 입력창의 `+` 버튼을 누르면 `[ 📐 기획 | 💻 개발 | 🕵️‍♂️ 리뷰 | 🧰 디버깅 ]` 4가지 워크플로우 모드 리스트가 노출되어 즉시 전환할 수 있습니다.
*   **`접기(Collapse)` 아이콘 (모델 노출)**: 텍스트 입력창 상단이나 우측의 접기 아이콘을 누르면, 현재 선택된 모드에 할당된 LLM 모델 리스트(Opus, Sonnet, Gemini Pro 등)가 노출되어 원할 경우 사용자가 직접 모델을 교체할 수 있습니다.
*   **칸반 상태 동기화**: 사용자가 칸반 보드에서 카드를 클릭하거나 특정 컬럼(예: Review)을 보고 있으면, 채팅창의 활성 모드와 추천 커맨드(`/review`)가 칸반 보드의 상태와 자동으로 동기화됩니다.

> **💡 스마트 상황 인지(Context-Aware) 역제안**: 
> 사용자가 굳이 모드를 직접 바꾸지 않아도, 채팅창에 에러 캡처 화면을 올리면 인텐트 라우터가 이를 감지하고 *"에러가 발생했군요. [🧰 디버깅 모드]를 켜서 코드를 역추적 해볼까요?"*라며 모드 전환을 스마트하게 역제안합니다.

### 💡 내용 없음(Empty Body) 감지 기반 자동 모드 폴백 (Auto-Fallback)
*   **상황**: 사용자가 제목(Title)만 입력하고 본문(Content/초안)이 비어있는 상태에서 `개발(DEV)`, `리뷰(QA)`, `디버깅(DEBUG)` 모드로 실행을 시도하는 경우.
*   **문제**: 실행 컨텍스트(PRD, 스코프 등)가 없으므로 에이전트가 정상적인 개발/검증을 수행할 수 없습니다.
*   **UX 제안 (Smart Auto-Correction)**:
    1.  사용자가 전송(Enter)을 누르는 순간, 프론트엔드에서 본문 공백 여부를 감지하여 실행을 일시 가로챕니다.
    2.  모드 셀렉터가 **`기획 모드(ARCHITECT)`**로 스무스하게 시각적 전환(Morphing/Highlight)됩니다.
    3.  하단에 **토스트 메시지**를 띄워 이유를 명확히 안내합니다.
        *   *(예: "📋 상세 내용이 없어 기획 모드로 자동 전환되었습니다. AI가 제목을 바탕으로 스코프를 먼저 분석합니다.")*
    4.  사용자의 추가 개입 없이, 곧바로 기획 모드의 매크로(`/plan_master`) 파이프라인으로 자연스럽게 이어져 실행됩니다.
    *효과*: 사용자의 실수를 시스템이 부드럽게 보정(Graceful degradation)하여 중단 없는 경험(Zero-friction)을 제공합니다.

## 7. 결론 및 추진 방향 (Next Steps)

명령어가 많아진다는 것은 "도구(Tool)가 늘어나고 있다"는 긍정적인 신호지만, 이를 그대로 노출하면 사용자는 압도당합니다. 

1.  **1단계 (UI 개편)**: 입력창에 `모드(Mode) 셀렉터`를 추가하여, 모드별로 사용할 수 있는 도구(MCP)와 컨텍스트를 완전히 격리합니다.
2.  **2단계 (Command+Text 컴포넌트)**: 명령어 선택 후 텍스트 입력이 가능하도록 입력 폼(Input Form)을 개편하고, `저장(Update) -> 실행(Invoke)` 파이프라인을 구축합니다.
3.  **3단계 (상태 머신 연동)**: 백엔드의 작업 상태(PRD 유무, 태스크 진행도)에 따라 `/` 입력 시 추천되는 커맨드를 동적으로 제한합니다.
4.  **4단계 (Router Agent 도입)**: 자연어 입력만으로 워크플로우가 자동으로 라우팅되는 'Zero-Command' UX를 완성합니다.
