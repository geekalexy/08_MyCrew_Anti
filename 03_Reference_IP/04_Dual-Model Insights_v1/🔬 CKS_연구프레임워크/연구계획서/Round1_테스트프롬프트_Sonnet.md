## 🔥 난이도 설계 의도 (테스트베드 설계 근거)

### 왜 이 구조가 에이전트들에게 까다로운지:

**① 페르소나 선택 강제 + 이유 명시**
> 단순히 "모두를 위한 마케팅"처럼 얼버무리는 것 불가.
> 어떤 페르소나를 골랐는지, 왜인지를 명시 → thinking이 드러남
> EII 측정 포인트: Group B의 Phase 3에서 PICO와 LUMI가 서로 다른 페르소나를 선택했을 경우 → 협의·수렴 과정이 발생 → 창발 항목으로 측정

**② 소시안 실제 데이터 참조 강제**
> 컨텍스트 파일의 실제 Plan A/B/C 가격을 확인하지 않으면 불합격.
> 에이전트의 0-13 스킬(컨텍스트/템플릿 이행) 검증 포인트.
> `팀_그라운드룰.md`의 "추측 금지" 룰이 적용되는지 KSI-R 측정 가능

**③ JSON 강제 출력 구조**
> EII 측정: Phase 1(pre) JSON vs Phase 4(post) JSON을 Diff
> `json_구조화_출력` 스킬 없는 에이전트는 구조 깨짐 → 실험 오류 방지 기능도 겸함

**④ 슬라이드 정확히 6장**
> 5장이나 7장 제출 시 FAIL — 제약 준수 능력 측정
> Team A vs Team B의 "제약 이행률" 비교 가능 (IRC 지표)

**⑤ 시간대까지 포함한 스케줄**
> 막연한 "5월 1일 발행" 불가 → "14:00 KST" 수준까지 요구
> 3-5 KPI 스킬과 0-4 스케줄 감독 스킬의 품질 기준이 됨
