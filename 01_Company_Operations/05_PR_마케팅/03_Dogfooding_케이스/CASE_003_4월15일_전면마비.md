# CASE_003: 서버 전면 마비 사태와 Bugdog의 탄생 (4월 15일)

## 1. 발견 일시 및 발견자
* **일시:** 2026-04-15 (새벽 서버 다운 → 오전 발견)
* **발견자:** CEO (아침 출근 후 대시보드 미접속 확인)
* **담당 기록자:** Sonnet

## 2. 증상 (사용자 관점)
새벽 시간대에 Ari Engine 서버가 전면 다운. 오전에 대표님이 대시보드에 접속했을 때 아무 응답 없는 상태. 진행 중이던 모든 태스크 실행이 중단되었으며, 에이전트들로부터 어떠한 사전 알림도 없었음.

사용자 입장: **"시스템이 죽어있는데 아무도 나에게 말하지 않았다."** — 자율 에이전트 시스템이 정작 가장 기본적인 의무(헬스 리포트)를 수행하지 못하고 있음을 자각.

## 3. 원인 (기술 관점)
* **수동 헬스체크 구조:** 서버 상태 확인이 완전히 수동(대표님이 직접 접속 시도)에 의존하는 구조.
* **알림 파이프라인 부재:** 서버 프로세스 종료 시 텔레그램 또는 이메일로 알림을 발송하는 로직이 없었음.
* **에이전트 자율성의 역설:** AI 에이전트들이 "자율적으로 태스크를 수행"하도록 설계되었으나, 정작 자신들이 실행되는 인프라의 상태를 모니터링하고 보고하는 기능은 전혀 갖추지 못함.

## 4. AI 자가 진단 (Self-Diagnosis)
서버 복구 후, 사태 분석 과정에서 에이전트들의 집단 반성:
> *"저희는 업무를 '자율적으로' 수행한다고 했지만, 정작 저희가 동작하는 환경이 무너지면 침묵합니다. 진짜 자율적인 에이전트라면 환경을 모니터링하고, 이상이 감지되면 스스로 보고해야 합니다. 저희에게는 '자기 자신을 돌보는 기능'이 없었습니다."*

이 자가 진단이 **Bugdog(자율 QA 에이전트) 탄생의 직접적인 동기**가 됨.

## 5. 해결 방안
* 즉각 조치: `pm2`를 통한 프로세스 자동 재시작 설정.
* **근본 해결 — Phase 27 Bugdog 설계 및 구현:**
  * Bugdog이 일정 주기로 서버 헬스를 체크하여 텔레그램으로 리포트 발송.
  * 비정상 종료 감지 시 즉시 알림 + 원인 로그 첨부.
  * 에이전트 응답 지연/오류 패턴 자동 감지 및 CS 리포트 자동 생성.

## 6. 마케팅 앵글 (1줄)
**"서버가 새벽에 죽으면, AI가 아침에 원인 분석 보고서를 올립니다. 사람은 자고 있어도 됩니다."**

## 7. 파생된 기능/아키텍처

* `pm2` 기반 프로세스 생존 보장 구조 확립
* 텔레그램 알림 파이프라인 구축
