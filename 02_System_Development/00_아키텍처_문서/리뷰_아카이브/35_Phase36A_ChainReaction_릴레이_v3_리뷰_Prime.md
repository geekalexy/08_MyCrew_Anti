## 1. 🔴 리스크(Risk) 대응 방안

### Risk #1: v2 코드(triggerPipelineRelay)와의 충돌 및 제거 계획
*   **분석**: `server.js`에 존재하는 `triggerPipelineRelay`는 `pipeline_step` 컬럼을 1 증가시키며 다음 카드를 찾는 중앙 통제식(v2) 로직입니다.
*   **대응 및 전환 계획**: 
    *   **Phase 36-A(v3)에서는 해당 로직을 전면 폐기**합니다.
    *   다음 카드로의 이행은 오직 에이전트가 호출하는 `create_next_sprint_task` 도구에 의해서만 이루어집니다.
    *   프로젝트 생성 초기 카드를 포함한 모든 상황에 v3가 완벽히 호환되므로, 불필요해진 v2 파이프라인 탐색 로직은 삭제(Diet)하고, 오직 새 카드가 생성되었을 때 담당자에게 `dispatch`하는 이벤트 드리븐 역할만 남깁니다.

### Risk #2: `create_next_sprint_task` LLM 의존도 및 서버사이드 안전망
*   **분석**: LLM이 제목, 내용, 담당자, sprint_no 등을 완벽하게 맞춰서 API를 호출할 것이라 기대하면 오류가 발생합니다.
*   **대응 (엔진 격리)**:
    *   LLM(에이전트)은 **새 카드의 제목, 내용(참조 태그 포함), 그리고 희망 담당자(Role)** 3가지만 넘깁니다.
    *   **서버 단 강제 주입**: `project_id`, `sprint_no`, `상태(IN_PROGRESS)`는 서버가 **현재 작업 중인 부모 카드**에서 무조건 자동으로 상속/주입합니다.
    *   **유효성 검증**: 사전에 정의된 `ALLOWED_IDS`를 검증하여 오류 발생 시 `dev_advisor`나 `dev_fullstack`으로 강제 보정(Fallback)하는 안전망을 구축합니다.

### Risk #3: 코드 리뷰어 교차 검증 (자기참조 차단)
*   **분석**: CTO가 코딩하고 자신(CTO)에게 리뷰를 맡기는 환각(Hallucination) 방지.
*   **대응**: 
    *   `create_next_sprint_task` 서버 API 로직 내부에 검증 룰을 추가합니다.
    *   만약 부모 카드의 작성자가 `dev_senior`(CTO)이고, 새 카드의 목적이 '리뷰'라면, 할당자(assignee)가 `dev_senior`일 경우 API 단에서 차단하거나 `dev_advisor` / `dev_fullstack`으로 자동 변경합니다.

### Ari 워치독 V3 적용 교차 확인

기획서 v3.0 §3.2 (L54):
> *"Ari 워치독의 Level 1/2/3 보완 트리거 개입 로직은 V3에서도 그대로 유지하여, 파이프라인이 멈추지 않고 스스로 복구되도록 생명줄 역할을 수행합니다."*

V3에서 워치독이 감지해야 할 **신규 시나리오**:

| V2 감지 대상 | V3 추가 감지 대상 |
|:---|:---|
| Advisor 크래시 (STUCK) | **에이전트가 `create_next_sprint_task`를 호출하지 않고 DONE 처리한 경우** |
| 10분 Hard Timeout | **동일 (AdapterWatcher 유지)** |
| Advisor 재할당 미실행 (ORPHAN) | **다음 카드가 IN_PROGRESS로 생성됐으나 5분간 응답 없는 경우** |

**구현 시 워치독 감지 조건 보강이 필요하다.** 특히 "에이전트가 바통을 안 넘기고 끝나는" 시나리오는 v2에 없던 v3 고유 리스크이므로, `DONE 상태 전환 시 동일 sprint_no의 IN_PROGRESS 카드 존재 여부` 확인 로직을 추가할 것.
