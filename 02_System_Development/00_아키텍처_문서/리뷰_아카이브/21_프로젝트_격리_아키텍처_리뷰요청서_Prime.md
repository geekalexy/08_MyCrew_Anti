## 1. 리뷰 요청 배경 (Context)
최근 기획 에이전트(NOVA)에게 모기업 '소시안' 관련 마케팅 글을 지시했을 때, 마이크루(MyCrew) 시스템 스펙을 쏟아내는 **컨텍스트 편향(Context Bias)** 오작동이 발생했습니다.
원인은 '하나의 전역(Global) 워크스페이스'에 마이크루 관련 개발 로그와 코드가 90% 이상 쌓여 있어, RAG 시스템이 문맥을 유추할 때 압도적인 정보량을 가진 마이크루 쪽으로 쏠렸기 때문입니다.

이를 해결하기 위해, 단순히 스코프 태그를 붙이는 수준을 넘어 시스템 전체(DB, 소켓, RAG 주입, UI)를 **'프로젝트 단위'로 완벽히 파티셔닝(격리)**하는 아키텍처 공사를 진행하고자 합니다. 코어(DB)를 건드리는 작업이므로 구현 전 Prime의 철저한 설계 검증을 요청합니다.

## 2. 제안하는 아키텍처 개요 (Proposed Architecture)

### 2.1. 데이터베이스(DB) 격리
* **변경:** `Task`, `TaskComment`, `SystemLog` (추가 예정) 테이블에 `project_id` 외래키 컬럼 추가.
* **효과:** 칸반 보드와 대시보드 로드 시 `?project_id=A` 쿼리로 완전 필터링하여 데이터 혼재 방지. 기존 데이터는 디폴트(Default) 프로젝트로 마이그레이션.

### 2.2. 소켓 통신 및 타임라인(UI) 격리
* **변경:** `server.js`의 브로드캐스트를 `io.emit`에서 `io.to(projectId).emit`으로 전환.
* **효과:** 클라이언트 우측 패널(LogDrawer)의 채팅과 타임라인이 오직 현재 접속한 프로젝트의 이벤트만 수신하고 렌더링함. (A 프로젝트에 접속 중일 때 B 프로젝트의 알람이 끼어들지 않음)

### 2.3. RAG 컨텍스트 및 AI 단기 기억(Memory) 격리
* **변경:** `context_injector.js`에서 `projectId`를 기반으로 읽어올 **RAG Root 폴더**를 스위칭.
    * *예: Project A(소시안 마케팅) -> `05_PR_마케팅` 폴더만 검색.*
    * *예: Project B(마이크루 엔진) -> `01_아리_엔진` 폴더만 검색.*
* **효과:** 이전 태스크나 코멘트 내역(`getCommentsWithTopology`)을 LLM에게 넘길 때도 현재 프로젝트 데이터만 넘기므로, "AI 자아와 지식의 완벽한 분리" 달성.

## 3. 중점 검토 요청 사항 (Questions for Prime)

이 아키텍처를 실무에 올리기 전, 다음 3가지 잠재적 리스크에 대한 고견을 부탁드립니다.

1. **글로벌 컨텍스트 공유 문제 (Global Assets):**
   * 프로젝트별로 RAG 폴더를 격리하면, `agents.json`, `SKILL.md`, `strategic_memory.md`처럼 **모든 에이전트가 공통으로 알아야 하는 메타 지식**의 접근성이 떨어질 수 있습니다. '글로벌 공통 풀'과 '프로젝트 전용 풀'을 RAG에서 섞어 쓸 때 발생할 수 있는 토큰 오버플로우나 충돌을 어떻게 우아하게 제어하는 것이 좋을까요?
2. **텔레그램 미니앱(TMA)과의 상태 동기화:**
   * 추후 Phase 32(텔레그램 미니앱)가 도입되면 데스크탑과 모바일에서 동시에 이벤트를 폴링하게 됩니다. 소켓 Room 분리가 모바일 세션에서도 상태 손실(Race Condition) 없이 완벽히 동작하기 위한 백엔드 세션 관리 팁이 있을까요?
3. **마이그레이션 전략 (Backward Compatibility):**
   * 이미 SQLite에 생성된 수많은 기존 `Task` 데이터를 날리지 않고 `project_id`를 삽입하기 위해, `db.run("ALTER TABLE...")` 이후에 레거시 카드들을 `DEFAULT_PROJECT`로 자동 바인딩하는 스크립트를 작성할 계획입니다. 혹시 SQLite 특성상 런타임에 발생할 수 있는 DB Lock 이슈 등 주의해야 할 점이 있습니까?

## 4. 코어 의존성 연관 분석 (Impact Analysis & 롤백 방지망)

과거 아키텍처 마이그레이션 중 "아리 채팅 및 타임라인 전역 버그"로 인해 롤백했던 뼈아픈 경험이 있습니다. 이를 반면교사 삼아, 본 아키텍처 변경이 시스템 전반에 미칠 연쇄 영향을 철저히 분석했습니다. Prime은 이 분석망에 누락된 지점이 없는지 검증해 주시기 바랍니다.

### 4.1. `TaskDetailModal.jsx` 및 칸반 카드 렌더링
* **의존성:** `tasks` API 응답 구조 의존.
* **리스크:** 백엔드에서 `tasks` 테이블 조회 시 `project_id`를 강제 필터링할 경우, 프론트엔드의 전역 상태(`projectStore` 등) 초기화 전에 API가 호출되면 텅 빈 배열(`[]`)이 반환되어 전체 UI가 붕괴될 수 있음.
* **방어 대책:** 프론트엔드 라우터/상태 관리자 레벨에서 `activeProjectId`가 세팅되지 않으면 Loading 화면을 띄우고, `undefined` 상태의 쿼리 요청을 원천 차단.

### 4.2. `LogDrawer` (타임라인 및 채팅 컴포넌트)
* **의존성:** `server.js`의 `broadcastLog()` 함수 및 소켓 이벤트(`log_added`) 수신 구조.
* **리스크:** 소켓 채널을 `io.emit`에서 `io.to(projectId).emit`으로 일괄 변경 시, 기존에 `projectId` 인자 없이 `broadcastLog(message)`를 호출하던 수십 개의 레거시 백엔드 로직들이 소켓 브로드캐스트를 잃어버리는(블랙홀 현상) 버그 발생 확률 100%.
* **방어 대책:** `broadcastLog` 함수 시그니처를 변경하되, `projectId`가 명시되지 않은 레거시 호출은 임시로 전역 브로드캐스트(`io.emit`)로 fallback하도록 유예(Graceful Degradation) 로직을 적용.

### 4.3. `executor.js` (아리 엔진 실행기) 및 RAG 주입
* **의존성:** `context_injector.js`를 통한 메타 데이터 로드.
* **리스크:** `Task`를 실행할 때 `project_id` 필드가 없는 구형 카드의 경우, RAG 주입기가 참조 폴더를 찾지 못해 컨텍스트 없이 빈깡통으로 LLM 호출이 날아감. (결과: 아리 지능 저하 버그)
* **방어 대책:** `project_id`가 null인 태스크를 실행할 때는 강제로 `default` 폴더 경로를 주입하는 방어 코드 필수 작성.
