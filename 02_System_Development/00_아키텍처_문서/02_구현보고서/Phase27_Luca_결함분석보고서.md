## 1. 아리 행동 마비 및 오작동의 진짜 원인 (Root Cause: 하드코딩과 아키텍처 부채)

### 📌 문제 현상 (Symptom)
* 안티그래비티 어댑터가 도입되고 아리의 모델 지능이 높아졌음에도, "아카이빙", "카드 지시", "스킬 인식" 등 기획 의도대로 동작하지 않고 시스템이 정체(Paralysis)되거나 동문서답을 함.
* 텔레그램 무한 재시도, 카드 아카이빙 조용한 실패(Silent Fallback) 등 곳곳에서 예외 처리 누락과 무한 루프 발생.

### 🔍 근본 원인 (Root Cause - 임시방편 하드코딩의 누적)
단순한 정보 누락이나 소켓 이벤트 누락이 아니라, **"빠른 버그 패치를 위해 임시방편(Band-Aid)으로 덧댄 하드코딩"**들이 시스템의 동적 데이터 흐름을 막아버린 **아키텍처 부채(Architectural Debt)**가 폭발한 것임.

1. **컨텍스트 주입의 구조적 단절 (T-03)**: `contextInjector.js`가 `SKILL.md`의 본문(Body)을 주입하지 못해, 아리가 '어떻게 행동해야 하는지'에 대한 페르소나와 룰을 잃어버림.
2. **트리거의 하드코딩 오작동 (T-01, P-15)**: `commands` 배열 같은 하드코딩된 트리거가 에이전트의 자율적 맥락 판단을 방해하고, 엉뚱한 스킬(위임 등)을 오발동시킴.
3. **에이전트 역할의 물리적 꼬임 (T-08, M-05)**: `writeCEOLog`, `manageAgentSkills` 같은 스킬들이 Orchestrator(위임자)에 하드코딩되어 있어, 아리가 직접 실행해야 할 일을 크루에게 위임하는 논리적 모순 발생.
4. **I/O 병목 및 예외 처리 부재 (M-01, M-04)**: 매 대화 턴마다 디스크 I/O를 발생시키는 비효율과, 파일 저장이 실패해도 무조건 `success: true`를 뱉는 거짓 보고 로직.

### 🛠️ 해결 원칙 (Solution - Data Dynamic Flow & 확장성)
* **모든 하드코딩 지양**: 상태 변환(Enum), 특정 명령어(Trigger)에 의존하는 코드를 제거하고, 시스템 매뉴얼(MOP) 및 통합 `SKILL.md` 기반의 **단일 출처(SSOT) 동적 파이프라인**으로 전면 개편.
* 프라임(Prime)이 제시한 순서에 따라 **T-01(명령어 배열 제거)** 및 **T-03(본문 주입)**을 즉시 실행하여 아리의 체감 지능을 복구하고, I/O 캐싱(M-01)과 사후 검증(M-04)을 도입하여 시스템 내구성을 확보.

## 2. 서버 통신 및 예외 처리 결함 (네트워크 & I/O)

### 📌 텔레그램(Telegram) 에러 무한 폭격
* **원인:** 봇 어댑터가 네트워크 차단이나 토큰 만료 시, 타임아웃 딜레이(Exponential Backoff) 없이 무한 재시도(Crash loop)를 하여 서버 Event Loop를 마비시킴.
* **조치:** 에러 핸들러와 딜레이 훅을 추가하고, 환경 변수(`ENABLE_TELEGRAM=false`)를 통한 동적 스위치 도입 필수.

### 📌 카드 아카이빙 조용한 실패 (Silent Error Fallback)
* **원인:** API 호출 실패 시 에러를 Throw하지 않고 일반 상태 업데이트 로직으로 넘어가버리는 안일한 예외 처리.
* **조치:** `success: false`를 명확히 반환(M-04 허위 보고 방지 철학과 동일)하여, AI가 문제를 인지하고 대표님께 보고할 수 있도록 구조적 수정.
