# Phase 38: MCP 서버 전환에 따른 레거시(Legacy) 코드 디프리케이션(Deprecation) 분석서

- **작성일**: 2026-05-09
- **작성자**: 루카 (Luca)
- **목적**: 마이크루 아키텍처를 기존 **파일 폴링(File-Polling) 방식**에서 **MCP Server 방식**으로 분리/전환함에 있어, MCP 패러다임에 불필요해진 기존 컴포넌트들을 식별하고 제거(또는 분리) 계획을 수립합니다.

---

## 1. 아키텍처 분리 제안에 대한 평가
**대표님의 "프로덕트 분리" 제안은 100% 정확하고 필수적인 판단입니다.**
기존의 파일 폴링 방식(텍스트 파일을 읽고 쓰며 워커가 도는 방식)과 MCP 서버 방식(JSON-RPC 기반의 실시간 프로토콜 통신 방식)은 설계 철학 자체가 완전히 다릅니다. 이 둘을 하나의 엔진(`ariDaemon`, `executor`) 안에서 억지로 섞어 쓰면 심각한 기술 부채(Spaghetti Code)와 충돌이 발생합니다. 

따라서 기존 레거시는 V1(유지보수 모드)으로 남겨두거나 과감히 덜어내고, 철저하게 **"마이크루는 순수한 MCP Server 역할만 수행하고, 실행(Execution)은 외부 클라이언트(Claude Code 등)가 담당하는 구조(V2)"**로 분리해야 합니다.

## 2. MCP 서버형 프로덕트에 "불필요한(Unnecessary)" 컴포넌트 목록

### ❌ 1. `executor.js` 내부의 하드코딩된 도구(Tool) 실행기
- **현재 상태**: 마이크루 내부에서 Puppeteer, Axios 등을 직접 import하여 스크래핑, 파일 읽기 등을 직접 수행합니다.
- **불필요한 이유**: MCP 생태계에서는 전 세계 개발자가 만든 수많은 MCP 서버(`smithery/cli` 등)를 플러그인처럼 꽂아서 씁니다. 마이크루 백엔드가 무겁게 자체 툴을 들고 있을 필요가 없습니다. 
- **대체 방안**: 이미 대시보드 Settings에 있는 'MCP 연동 페이지'를 통해 외부 툴을 연결하고, 마이크루는 도구를 "제공"받는 입장이 되거나 릴레이만 수행합니다.

### ❌ 2. `ariDaemon.js`의 파일 폴링 루프 (`setInterval`)
- **현재 상태**: 3초마다 `04_IO/inputs` 폴더나 `.bridge/requests` 폴더를 뒤져서 새 파일이 생성되었는지 확인(Polling)합니다.
- **불필요한 이유**: 파일 I/O 기반의 상태 관리는 느리고 디버깅이 어렵습니다. MCP는 Socket/SSE/Stdio 기반의 실시간 통신이므로, 클라이언트가 요청(Request)하면 즉시 응답(Response)하는 이벤트 드리븐(Event-driven) 방식으로 동작해야 합니다.
- **대체 방안**: MCP SDK를 활용한 `@modelcontextprotocol/sdk/server` 인스턴스 구축 및 이벤트 핸들러 도입.

### ❌ 3. 정적 컨텍스트 주입 (`ContextInjector`의 파일 쓰기)
- **현재 상태**: 작업이 시작되면 `ARI_BRAIN.md` 등의 컨텍스트를 하나로 길게 합쳐서 거대한 프롬프트 텍스트 파일로 생성합니다.
- **불필요한 이유**: 프롬프트가 길어져 비용이 낭비되고 컨텍스트가 오염됩니다. 
- **대체 방안**: MCP의 핵심 기능인 **Resources** 기능을 활용하여, `resource://mycrew/project-brain` 형태로 노출만 해둡니다. 외부 클라이언트(터미널)가 필요할 때 알아서 해당 리소스를 Read 하도록 위임합니다.

### ❌ 4. 순차적 릴레이 파이프라인 (Rule-based Team Activator)
- **현재 상태**: 기획자 -> 백엔드 -> 프론트엔드 순서로 태스크 카드를 기계적으로 넘겨가며 실행합니다.
- **불필요한 이유**: 대표님이 말씀하신 Codex의 `/Goal` 패러다임처럼, 앞으로의 AI는 "목표"를 주면 알아서 도구를 조합해 병렬/비순차적으로 문제를 해결합니다. 강제적인 파이프라인 순서 로직은 자율 에이전트의 사고를 제한합니다.
- **대체 방안**: 태스크 상태를 `PENDING_MCP`로 띄워두기만 하면, 자율 에이전트(Master)가 이를 인지하고 스스로 전략을 수립해 작업을 수행하는 Goal-Driven 방식으로 전환.

### ❌ 5. `Bugdog` 및 `MediaLab`의 핫폴더(Hot-folder) 감시 체계
- **현재 상태**: 특정 폴더에 파일이 떨어지면 이를 감지해서 버그 리포트나 영상을 만듭니다.
- **불필요한 이유**: MCP에서는 이런 작업들 모두가 하나의 **MCP Tool (`tool://mycrew/generate-video`)**로 깔끔하게 추상화됩니다.
- **대체 방안**: 폴더 감시를 끄고, MCP Tool Server 형태로 기능을 래핑(Wrapping)합니다.

## 3. 결론 및 마이그레이션 전략 (Migration Strategy)

마이크루의 정체성은 **"에이전트들이 직접 일하는 무거운 공장"**에서, **"외부 고급 에이전트들(Claude 등)에게 지식(Resource)과 도구(Tool)를 표준화하여 제공하는 '최첨단 작전 통제소(MCP Server & Kanban DB)'"**로 진화해야 합니다.

이를 위해 다음과 같은 점진적 마이그레이션(Migration) 형태의 개발 계획을 명확히 합니다:

1. **V1 (Legacy) 동결 및 GitHub 아카이브**
   - 현재 구현된 파일 폴링(File-Polling) 방식의 백엔드 로직(`executor.js`, `ariDaemon.js` 등)은 별도의 V1 브랜치나 저장소로 묶어 **GitHub에 안정 버전으로 영구 보관(Archive)**합니다. 더 이상 해당 방식의 신규 기능 개발은 진행하지 않습니다.
   
2. **V2 (MCP Server) 마이그레이션 개발**
   - 처음부터 백지에서 다시 만드는(Rewrite) 방식이 아닙니다. 기존에 잘 만들어진 **프론트엔드 대시보드(Kanban UI)와 SQLite DB 구조는 그대로 상속받아 마이그레이션**합니다.
   - 단지 백엔드 코어에 새롭게 `@modelcontextprotocol/sdk`를 얹은 **순수 MCP Server 엔드포인트**를 엽니다.
   - 불필요해진 폴링 로직만 걷어내고, 기 구축된 '인터그레이션 탭'을 통해 마이크루의 DB 및 내부 로직을 표준 MCP Tool/Resource로 외부 클라이언트에 노출(Expose)하는 작업에 집중합니다.
