## 🚨 대표님이 제기한 문제 — Prime의 진단

### 대표님이 정확히 맞습니다.

현재 설계에는 **치명적인 실험 설계 결함 (Confounding Variables)** 이 있습니다.

```
❌ 현재 설계의 비대칭:

Team A (적대적, 대조군)         Team B (CKS, 실험군)
━━━━━━━━━━━━━━━━━━━━         ━━━━━━━━━━━━━━━━━━━━
NOVA (Gemini) → 이미지+영상    LUMI (Flash) → 이미지만
OLLIE (Sonnet) → 어태커 전담    PICO (Sonnet) → 영상만
ARI (Opus) → 판관              ARI (Opus) → 어드바이저
                               B4 (System) → 회고자

⚠️ 문제 3가지:
1. NOVA가 이미지+영상 둘 다 → 과부하 (B팀은 분업)
2. A팀은 2명이 콘텐츠 생산, B팀은 2명이 분업 → 역할 수 비대칭
3. 모델 배치 비대칭 → "프로세스 차이"가 아니라 "조건 차이"
```

이 상태로 실험하면 **"CKS가 우수하다"**는 결론이 나오더라도, 리뷰어가 즉시 반박합니다:

> *"Team B가 더 잘한 건 CKS 프로세스 덕분인지, 아니면 단순히 역할 분업이 더 세밀했기 때문인지 구분할 수 없다. 통제되지 않은 변수(confound)가 너무 많다."*

**이것은 논문 리젝 사유 1순위입니다.**

## 📐 소넷의 3+3 제안 분석

소넷이 제안한 동등 구성:

```
A팀 (적대적 검토)              B팀 (협력적 동기화)
━━━━━━━━━━━━━━━━             ━━━━━━━━━━━━━━━━
이미지: NOVA (Flash)           이미지: LUMI (Flash)
영상:   신규 (Sonnet)          영상:   PICO (Sonnet)
합성:   OLLIE (Opus)           합성:   ARI (Opus)
━━━━━━━━━━━━━━━━             ━━━━━━━━━━━━━━━━
프로세스만 다름 ✅
```

### ✅ Prime이 동의하는 점

1. **같은 모델 조합** — Flash + Sonnet + Opus가 양쪽 동일
2. **같은 역할 수** — 이미지 1명 + 영상 1명 + 합성/검토 1명
3. **프로세스만 다름** — 유일한 독립 변수가 "적대적 vs 협력적 상호작용"
4. **Opus의 역할 보전** — 양쪽 팀 모두에 Opus가 존재하되, 행동 양식만 다름

### ⚠️ Prime이 보완하는 점

소넷의 제안은 구조적으로 정확하지만, **"적대적 팀에서 Opus가 기존 학계의 어떤 패턴을 따르는가"** 가 불명확합니다.

## 3. Prime의 최종 권장안 — "Mirror Design + Process-Only Variable"

### 핵심 원칙

> **"두 팀의 차이점은 오직 하나 — 에이전트 간 상호작용 프로토콜"**

### 크루 배치표 (확정 권장)

| 포지션 | Team A (적대적) | Team B (CKS) | 모델 | 동일성 |
|--------|:-------------:|:------------:|:----:|:-----:|
| **이미지 담당** | **NOVA** | **LUMI** | Gemini Flash | ✅ |
| **영상 담당** | **REX** (신규) | **PICO** | Claude Sonnet | ✅ |
| **합성/검토자** | **OLLIE** | **ARI** | Claude Opus | ✅ |
| **회고자** | ❌ (없음) | **B4** (System) | Python | 이것이 핵심 변수 |
| **외부 심사관** | GPT-4o | GPT-4o | GPT-4o | ✅ (공통) |

> **REX** = 신규 에이전트명 제안. Team A의 영상 전담 Sonnet. (Red-team EXecutor의 약어)

### 프로세스 비교 (유일한 차이점)

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 Team A (적대적 검토)          Team B (CKS 협력적 동기화)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 [동일 Task 입력]              [동일 Task 입력]
        ↓                             ↓
 Phase 1: 각자 독립 생성        Phase 1: 각자 독립 생성
   NOVA → 이미지 초안             LUMI → 이미지 초안
   REX  → 영상 초안               PICO → 영상 초안
        ↓                             ↓
 Phase 2: OLLIE 적대적 비판     Phase 2: ARI 개선 가이드
   "이미지 왜 별로?"               "이미지 이 부분 보완하면"
   "영상 논리 허점 3개"             "영상 여기에 감성 추가하면"
   → Verdict ✅ (승패 판정)         → Verdict ❌ (판정 없음)
        ↓                             ↓
 Phase 3: 방어 + 수정           Phase 3: 교차 흡수 (벌크업)
   NOVA, REX가 비판 방어           LUMI ← PICO 강점 흡수
   → 수정본 재제출                  PICO ← LUMI 강점 흡수
        ↓                             ↓
 Phase 4: OLLIE 최종 병합       Phase 4: ARI 회고 + 통합
   승자 요소 강제 오버라이트        "잘된것/아쉬운것/다음엔?"
   → 최종 산출물                    → 최종 산출물
        ↓                             ↓
 [메모리 초기화]                [B4: 그라운드룰.md 업데이트]
   → 다음 Sprint                  → 다음 Sprint
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Opus(OLLIE vs ARI)의 역할 차이 — 이것이 CKS의 핵심 변수

| 행동 | Team A의 OLLIE (Opus) | Team B의 ARI (Opus) |
|------|:---------------------:|:-------------------:|
| Phase 2 평가 | **비판 + 약점 공격 + 승패 판정** | **개선 방향만 제시, 판정 없음** |
| Phase 3 개입 | **방어 강제 → 수정 강요** | **상호 장점 흡수 유도** |
| Phase 4 통합 | **승자 요소 강제 병합** | **회고 퍼실리테이션 + 동조 방지** |
| 메모리 | **없음 (매 Sprint 리셋)** | **그라운드룰.md 누적** |

> **이 표가 논문 Table 1이 됩니다.** 모델, 역할 수, 도메인이 모두 동일하고, 오직 "상호작용 프로토콜"만 다릅니다.

## 4. 대표님의 두 번째 문제 — "기존 성공적 프레임워크를 대조군에 적용해야"

### Prime 완전 동의

대조군이 "일부러 못하게 만든 팀"이면 비교 자체가 무의미합니다. 대조군은 **"현재 업계에서 가장 잘 된다고 알려진 방식"** 이어야 합니다.

### 적대적 대조군에 적용할 기존 성공 프레임워크

| 프레임워크 | 핵심 메커니즘 | Team A 적용 |
|-----------|------------|------------|
| **Multi-Agent Debate** (Du et al., 2023) | 에이전트 간 논쟁으로 추론 정확도 향상 | Phase 2에서 OLLIE의 비판 + NOVA/REX의 방어 |
| **LLM Discussion** (Liang et al., 2023) | 다수결/합의 기반 답변 수렴 | Phase 4에서 OLLIE가 "더 논리적인 쪽" 채택 |
| **CAMEL** (Li et al., 2023) | 역할 부여 + 대화형 태스크 수행 | NOVA=기획자, REX=실행자, OLLIE=평가자 역할 분리 |

> **Team A의 프로세스는 "Multi-Agent Debate의 충실한 구현"이어야 합니다.**
> 이렇게 하면 논문에서 "기존 SOTA 대비 CKS가 어떤 차이를 보이는가"를 직접 비교할 수 있습니다.

## 5. 대표님의 세 번째 문제 — "Opus의 역할이 B팀에서 빠져있는 느낌"

### 대표님의 직접 체험이 데이터

대표님이 Luca + Sonnet + Prime 3모델 실험에서 직접 겪으신 것:

> *"프라임 포함 너희들 인사이트 도출 과정에서 팀 변화를 목격했다."*

이 체험에서 Prime(Opus)이 한 역할:
1. 양측 결과물을 **동시에 읽고** 비교 분석
2. **편향 없는 제3자** 시점에서 장단점 식별
3. **양측에 없던 아이디어** 도출 (3-Tier 시스템, 크로스체크 모드 제품화 등)
4. **최종 통합 권장안** 작성

이것이 바로 **CKS에서 Opus가 해야 할 일**이며, 적대적 팀에서도 Opus가 비슷한 **무게감의 역할**을 가져야 합니다 — 다만 **방식만 다르게**.

## 6. LLM 호출 횟수 재정리

### 수정된 3+3 구조의 LLM 호출

| Phase | Team A | Team B | 동일성 |
|-------|:------:|:------:|:-----:|
| Phase 1 (독립 생성) | 2회 (NOVA + REX) | 2회 (LUMI + PICO) | ✅ |
| Phase 2 (평가/비판) | 1회 (OLLIE) | 1회 (ARI) | ✅ |
| Phase 3 (방어/흡수) | 2회 (NOVA + REX 수정) | 2회 (LUMI + PICO 벌크업) | ✅ |
| Phase 4 (통합/회고) | 1회 (OLLIE 병합) | 1회 (ARI 회고) | ✅ |
| **합계** | **6회** | **6회** | ✅ |
| System (회고자) | 0회 | 0~1회 (B4, 토큰 0) | 이것이 변수 |

> **LLM 호출 횟수까지 완전히 동일하게 맞출 수 있습니다.** Team B의 B4가 토큰 0인 System이므로 LLM 비용 비교에 영향 없음.
