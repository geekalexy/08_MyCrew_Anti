## 1. 개요 (Overview)
본 기획서는 사용자의 파편화된 요구사항을 입력받아 **"이번에 만들 것(MVP)"**과 **"나중에 만들 것(Roadmap)"**을 스스로 분리하고, 버전별 기획서(PRD)를 작성 및 보관한 뒤 사용자와 최종 확정하는 **'AI 기획자(Product Manager)'** 역할의 MCP 아키텍처를 정의합니다.

이 설계는 우리가 분석한 **Shrimp(협상/개입), Claude Task Master(PRD 중심주의), Sequential Thinking(다각도 추론)**의 장점들을 완벽하게 융합한 결과물입니다.

## 2. 코어 워크플로우 (Core Workflow)

### Step 1: 요구사항 분석 및 추론 (by Sequential Thinking)
사용자가 "이런 앱 만들어줘"라고 요구사항을 텍스트로 던집니다.
*   **Sequential Thinking 발동**: MCP는 즉시 코딩에 들어가지 않고, 스스로 생각의 가지(Branch)를 뻗습니다.
    *   *Thought 1*: "사용자가 요구한 결제, 소셜 로그인, 게시판, 채팅 중 핵심 가치는 무엇인가?"
    *   *Thought 2*: "채팅은 개발 코스트가 높다. MVP(v1.0)에서는 게시판 댓글로 대체하는 것이 좋다."
    *   *Thought 3 (Branch A)*: "만약 채팅이 필수라면 결제를 v1.1로 미루자."

### Step 2: 버전별 PRD 초안 생성 및 보관 (by Claude Task Master)
추론이 끝나면, 요구사항을 잘라 **버전별 PRD 파일**을 물리적으로 분리 생성하여 시스템에 보관합니다.
*   📁 `.mycrew/docs/roadmaps/`
    *   📄 `v1.0_MVP_PRD.txt`: 즉시 개발에 들어갈 최소 기능 명세.
    *   📄 `v1.1_FastFollow_PRD.txt`: 출시 직후 붙일 2차 기능 (예: 소셜 로그인).
    *   📄 `v2.0_ScaleUp_PRD.txt`: 장기 비전 기능 (예: 실시간 채팅, AI 추천).
*   **효과**: 단일 진실 공급원(PRD.txt)을 강제하여 개발 단계에서 에이전트가 오버엔지니어링(Over-engineering)하는 것을 원천 차단합니다.

### Step 3: 사용자와의 스코프 협상 및 확정 (by Shrimp UX)
버전별 분리가 끝나면, MCP는 사용자에게 협상(Intervention)을 시도합니다.
*   **AI**: *"대표님, 제안하신 기능 중 '채팅'은 v1.0(MVP)에 넣기엔 일정 리스크가 큽니다. v1.0은 '게시판'까지만 구현하고, 채팅은 작성해둔 `v2.0_PRD.txt`로 이관하여 보관해 두었습니다. 이렇게 v1.0 개발을 확정(Confirm)하고 시작할까요? 아니면 채팅을 v1.0으로 다시 가져올까요?"*
*   **사용자 개입**: 사용자는 동의하거나(`Yes`), 일부 기능을 버전에 맞게 수정 지시합니다.
*   **기획 최종 확정 및 종료**: 사용자가 스코프를 확정(Confirm)하면 `v1.0_MVP_PRD.txt`를 시스템에 영구 락온(Lock-on)하고 기획 파이프라인을 공식적으로 종료합니다. (실제 개발 파이프라인으로의 진입 여부는 이후 사용자가 개별적으로 판단하여 가동합니다.)

## 3. MCP 도구 스펙 (Tool Specifications)

이 기획자 MCP 서버가 제공해야 할 핵심 도구(Tools)는 다음과 같습니다.

### 1) `analyze_scope`
*   **기능**: 사용자의 입력 텍스트와 **'개발 완료 희망 기간'**을 입력받아 핵심(Must-have)과 부가(Nice-to-have) 기능으로 분류.
*   **내부 로직**: 기능별 구현 난이도와 예상 시간을 스스로 평가하여 스코프를 나눔.
    *   **스코프 방어 로직**: MyCrew 에이전트는 기획부터 개발 구현 계획, 태스크 리스트 작성까지 정확하고 치밀하게 마스터하므로, 구현 난이도가 극단적으로 높은 기능은 드뭅니다. 따라서 **사용자의 희망 기간이 짧다는 이유만으로 MVP 기능을 터무니없이 과도하게 제거(Drop)해서는 안 됩니다.**

### 2) `make_roadmaps`
*   **기능**: 분류된 기능을 바탕으로 MVP 및 미래 로드맵 파일 생성.
*   **연동**: 대시보드 백로그 칸반 카드 생성 및 Graphify(지식망)에 미래 스코프 노드로 자동 등록.

### 3) `confirm_mvp`
*   **기능**: 사용자에게 v1.0 범위를 브리핑하고 확정(Confirm)을 대기하는 도구.
    *   **설득 및 브리핑 전략**: 기간 제약으로 인해 소규모 기능의 MVP를 제안하는 경우, 단순히 기능이 빠졌다고 통보하는 것이 아니라 **좋은 점을 반드시 설명**해야 합니다. *(예: "소규모 기능으로 MVP를 시작하면, 빠른 런칭 후 다음 스프린트로 즉시 이어갈 수 있어 결과적으로 최종 개발 완료 시점을 앞당길 수 있습니다." 등 설득력 있는 보고 진행)*
*   **반복 피드백 루프 (Iterative Review)**: 브리핑 도중 사용자가 요구사항을 추가하거나 제거할 경우, 즉시 `analyze_scope` 및 `make_roadmaps` 도구를 재호출하여 로드맵을 다시 분석하고 수정된 결과를 재보고합니다.
*   **상태 제어**: 사용자의 최종 확정(Confirm)이 떨어지면 기획 세션을 공식적으로 종료하고 안전하게 대기 상태로 전환합니다. (최종 보고 및 확정이 이루어지기 전까지 이 기획-피드백 루프는 무중단으로 유지됩니다.)

### 4) 기획 재작업 시 불변성 유지 (Immutable Plan Forking - Phase 44 보강)
*   **배경**: Auto Run 재작업 시 원본 카드를 유지하고 새 카드를 Fork하는 "불변성(Immutability) 정책"이 Plan Master에도 동일하게 적용됩니다.
*   **기능**: 확정(Confirm)되어 영구 락온(Lock-on)된 기존 PRD(`.txt`) 파일이나 연관 태스크 카드가 존재하는 상태에서 기획 수정/재작업 지시가 떨어질 경우, **기존 파일과 태스크를 덮어쓰거나 오염시키지 않습니다.**
    *   이 때, 단순히 새 지시사항만으로 기획을 갈아엎는 것이 아니라, **CEO의 새로운 요구사항을 기존 요구사항(Original Context)에 더해서(Append) 주입**하여 연속성을 보장합니다.
*   **내부 로직**:
    - 기존 `v1.0_MVP_PRD.txt`는 아카이브 보존(Read-only).
    - 기존 요구사항과 CEO의 추가/수정 요구사항을 종합 분석하여 `v1.1_MVP_PRD.txt` (또는 revision 버전)를 새롭게 생성(Fork).
    - 기존 백로그 태스크들을 삭제하는 대신, 새로운 스코프에 맞는 완전히 새로운 백로그 태스크 그룹을 생성. 이를 통해 과거 기획 이력과 결과물을 투명하게 남기면서 안전한 재작업이 이루어집니다.

## 4. 다중 모델 라우팅 전략 (Multi-Model Strategy)

비싼 하이엔드 모델의 레이트 리밋(Rate Limit) 고갈을 방지하기 위해, 단계별로 최적화된 모델을 라우팅(Quota Defender 연동)합니다.
*   **고품질 초기 스코프 및 로드맵 초안 도출**: **`anti-claude-sonnet-4.6-thinking`**을 전면에 세워 사용자의 의도를 정밀하게 파악하고, 단순 정리를 넘어 완성도 높은 1차 스코프 분류 및 로드맵 초안을 먼저 뽑아냅니다.
*   **심층 평가 및 기획 고도화 (Plan_Master)**: 도출된 1차 로드맵 초안 자료를 바탕으로, 비즈니스 리스크 분석, 기능 간 모순 검토, 최종 PRD 분할 등 고난도 심층 평가 및 추론에 **`anti-claude-opus-4.6-thinking`** 또는 **`anti-gemini-3.1-pro-high`**를 투입하여 기획의 밀도를 극한으로 끌어올립니다.
*   **기획 종료 및 컨펌 (Task Terminated)**: 기획(PRD)이 최종 픽스되고 사용자 컨펌을 받으면, Plan_Master MCP는 역할을 완료하고 즉시 퇴장합니다. (이후 코딩 파이프라인의 가동 여부는 사용자의 판단에 맡깁니다.)

## 5. 핵심 메커니즘: JSON 스키마 기반 사고 구조화 강제 (Sequential Thinking)

Sequential Thinking MCP의 설계 철학을 차용하여, 모델이 기획을 심층 분석할 때 무형의 혼잣말(자연어)을 늘어놓는 대신 **엄격한 JSON 스키마를 따르는 도구(Tool) 호출을 강제**합니다.
*   **사고의 데이터 통제 (이과적 통제)**: 프롬프트로 "Step-by-step으로 생각하라"고 문과적으로 지시하는 것을 넘어, `thoughtNumber`, `nextThoughtNeeded` 등 명시적인 JSON 변수값을 반환하도록 시스템 레벨에서 강제함으로써 모델의 논리력과 메타 인지 능력을 비약적으로 상승시킵니다.
*   **추론 과정의 시각화 및 타임라인 추적 (UX)**: 모델이 내부적으로 고민하는 단계를 투명한 JSON 데이터로 받아, 해당 태스크 카드의 **'Activity 탭 및 타임라인'**에 단계별로 기록하여 렌더링합니다. 이를 통해 사용자는 결과물뿐만 아니라 AI가 어떤 논리적 단계를 거쳐 이 결론에 도달했는지 상세히 추적(Audit)할 수 있습니다.
*   **Graphify 지식망 연동 및 스플릿 뷰 프리뷰 (UX)**: 에이전트가 "왜 특정 기능을 MVP에서 제외했는지"에 대한 추론 과정(Thought) 자체를 텍스트가 아닌 JSON 메타데이터로 파싱하여 Graphify 노드에 영구 아카이브합니다. 특히 **플랜 모드(Plan Mode) 종료 후 생성된 기획/로드맵 카드**의 경우, 상세 모달(Task Detail Modal) 상단의 **'Preview(미리보기)' 버튼을 클릭했을 때 우측 스플릿 뷰(Split-view) 영역에 지식그래프(Graphify) 노드가 즉시 렌더링**되도록 UX를 구성합니다. 이를 통해 대표님은 좌측에서 기존 텍스트 컨텍스트를 유지한 채, 우측에서 전체 아키텍처 노드를 시각적으로 조망하며 피드백을 주실 수 있습니다.

## 6. 기대 효과 (Why this is powerful)
1.  **AI의 오버엔지니어링 방지**: 비개발자 사용자가 "이것도 저것도 다 넣어줘"라고 했을 때, AI가 전부 다 짜려다가 코드가 꼬이고 에러가 폭발하는 고질적 문제를 **'버전 분리'**를 통해 완벽히 해결합니다.
2.  **기획 자산화 및 백로그 연동**: "언젠간 해야지" 했던 2차 아이디어들은 단순 텍스트(`v2.0.txt`)로만 남는 것이 아닙니다. **대시보드의 칸반 보드 'Backlog' 컬럼에 실제 카드로 자동 생성**되며, 동시에 **Graphify 지식 그래프의 'Future Scope' 노드로 시각화되어 배치**되므로 영원히 유실되지 않습니다.
3.  **마이크루의 PM 화**: 마이크루가 단순한 '코딩 머신'을 넘어, 사용자의 사업적 리스크를 관리해주는 '프로덕트 매니저' 포지션으로 격상됩니다.
