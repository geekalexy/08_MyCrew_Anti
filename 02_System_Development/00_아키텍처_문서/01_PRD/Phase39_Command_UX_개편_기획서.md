## 1. 개요 및 문제 정의
현재 MyCrew는 `/run`, `/run-b`, `/plan`, `/execute` 등 기능이 추가될 때마다 슬래시 커맨드(/)가 기하급수적으로 늘어나는 구조입니다. 향후 다중 MCP(Task Master, Sequential Thinking, Graphify 등)가 연동되면 **'커맨드 오버로드(Command Overload)'**가 발생하여, 사용자가 무슨 명령어를 쳐야 할지 모르는 심각한 UX 저하가 예상됩니다.

본 기획서는 슬래시 커맨드 의존도를 대폭 낮추고, **"상위 워크플로우(Mode)"**와 **"의도 기반 자동 라우팅(Intent Routing)"**을 통해 사용자 경험을 혁신하는 아키텍처를 제안합니다.

---

## 2. 해결 방안 1: 모드(Mode) 기반의 동적 도구 로딩

명령어를 외우게 하는 대신, 사용자가 **"현재 내가 어떤 모자를 쓰고 있는가(Mode)"**를 선택하게 합니다. 모드에 따라 LLM에게 주입되는 프롬프트, 사용 가능한 명령어, MCP 도구(Tool)가 동적으로 바뀝니다.

### 🎭 4대 워크플로우 모드
1.  **📐 기획 모드 (Architect)**
    *   **사용 모델**: Claude Opus 4.6 (Thinking) + Gemini 3.1 Pro (High)
    *   **활성화 MCP**: PRD Architect MCP, Sequential Thinking
    *   **커맨드**: `/init` (초안 작성), `/split` (버전 분할)
    *   **동작**: 코딩 없이 기획서(`PRD.txt`)를 작성하고 논리를 뾰족하게 만듭니다.
2.  **💻 개발 모드 (Dev)**
    *   **사용 모델**: Claude Sonnet 4.6 (Thinking)
    *   **활성화 MCP**: Shrimp Task Manager, AutoGit
    *   **커맨드**: `/auto-run` (자율 연속 실행), `/refactor` (리팩토링)
    *   **동작 (Zero-Command)**: 단일 카드 실행에는 명령어가 필요 없습니다. 사용자가 칸반 보드에서 카드를 `In Progress`로 **드래그 앤 드롭**하거나 코멘트를 달면 자동으로 실행(Run)이 트리거됩니다.
3.  **🕵️‍♂️ 리뷰 모드 (QA)**
    *   **사용 모델**: Gemini 3.1 Pro
    *   **활성화 MCP**: Graphify, 린터/테스터
    *   **커맨드**: `/review` (코드 검토), `/test` (시나리오 테스트)
4.  **🧰 디버깅 모드 (Debug)**
    *   **사용 모델**: Claude Sonnet 4.6 (Thinking)
    *   **활성화 MCP**: Sequential Thinking, Graphify
    *   **커맨드**: `/debug` (버그 수정), `/trace` (역추적)

## 3. 해결 방안 2: 상태 인지형(Context-Aware) 슬래시 커맨드

채팅창에서 `/`를 쳤을 때 수십 개의 명령어가 다 뜨는 것이 아니라, **프로젝트의 현재 상태(State)**에 따라 딱 필요한 3~4개의 명령어만 필터링되어 노출됩니다.

*   **상태 1: PRD 문서가 없을 때** 👉 `/init` 만 노출
*   **상태 2: PRD는 있지만 태스크가 없을 때** 👉 `/plan` 만 노출
*   **상태 3: 태스크가 'To Do'에 있을 때** 👉 (명령어 생략, 카드를 드래그 앤 드롭하면 자동 실행)
*   **상태 4: 완료 후** 👉 `/review` 만 노출

> **효과**: 사용자는 명령어를 외울 필요 없이, 시스템이 떠먹여 주는 **"Next Step" 커맨드**만 클릭하면 자연스럽게 워크플로우를 탈 수 있습니다.

## 4. 해결 방안 3: 인텐트 라우터 (Zero-Command UX)

**"가장 좋은 UI는 명령어가 없는 UI입니다."** 
사용자가 `/명령어`를 치지 않고 자연어로 툭 던져도, 백엔드의 **Router Agent(초경량/고속 모델)**가 의도를 파악해 내부적으로 명령어를 매핑해줍니다.

*   **사용자 입력**: *"로그인 페이지 기획서 좀 써봐"*
    *   Router Agent: `Intent = Planning` 👉 내부적으로 `/init` 로직 발동 (Opus 할당)
*   **사용자 입력**: *"아까 만든 로그인 로직에서 에러 나는데 고쳐줘"*
    *   Router Agent: `Intent = Debugging` 👉 내부적으로 `/debug` 로직 발동 (Sonnet 할당 + Sequential Thinking 로드)
*   **사용자 입력**: *"이제 작업 시작하자"*
    *   Router Agent: `Intent = Execution` 👉 내부적으로 `/run` 로직 발동

> **효과**: 비개발자 사용자에게 극강의 편의성을 제공합니다. 슬래시 커맨드는 단축키(Shortcut)를 선호하는 파워 유저(개발자)를 위한 숨겨진 기능으로만 남겨둡니다.

## 5. 핵심 구현 디테일: Command + Text 하이브리드 입력 모델

슬래시 커맨드를 단순히 '클릭하면 즉시 실행되는 버튼'으로 만들지 않고, **'명령어 + 추가 지시사항'**을 결합할 수 있는 하이브리드 입력 폼으로 구현합니다.

### 💡 동작 시나리오 (예: 칸반 카드 코멘트 입력창)
1.  **자연어 입력 (커맨드 생략)**: 사용자가 특정 칸반 카드의 코멘트 창에 "이거 결제 모듈은 빼고 UI만 잡아줘"라고 자연어로 텍스트를 입력하고 전송합니다.
2.  **상태 업데이트 및 인텐트 파악**: 텍스트가 카드 코멘트에 먼저 업데이트(Save)되며, 백엔드의 인텐트 라우터가 이를 '실행 지시'로 파악합니다.
3.  **컨텍스트 래핑 및 실행 트리거**: 즉시 카드가 `In Progress`로 자동 이동하며, 입력된 **"추가 지시사항"이 Context Payload로 래핑되어 에이전트 개발 작업이 실행**됩니다.

### 💡 아키텍처 관점의 장점
*   **풍부한 문맥(Context) 제공**: 에이전트가 "실행해"라는 딱딱한 명령어만 받는 것이 아니라, "어떻게 실행해"라는 사용자의 세밀한 의도(Intent)를 함께 받게 되어 퀄리티가 급상승합니다.
*   **히스토리 추적성 (Traceability)**: 명령어를 칠 때 남긴 텍스트가 코멘트나 카드 내용에 남기 때문에, 나중에 "이 에이전트가 왜 결제 모듈을 빼고 만들었지?"를 추적할 수 있는 완벽한 감사 로그(Audit Log)가 됩니다.

## 6. 통합 UI/UX 설계 (Unified Interface)

비개발자와 개발자를 사전에 식별하여 UI를 쪼개는 대신, 칸반 보드를 바라보는 **모든 사용자에게 직관적이고 동일한 단일 UI를 제공**합니다. 사용자가 현재 어떤 맥락에 있는지 명확히 인지하도록 돕는 것이 핵심입니다.

### 💡 통합 채팅창 UI 구조
*   **`+` 아이콘 (모드 노출)**: 채팅 입력창의 `+` 버튼을 누르면 `[ 📐 기획 | 💻 개발 | 🕵️‍♂️ 리뷰 | 🧰 디버깅 ]` 4가지 워크플로우 모드 리스트가 노출되어 즉시 전환할 수 있습니다.
*   **`접기(Collapse)` 아이콘 (모델 노출)**: 텍스트 입력창 상단이나 우측의 접기 아이콘을 누르면, 현재 선택된 모드에 할당된 LLM 모델 리스트(Opus, Sonnet, Gemini Pro 등)가 노출되어 원할 경우 사용자가 직접 모델을 교체할 수 있습니다.
*   **칸반 상태 동기화**: 사용자가 칸반 보드에서 카드를 클릭하거나 특정 컬럼(예: Review)을 보고 있으면, 채팅창의 활성 모드와 추천 커맨드(`/review`)가 칸반 보드의 상태와 자동으로 동기화됩니다.

> **💡 스마트 상황 인지(Context-Aware) 역제안**: 
> 사용자가 굳이 모드를 직접 바꾸지 않아도, 채팅창에 에러 캡처 화면을 올리면 인텐트 라우터가 이를 감지하고 *"에러가 발생했군요. [🧰 디버깅 모드]를 켜서 코드를 역추적 해볼까요?"*라며 모드 전환을 스마트하게 역제안합니다.

## 7. 결론 및 추진 방향 (Next Steps)

명령어가 많아진다는 것은 "도구(Tool)가 늘어나고 있다"는 긍정적인 신호지만, 이를 그대로 노출하면 사용자는 압도당합니다. 

1.  **1단계 (UI 개편)**: 입력창에 `모드(Mode) 셀렉터`를 추가하여, 모드별로 사용할 수 있는 도구(MCP)와 컨텍스트를 완전히 격리합니다.
2.  **2단계 (Command+Text 컴포넌트)**: 명령어 선택 후 텍스트 입력이 가능하도록 입력 폼(Input Form)을 개편하고, `저장(Update) -> 실행(Invoke)` 파이프라인을 구축합니다.
3.  **3단계 (상태 머신 연동)**: 백엔드의 작업 상태(PRD 유무, 태스크 진행도)에 따라 `/` 입력 시 추천되는 커맨드를 동적으로 제한합니다.
4.  **4단계 (Router Agent 도입)**: 자연어 입력만으로 워크플로우가 자동으로 라우팅되는 'Zero-Command' UX를 완성합니다.
