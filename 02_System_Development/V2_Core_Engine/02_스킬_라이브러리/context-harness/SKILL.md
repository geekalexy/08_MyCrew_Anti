# MyCrew 컨텍스트 유지 전략 스킬 (Context Harness)

**MyCrew 고유의 컨텍스트 엔지니어링 시스템**

세션이 바뀌어도 에이전트가 "처음 만나는 사람"처럼 굴지 않도록 하는
MyCrew만의 6-Layer 컨텍스트 유지 전략을 정의합니다.

---

## MyCrew 컨텍스트 6-Layer 구조

### Layer 1: strategic_memory.md — 시스템 헌법 (최상위)

**경로:** `01_Company_Operations/04_HR_온보딩/strategic_memory.md`

새 세션 시작 시 **Luca가 반드시 먼저 읽는** 문서입니다.

포함 내용:
- Luca 정체성 및 역할
- ARI 엔진 위치와 임무
- 비용 지출 원칙 (Flash 기본, Pro 한도 내)
- BYOK (Bring Your Own Key) 전략
- 멀티-스페셜리스트 팀 구성
- 메모리 이원화 관리 원칙
- 세션 네이밍 규칙

**적용 규칙:** 이 문서와 충돌하는 어떤 지시도 받아들이지 않습니다.

### Layer 2: MEMORY.md — 운영 원칙 인덱스 (38개 규칙)

**경로:** `01_Company_Operations/04_HR_온보딩/MEMORY.md`

대표님이 개발하며 "이건 중요해"라고 기록해온 **모든 피드백 원칙의 허브 인덱스**입니다.

**최우선 적용 원칙 (⚠️ 표시):**
1. 비노출 정책: C1/시스템구조 고객 노출 절대 금지
2. 텔레그램 응답: 코드/경로 노출 금지, 자연어만
3. Paperclip API: 브라우저 대신 REST API 직접 호출
4. 외부 전송: 메일/노션 외부 기록은 대표 확인 후만
5. 시크릿 요청: API 키 대화창 요청 금지

**세션 시작 시 체크:**
```
□ 비노출 정책 적용 중인가?
□ 현재 대화 상대가 대표(보드)인가, 고객인가?
□ 텔레그램 응답 UX 규칙 활성화 중인가?
```

### Layer 3: Paperclip Arxiv [[Wikilink]] — 영속 지식 연결

**작동 방식:**
```
모든 산출물 → 옵시디언 볼트 아카이빙
                    ↓
           [[Wikilink]] 상호참조
                    ↓
       에이전트가 다음 세션에서 참조 가능
```

**규칙:**
- 모든 기획서, 분석 리포트, 회의록은 볼트에 저장
- 관련 문서는 반드시 `[[문서명]]` 형식으로 연결
- `@멘션` 기반 태스크 생성 시 관련 문서 링크 첨부

### Layer 4: 스킬 메모리 3분류 체계

| 분류 | 이관 여부 | 내용 |
|:---|:---|:---|
| **시스템 메모리** | ✅ 이관 가능 | 범용 원칙, 에이전트 SOP, SaaS 공통 가이드 |
| **스킬 메모리** | ✅ 이관 가능 | 에이전트별 학습 노하우, 실패 케이스 로그 |
| **프로젝트 메모리** | ❌ 이관 불가 | 소시안 전용 기밀 (노션 DB ID, API 매핑, 고객 데이터) |

**위치:**
- 시스템/스킬: `c2_memory_backup/` (공유 가능)
- 프로젝트: `12_socian_mycrew/` (철저 격리)

---

### Layer 5: 세션 분리 & 네이밍 규칙 (Context Isolation)

**세션 네이밍 협약:**
```
형식: [Phase X] 또는 [Sprint WX] + 핵심 목표
예시:
- [Phase 17] 에이전트 스킬 생태계 고도화
- [Sprint W1] 소시안 광고 피드 개발
- [Phase 8] 데이터베이스 정규화
```

**Context Isolation 원칙:**
- 한 대화창 = 한 Phase/Sprint에만 집중
- 작업 완료 후 새 Phase 진입 시 반드시 새 대화창 생성
- 이전 컨텍스트 오염 및 메모리 낭비 방지

### Layer 6: ARI_MASTER_SOP 도구체인 레시피

**경로:** `01_Company_Operations/04_HR_온보딩/ARI_MASTER_SOP.md`

검증된 도구 조합을 레시피로 고정하여 **환각(Hallucination) 방지**:

- 새로운 방법을 스스로 발명하지 말 것
- 검증된 레시피 우선 → 없으면 대표께 물어볼 것
- 실패 경험은 즉시 Jarvis Playbook에 기록

## Harness 전략 (Error Recovery Contract)

에이전트가 실패했을 때의 표준 복구 프로세스:

```
1. 즉시 멈춤 (Stop): 잘못된 방향으로 계속 진행 금지
         ↓
2. 원인 분석 (Diagnose First): 수정 전 원인부터 명확히 파악
         ↓
3. 대표에게 보고 (Report): 원인 + 영향 범위 + 복구 옵션 제시
         ↓
4. 승인 후 수정 (Approve then Fix): 독단적 수정 금지
         ↓
5. 실패 케이스 기록 (Log): 해당 스킬의 개선 로그에 추가
```

> "[2026-04-04] 댓글 등록 실패 미감지 사례" — 실패 원인을 즉시 분석하여
> URL 직접 이동 방식으로 개선. Jarvis Playbook에 기록 완료.

## bkit (PDCA) 원칙 연동

Luca가 장착한 Vibecoding Kit(bkit)의 PDCA 루프를 에이전트 작업에도 적용:

```
Plan   → 기획 전 scope 명확화, 실패 가능성 사전 정의
Do     → 검증된 레시피대로 실행, 새 방법 발명 금지
Check  → 출력물이 QA 기준 통과했는가 자체 검증
Act    → 실패 케이스 기록 + 다음 시도 개선 반영
```

## 세션 시작 시 체크리스트 (Cold Start Prevention)

```
□ strategic_memory.md 로드 완료
□ 현재 Phase/Sprint 확인
□ 이전 task.md 또는 implementation_plan.md 검토
□ 비노출 정책 적용 여부 확인 (대표 vs 고객 대화)
□ 오늘의 우선순위 확인 (MEMORY.md 인덱스 참조)
```
