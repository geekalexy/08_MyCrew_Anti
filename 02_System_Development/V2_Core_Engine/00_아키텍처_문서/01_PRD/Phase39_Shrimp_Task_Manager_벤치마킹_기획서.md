## 1. 개요 (Overview)
본 문서는 LLM 전용 작업 관리 MCP 서버인 **[mcp-shrimp-task-manager]**의 아키텍처와 UX 철학을 심층 벤치마킹합니다. 
단순한 기능 나열이나 타 시스템과의 우월성 비교를 배제하고, Shrimp가 **프롬프트 문서를 어떻게 구조화**했는지, **인간과 AI의 인터랙션(UX)을 어떻게 매끄럽게 연결**했는지, 그리고 그 고유한 장점을 우리 시스템에 어떻게 순수하게 학습하고 적용할 수 있을지에 집중합니다.

---

## 2. 문서의 구조화 및 역할 분석 (Modular Prompt Architecture)

Shrimp Task Manager의 가장 큰 특징은 거대한 프롬프트 덩어리에 의존하지 않고, 에이전트의 뇌(Prompt)를 목적별 마크다운 문서로 잘게 쪼개어 모듈화했다는 점입니다.

| 핵심 문서 | 역할 및 내용 | 배울 점 (Insight) |
| :--- | :--- | :--- |
| **`system.md`** | 에이전트의 페르소나와 작업 수행의 대원칙(절대 어기면 안 되는 규칙)을 정의하는 최상위 시스템 프롬프트. | 에이전트의 정체성을 툴(Tool) 로직과 분리하여, 시스템의 안정성과 확장성을 높임. |
| **`tools.md`** | 에이전트가 사용할 수 있는 도구(`plan task`, `execute task` 등)의 명세서. 어떤 상황에서 어떤 도구를 써야 하는지 판단 기준(Trigger)을 설명. | 툴의 파라미터만 던져주는 것이 아니라, "이 툴을 왜, 언제 써야 하는지"를 명확히 글로 서술하여 AI의 문맥 이해도를 극대화함. |
| **`SUB_AGENTS_SPECIFICATION.md`** | 서브 에이전트(리서처, 코더 등)의 명세서. 특정 태스크를 위해 전문화된 에이전트들의 역할 분담표. | 복잡한 태스크를 단일 모델이 아닌 다중 에이전트로 위임(Delegate)하는 멀티 에이전트 표준 규격. |
| **`shrimp-rules.md`** | 프로젝트 생성 시 사용자 프로젝트 디렉토리에 복사되는 코딩 컨벤션 및 템플릿 문서. | 시스템 단의 룰(`system.md`)과 사용자(프로젝트) 단의 룰(`shrimp-rules.md`)을 완벽히 격리하여 사용자 맞춤형 확장을 지원함. |

## 3. UX와 기능의 연결 (User Experience Flow)

Shrimp는 인간(사용자)의 개입이 에이전트의 워크플로우를 방해하지 않고 오히려 돕도록 정교한 UX를 설계했습니다.

### 💡 3.1. 명시적 명령어 기반의 인터랙션 (Command-Driven)
*   **UX 흐름**: 자연어로 모호하게 대화하는 대신, `/init project rules` → `/plan task` → `/execute task` → `/list tasks` 로 이어지는 **명확한 명령어 기반 흐름**을 제공합니다.
*   **배울 점**: 사용자는 자신이 에이전트에게 어떤 상태를 요구하고 있는지 직관적으로 인지할 수 있으며, 에이전트 역시 텍스트 파싱 오류 없이 정확한 Tool을 즉각적으로 트리거할 수 있습니다.

### 💡 3.2. 인간 개입(Intervention)과 궤도 복구의 유연성
*   **UX 흐름**: 에이전트가 작업을 수행한 뒤 사용자가 직접 테스트(`npm run dev`)를 돌려 에러 로그를 던집니다. 에이전트는 원래 하던 작업의 컨텍스트를 유지한 채 서브 태스크(에러 수정)로 진입하고, 수정을 마치면 다시 원래의 작업으로 부드럽게 복귀합니다.
*   **배울 점**: 사용자의 피드백이 '방해'가 아니라 자연스러운 '협업 프로세스'로 녹아듭니다. 시스템이 현재의 목표(Current Task)를 잃지 않도록 백그라운드 프롬프트를 계속 붙잡아주는 훌륭한 상태 관리(Stateful Memory)입니다.

### 💡 3.3. 수동/자율 모드의 심리적 안정감 제공
*   **UX 흐름**: 처음에는 `/execute` 명령어로 한 단계씩(Step-by-step) 코딩 결과를 눈으로 확인하고, 신뢰가 쌓이면 `/continuous mode`로 전환하여 남은 작업을 에이전트에게 온전히 맡깁니다.
*   **배울 점**: 사용자에게 처음부터 끝까지 통제권을 주어 불안감을 없애고, 필요할 때만 자율성을 극대화하는 매우 사용자 친화적인(User-Centric) 전환 설계입니다.

## 4. 핵심 특징 및 배울 점 (Insights to Apply)

Shrimp Task Manager의 철학에서 우리가 배워야 할 핵심 가치는 다음과 같습니다.

1.  **"생각의 구조화 (Chain-of-Thought Enforcement)"**
    *   `/plan task` 시 단순히 리스트만 짜는 것이 아니라 내부적으로 기술 요구사항 분석(`analyze`)과 검토(`reflect`)를 강제로 거치게 만듭니다. 이 검증 흐름을 우리 파이프라인(Opus 모델 기획 → Gemini 모델 리뷰)에 도입하여 기획의 완성도를 높일 수 있습니다.
2.  **"작업의 원자성 (Atomic Task Decomposition)"**
    *   거대한 목표를 의존성(Dependencies)이 얽힌 가장 작은 단위의 태스크로 잘게 쪼갭니다. 이 사상을 MyCrew의 칸반 DB 설계에 차용하여, `Task` 테이블에 `dependencies` 배열을 도입하고 스케줄링 로직을 고도화할 수 있습니다.
3.  **"모듈형 프롬프트 (Modular Context Injection)"**
    *   모든 프롬프트를 한 번에 주입하는 대신, 사용자의 명령어와 현재 진행 상태에 따라 `system.md`, `tools.md`, `shrimp-rules.md`를 상황에 맞게 엮어서 주입하는 동적 컨텍스트 관리를 배워야 합니다.

---

## 5. 결론 및 적용 방향
Shrimp의 철저하게 모듈화된 문서 구조와 인간 개입을 부드럽게 포용하는 UX는 MyCrew가 지향하는 **"에이전트 친화적(Agent-Friendly) AI OS"**의 훌륭한 교과서입니다. 

다음 스텝으로 Shrimp의 슬래시 명령어 흐름(`/plan`, `/execute`, `/continuous`)을 MyCrew 익스텐션 UI에 차용하고, 각 상태에 따라 필요한 마크다운 문서 프롬프트(룰셋)를 동적으로 주입하는 **Stateful Context Manager**를 설계하겠습니다.
