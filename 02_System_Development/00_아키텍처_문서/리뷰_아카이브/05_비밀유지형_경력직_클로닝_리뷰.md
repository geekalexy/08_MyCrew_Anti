## 🔴 P0-1: "비식별 회고" LLM 위임의 치명적 약점

### 기획서 설계

```
작업 성공 → Flash 모델 추가 호출 → "고유명사 제거하고 범용 패턴만 요약해" → SKILL.md 저장
```

### Prime 판정: 🔴 LLM은 이 작업에 부적합합니다

**근본적 문제: LLM은 "무엇이 기밀인지"를 판단할 수 없습니다.**

LLM에게 "고유명사를 제거하라"고 하면, 어떤 일이 벌어지는지 보겠습니다:

```
[원본 — 대표님의 실제 프롬프트]
"Socian의 인스타그램 릴스 마케팅을 기획해줘. 
 경쟁사 Hootsuite Korea가 최근 릴스 30초 포맷으로 
 팔로워 3만 → 8만 달성한 전략을 역설계하고, 
 우리 CTO 알렉스가 직접 출연하는 릴스를 기획해줘."

[LLM 비식별 회고 결과 — "범용화"]
"릴스 마케팅 기획 시, 
 경쟁사의 30초 포맷 전략을 역설계하고 
 핵심 인물이 직접 출연하는 릴스가 효과적이다.
 팔로워 3만→8만 성장 사례를 참고할 것."
```

> [!CAUTION]
> **문제 발견:**
> 1. "팔로워 3만→8만"이 그대로 남아있음 — 이 숫자 조합이 특정 회사를 역추적할 수 있는 핑거프린트
> 2. "30초 포맷 전략"+ "팔로워 성장" 조합만으로도 Hootsuite Korea를 특정할 수 있음
> 3. LLM은 "3만→8만"이 기밀인지 아닌지 맥락적으로 판단할 능력이 없음 — 단순히 고유명사가 아니기 때문

**이것은 Phase 17 1차 리뷰(Sanitization Layer)에서 Prime이 이미 경고한 문제의 정확한 재현입니다.** 그때 결론: "LLM 자율 판단에 맡기지 말고 **Entity Scrubbing + Numeric Generalization을 강제 적용**하라."

### Prime 권고: 2단계 필터링 (LLM 의존 최소화)

```
[Step 1: Deterministic Scrubbing — 기계적, 예외 없음]
──────────────────────────────────────────────────────
입력: actualContent (원문)

// 1-A: Named Entity 강제 마스킹
const COMPANY_ENTITIES = ['socian', 'Socian', '소시안'];
const PERSON_ENTITIES  = ['알렉스', 'Alex', '대표님'];
const COMPETITOR_ENTITIES = ['hootsuite', 'Hootsuite'];

let sanitized = actualContent;
[...COMPANY_ENTITIES, ...PERSON_ENTITIES, ...COMPETITOR_ENTITIES]
  .forEach(e => sanitized = sanitized.replaceAll(e, '[ENTITY]'));

// 1-B: URL 강제 제거
sanitized = sanitized.replace(/https?:\/\/\S+/g, '[URL]');

// 1-C: 숫자 범위화 (Numeric Generalization)
sanitized = sanitized.replace(/\d{2,}/g, (match) => {
  const n = parseInt(match);
  if (n < 100) return '[이하100]';
  if (n < 1000) return '[100~1000]';
  if (n < 10000) return '[수천]';
  return '[수만이상]';
});

출력: "[ENTITY]의 인스타그램 릴스 마케팅을 기획해줘. 
       경쟁사 [ENTITY]가 최근 릴스 [이하100]초 포맷으로 
       팔로워 [수만이상] → [수만이상] 달성한 전략을 역설계..."
```

```
[Step 2: LLM 일반화 — Scrubbed 텍스트에서만 패턴 추출]
──────────────────────────────────────────────────────
입력: Step 1의 sanitized 텍스트 (이미 기밀 제거됨)

프롬프트: "다음 익명화된 업무 기록에서, 
범용적인 마케팅 프레임워크/프로세스 패턴만 3줄로 추출하시오."

출력: "릴스 마케팅 기획 시:
  1. 경쟁사 포맷을 역설계하여 콘텐츠 길이(초)를 벤치마킹
  2. 핵심 인물의 직접 출연이 성장률에 긍정적 영향
  3. 팔로워 성장 목표는 배수(x2~x3) 단위로 설정"
```

**핵심: Step 1에서 기밀이 이미 물리적으로 파괴된 후에만 LLM이 개입합니다.**
Step 2에서 LLM이 아무리 환각을 일으켜도, 입력에 기밀이 없으므로 출력에도 기밀이 나올 수 없습니다.

## 🔴 P0-2: Cross-Pollination의 "실패 후 성공" 트리거가 현재 아키텍처에서 구현 불가능

### 기획서 설계

> "에이전트가 1차 시도에서 실패를 겪었으나, 스스로 문제를 파악하고 재시도하여 성공으로 극복했을 때 작동"

### Prime 판정: 🔴 현재 executor.js는 이 시나리오를 추적할 수 없습니다

현재 `executor.run()`의 구조:

```javascript
// executor.js의 실행 구조
async run(taskContent, preEvaluated, agentId) {
  // ... 실행 ...
  const result = await geminiAdapter.generateResponse(...);
  return { text, model, category, score };  // ← 단발성 호출, 반환 후 종료
}
```

**문제:** `executor.run()`은 **단발성(One-shot) 호출**입니다.
- "1차 시도 → 실패 → 문제 파악 → 재시도 → 성공"이라는 **다회차 루프가 존재하지 않습니다**
- 현재 구조에서 "실패"는 `catch` 블록의 에러 → 사용자에게 에러 메시지 반환 → **끝**
- 재시도는 **사용자가 수동으로 다시 메시지를 보내는 것**이지, 에이전트가 자율적으로 하는 것이 아닙니다

따라서 "실패를 극복한 지혜를 공유"하려면, **실패와 성공을 연결하는 구조**가 먼저 필요합니다:

### Prime 권고: Task 테이블에 Retry Chain 추가

```sql
-- database.js에 추가할 마이그레이션
ALTER TABLE Task ADD COLUMN retry_count INTEGER DEFAULT 0;
ALTER TABLE Task ADD COLUMN parent_task_id INTEGER DEFAULT NULL;
-- parent_task_id: 이전 실패 태스크를 가리킴 → 재시도 체인 구성
```

```javascript
// executor.js — 향후 구현할 Retry-Aware Self-Learning
async run(taskContent, preEvaluated, agentId, parentTaskId = null) {
  // ... 기존 로직 ...
  
  // 성공 시 — 부모(실패) 태스크가 있으면 = "극복 성공"
  if (result.text && parentTaskId) {
    const parentTask = await dbManager.getTaskById(parentTaskId);
    if (parentTask && parentTask.status === 'error') {
      // 🎯 Cross-Pollination 트리거 조건 충족!
      await appendToSharedWisdom({
        failureContext: parentTask.content.slice(0, 100),
        recoveryPattern: sanitize(result.text.slice(0, 200)),
        category: evaluation.category,
      });
    }
  }
}
```

**이 구조 없이는 Cross-Pollination이 작동할 수 없으므로, Section 4 전체가 현재 시점에서 구현 불가능합니다.**

## 🟡 P0-3: "비식별 회고"의 숨겨진 API 비용 폭탄

### 기획서 설계

> "백그라운드로 소형 모델(gemini-3-flash 등)을 하나 더 호출"

### Prime 판정: ⚠️ 비용이 2배로 뛸 수 있습니다

현재 아리 엔진의 API 호출 파이프라인:

```
[메시지 1건 처리 — 현재]
1. modelSelector.selectModel(content)  →  Flash 호출 1회 (분석용)
2. geminiAdapter.generateResponse()    →  Flash/Pro 호출 1회 (실행용)
─────────────────────────────────────
합계: LLM 호출 2회/메시지

[메시지 1건 처리 — Generalization Reflection 추가 시]
1. modelSelector.selectModel()         →  Flash 호출 1회
2. geminiAdapter.generateResponse()    →  Flash/Pro 호출 1회
3. generalizationReflection()          →  Flash 호출 1회 (NEW!)
─────────────────────────────────────
합계: LLM 호출 3회/메시지 (50% 증가)
```

그런데 Step 3은 **score >= 0.8인 성공 작업에서만** 발동하므로, 전체 메시지의 약 60~70%에서 트리거될 것입니다. 하루 50건의 메시지가 오간다면:

```
현재: 50 × 2 = 100회 호출/일
추가 후: 50 × 2 + 35 × 1 = 135회 호출/일 (35% 증가)

월간 추가 비용 (Flash 기준, 입력 1K토큰 + 출력 500토큰):
≈ 35 × 30 × $0.001 = 약 $1.05/월  ← 금액 자체는 미미
```

**금액은 미미하지만, 원칙의 문제입니다.** P0-1에서 제시한 "2단계 필터링"을 적용하면:
- Step 1(Regex Scrubbing)은 **LLM 호출 0회** — 순수 문자열 처리
- Step 2(LLM 일반화)만 호출 → **비용 동일하되 보안은 100배 강화**

> [!TIP]
> **당장의 MVP에서는 Step 1(Regex Scrubbing)만으로도 충분합니다.**
> 
> 현재 Self-Learning 코드의 출력이 "3줄 요약"이면, LLM 없이 **고정 템플릿**으로도 가능합니다:
> ```
> `- [${category}] ${evaluation.reason} (Score: ${score})`
> ```
> 원문(`actualContent`)을 아예 저장하지 않는 것이 가장 확실한 방어입니다.

## 🟢 기획서에서 정확히 맞힌 부분들

### ✅ 3-Layer 메모리 분류 (Context / History / Wisdom)

```
[개인/회사 기억] MYCREW.md / IDENTITY.md  → 클론 시 초기화
[업무 내역]     database.sqlite           → 클론 시 초기화
[성공 스킬]     SKILL.md                  → 클론 시 그대로 이식
```

이 분류는 **인지과학의 Episodic vs Semantic Memory 모델**과 정확히 일치합니다. 학술적으로도 건전한 설계입니다.

### ✅ 현재 코드의 위험성을 정확히 짚었습니다

```javascript
// executor.js L229-230 — 현재 코드
const logEntry = `- **요청**: ${actualContent.slice(0, 150).replace(/\n/g, ' ')}\n`
```

이 한 줄이 **기밀 유출의 직접적 경로**입니다. `actualContent`에는 대표님이 텔레그램에서 보낸 원문이 그대로 담겨 있고, 이것이 `SKILL.md` 파일에 appendFile됩니다.

그런데 `SKILL.md`는 Git 저장소에 포함되어 있으므로:
```
대표님이 "삼성전자 마케팅 전략 짜줘"라고 보냄
→ SKILL.md에 "삼성전자 마케팅 전략..." 기록
→ git add . && git commit
→ Git 히스토리에 영구 보존
→ 클론 패키징 시 .git을 포함하면 → 기밀 유출
→ .git을 제외해도 → 파일 자체에 원문이 남아있음
```

**기획서의 문제 인식은 100% 정확합니다.**

### ✅ "힌트 리스트" 접근법 (법칙이 아닌 참조)

```
❌ "반드시 이렇게 해라" (강제 주입)
✅ "이전에 다른 크루가 이런 방식으로 돌파했는데" (참조 제안)
```

이 설계가 왜 중요하냐면, 강제 주입 시 **의미론적 드리프트(Semantic Drift)** 문제가 발생합니다. 마케팅 크루의 성공 패턴이 리서치 크루에게 잘못 적용되면 오히려 성능이 저하됩니다. "힌트"로 제공하고 에이전트가 자율 판단하게 하는 것은 건전한 설계입니다.

## 🔵 추가 발견: 즉시 수정해야 할 코드 (Pre-existing Risk)

### executor.js L229 — 원문 저장 즉시 제거 권고

이 기획이 "나중에 구현"으로 분류되어 있지만, **현재 이미 작동 중인 Self-Learning 코드가 원문을 저장하고 있습니다:**

```javascript
// ⚠️ 현재 프로덕션에서 돌고 있는 코드
const logEntry = `\n### [${new Date().toISOString().slice(0,10)}] Self-Learning Pattern 🧠\n` +
  `- **요청**: ${actualContent.slice(0, 150).replace(/\n/g, ' ')}\n` +  // ← 원문!
  `- **모델**: ${result.model}\n` +
  `- **카테고리**: ${evaluation.category}\n` +
  `- **스코어**: ${evaluation.score}\n`;
```

**이것은 "나중에 바꾸장" 문제가 아닙니다.** 지금 당장 대표님이 텔레그램에서 기밀이 포함된 메시지를 보낼 때마다 `SKILL.md`에 기록되고 있습니다.

```diff
 // 즉시 패치 — actualContent 제거, 메타데이터만 보존
 const logEntry = `\n### [${new Date().toISOString().slice(0,10)}] Self-Learning Pattern 🧠\n` +
-  `- **요청**: ${actualContent.slice(0, 150).replace(/\n/g, ' ')}\n` +
+  `- **패턴**: [${evaluation.category}] ${evaluation.reason}\n` +
   `- **모델**: ${result.model}\n` +
   `- **카테고리**: ${evaluation.category}\n` +
   `- **스코어**: ${evaluation.score}\n`;
```

이 1줄 변경으로:
- 원문 유출 경로 차단
- `evaluation.reason`은 modelSelector가 이미 생성한 메타데이터이므로 추가 LLM 호출 불필요
- 로그의 가치는 유지 (어떤 유형의 작업이 성공했는지 추적 가능)
