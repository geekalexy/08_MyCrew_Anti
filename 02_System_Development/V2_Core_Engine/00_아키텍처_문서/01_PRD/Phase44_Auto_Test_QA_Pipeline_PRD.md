## 1. 개요 및 목적
본 기획서는 Phase 43에서 완성된 `/auto_run` 파이프라인의 후속 단계로, 생성된 코드를 자율적으로 검증하는 **`/auto_test` (QA 에이전트 파이프라인)**의 동작 원리를 정의합니다. 

아울러, 기획/개발/테스트가 꼬리를 물고 반복(Iterative)되는 과정에서 에이전트의 컨텍스트가 오염되는 것을 방지하기 위해 **"재작업 불변성(Immutable Rerun Isolation)"**과 **"이력 영속성(Banner Persistence)"** 아키텍처를 시스템 레벨에서 확립합니다.

---

## 2. 배너 게이트웨이 및 영속성 (Gateway & Persistence)

`/auto_run`이 완료된 태스크는 단순히 상태만 `REVIEW`나 `DONE`으로 변하고 끝나는 것이 아닙니다. 완료된 태스크는 뚜렷한 "개발 완료 이력"을 지니며, 다음 파이프라인의 **트리거 게이트웨이**가 됩니다.

### 2.1. DB 기반 이력 영속성
- **문제점**: 프론트엔드 React 상태만으로 관리되던 배너는 새로고침 시 증발하여 이력을 추적할 수 없었습니다.
- **해결책**: `tasks` 테이블에 `last_autorun_status`, `last_autorun_step`, `last_autorun_max_steps`, `last_autorun_at` 컬럼을 신설합니다.
- **결과**: 사용자는 어떤 카드가 언제, 몇 스텝만에 Auto Run으로 완성되었는지 영구적인 배지(Badge) `✅ Auto Run 완료 — Step 15/15` 형태로 언제든 확인할 수 있습니다.

### 2.2. 게이트웨이(Gateway) 역할
- 완료 배너는 단순한 표식이 아니라 **다음 액션을 위한 UI/UX 진입점**입니다.
- 완료된 DEV 카드에서 즉시 **"QA 검증(Auto Test) 시작"** 버튼(또는 명령어)을 노출하여, 끊김 없는 파이프라인 경험을 제공합니다.

## 3. `/auto_test` 코어 워크플로우 (QA Pipeline)

사용자가 완료된 Auto Run 카드에서 `/auto_test`를 트리거하면 다음과 같은 절차가 진행됩니다.

### Step 1. 독립된 QA 카드(태스크) 자동 생성
- `/auto_test`는 원래의 DEV 카드 안에서 실행되지 않습니다. (컨텍스트 혼합 방지)
- 시스템은 해당 DEV 카드를 부모로 삼는 **완전히 새로운 QA 모드 태스크 카드**를 대시보드에 생성(Fork)합니다.

### Step 2. 검증 시나리오 자율 작성 (Test Plan Generation)
- QA 에이전트는 기획서(Plan Master가 작성한 `.txt` PRD)와 DEV 에이전트가 생성한 **코드베이스 전체**를 타겟으로 삼아 분석합니다.
- 특정 카드의 지엽적인 작업물만 검증하는 것이 아니라, 해당 작업이 전체 아키텍처(Graphify)에 미치는 영향을 바탕으로 **종합적인 검증 시나리오**를 스스로 작성합니다.

### Step 3. 자율 테스트 실행 및 피드백 (Execution)
- QA 에이전트는 작성된 시나리오를 바탕으로 쉘(Shell) 스크립트 실행, E2E 봇(Puppeteer 등) 가동, 또는 API 호출을 통해 실제 테스트를 수행합니다.
- 실패(Fail) 시, 에러 로그와 함께 수정 가이드를 정리하여 기록합니다.

## 4. 재작업 시 원본 카드 격리 (Immutable Rerun Isolation)

테스트가 실패하여 재작업이 필요하거나, CEO가 마음에 들지 않아 새로운 `/run` 또는 `/auto_run`을 지시할 때 가장 중요한 룰입니다.

### 4.1. 컨텍스트 오염 (인지 부조화) 방지
- **기존 방식**: 동일한 카드 안에서 에이전트를 다시 호출하면, 이전 Auto Run이 남긴 수십 개의 코멘트(툴 호출 로그, 생성된 코드 파편 등)가 고스란히 컨텍스트 창에 입력됩니다. 에이전트는 "이미 코드를 다 짰는데 왜 또 하라는 거지?"라는 인지 부조화에 빠집니다.
- **불변성(Immutability) 강제**: 한 번 `COMPLETED`된 (이력이 남은) 카드에서는 어떠한 에이전트 실행 명령어(`/auto_run`, `/run`)도 원본을 재활용하지 않습니다.

### 4.2. 자동 Fork 아키텍처
1. 사용자가 재작업을 지시합니다.
2. 백엔드 루틴은 현재 카드를 락(Lock) 걸고 아카이브(Archived) 처리하여 "수정 불가한 이력"으로 영구 보존합니다.
3. 동일한 기획서, 동일한 프롬프트를 바라보는 **새로운 DEV 카드(버전 2)를 Fork(복제) 생성**합니다.
4. 새로운 카드는 깨끗한 빈 코멘트 창을 가지므로, 에이전트는 오염 없는 최상의 컨텍스트에서 오류를 바로잡은 2차 Auto Run 파이프라인을 재가동합니다.
5. 이 원칙은 **Plan Master** 기획 재작업 시에도 동일하게 적용됩니다 (새 버전의 PRD `.txt` 생성 및 새 백로그 그룹 생성).
