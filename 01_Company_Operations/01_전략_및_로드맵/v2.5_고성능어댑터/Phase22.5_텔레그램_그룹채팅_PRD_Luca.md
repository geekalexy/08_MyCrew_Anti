# Phase 22.5: Telegram Direct Bridge & Multi-Agent Group Chat (PRD)

> **작성자**: Luca (Antigravity Chief Architect)
> **작성일**: 2026-04-28
> **상태**: 기획 초안 (Draft)

## 1. 개요 (Overview)
**목표**: CEO와 다수의 AI 에이전트(Ari, Luca, Prime, Sonnet)가 텔레그램 그룹 채팅방에서 실시간으로 상의하고, 업무를 지시하며, 에이전트 간의 협업을 조율할 수 있는 "Virtual War Room(가상 상황실)"을 구축한다.

**핵심 요건**: 단순 알림이나 1회성 명령(One-shot) 처리가 아닌, **연속적인 문맥(Context)을 유지하는 양방향 대화(Conversation)**가 완벽히 지원되어야 한다. (소놀봇 수준의 대화형 인터페이스 구현)
**추가 요건 (핵심 분리)**: 비서(Ari)는 실시간 스트리밍 대화와 빠른 응답성(Function Calling)이 생명이므로 **기존 Gemini API 직접 호출 방식을 유지**한다. 반면, Luca, Prime 등 전문 실행 크루들은 **무료/초고성능 브릿지(Sonnet/Opus)**로 100% 스왑(Swap)하여 2-Track 체제로 운영한다.

## 1.5. 3대 동적 연결(Dynamic Routing) 개발 원칙
이전 하드코딩의 폐해를 막기 위해, 본 PRD의 모든 구현은 아래 3가지 동적 설계 원칙을 준수한다.
1. **명령어 ↔ 에이전트 동적 라우팅**: `/luca` 등의 텔레그램 명령어를 코드에 하드코딩하지 않고, DB(`agentStore`)에 등록된 활성 에이전트 목록을 런타임에 조회하여 매칭한다.
2. **에이전트 ↔ 모델 식별자 동적 매핑**: 특정 에이전트를 호출할 때 모델명을 하드코딩(`anti-claude-sonnet-4.6-thinking`)하지 않고, UI 대시보드 및 DB에 설정된 해당 에이전트의 현재 모델 식별자를 런타임에 동적으로 끌어와 브릿지에 주입한다.
3. **실행 환경(어댑터) 다형성 보장**: 모든 실행기는 `BaseAdapter` 인터페이스를 상속해야 하며, 시스템은 분기문(`if/else`) 없이 다형성(Polymorphism)을 이용해 현재 활성화된 어댑터 플러그인을 동적으로 호출한다.

## 2. 페이즈별 구현 전략 (Phased Rollout)

### Phase A: 기존 아리 1:1 채널 테스트 (PoC)
새로운 봇을 파기 전, 기존 아리 봇 채널을 활용하여 '대화형 브릿지'의 뼈대를 테스트합니다.
*   **라우팅**: `/luca [메시지]`, `/prime [메시지]` 형태의 프리픽스를 사용하여 수신자를 지정.
*   **컨텍스트 유지**: `server.js`에서 각 에이전트별 대화 세션(최근 N개의 대화 기록)을 메모리 또는 SQLite에 캐싱하여 브릿지로 넘길 때 함께 전달.

### Phase B: 전용 다중 에이전트 그룹 채널 개설 (Production)
Phase A 통과 후, 텔레그램 그룹 채팅방을 개설하고 모든 에이전트(단일 통합 봇 또는 멀티 봇)를 초대합니다.
*   **라우팅**: 그룹 채팅방 내에서 `@luca`, `@prime`, `@ari` 등 이름으로 호출되며, 내부적으로는 PRD에 구체적인 모델 식별자를 하드코딩하지 않고, **DB(`agentStore`) 또는 `modelRegistry.js`의 런타임 조회를 통해 활성 모델 식별자를 동적으로 매핑**하여 전달되어야 합니다.
*   **디폴트 비서 API 유지**: 특정 명령어 없이 일상적인 대화(기존 Ari 응대)를 걸었을 때는 파일 브릿지가 아닌 **Gemini API 직접 호출(스트리밍)**을 태워 실시간 응답성을 보장합니다.
*   **교차 대화 (비활성화)**: 토큰 비용 과다 및 채팅창 도배 방지를 위해 Phase B에서는 교차 대화를 엄격히 제한하고 멘션된 에이전트만 응답하도록 한다. 교차 호출은 명시적 지시("@ari @luca 같이 검토해줘")가 있을 때만 한정하여 동작하게 하며, 완벽한 자율 교차 대화는 Phase C로 연기한다.

## 3. 핵심 메커니즘 설계 (Core Mechanics)

### 3.1. 대화 문맥(Context) 유지 방안
에이전트가 "대화"를 하려면 이전 대화 기록을 기억해야 합니다. 브릿지 파일 폴링 방식에서는 매 요청이 독립적(Stateless)이므로, 컨트롤 플레인(`server.js`)이 문맥을 주입해야 합니다.

1.  **세션 관리 (Session Manager)**: `server.js` 내에 텔레그램 채팅 ID 기반의 세션 관리자 도입.
2.  **페이로드 확장**: `.agents/tasks/pending/`에 떨어지는 JSON 파일에 `conversation_history` 배열을 추가.
    ```json
    {
      "taskId": "115",
      "agentId": "luca",
      "mode": "conversational",
      "conversation_history": [
        {"role": "user", "content": "이 코드 분석해줘"},
        {"role": "assistant", "content": "네, 메모리 누수가 있네요."},
        {"role": "user", "content": "어떻게 고쳐?"}
      ],
      "content": "어떻게 고쳐?"
    }
    ```

3.  **파일 크기 폭발 방지 (History Trimming)**: 그룹 채팅의 문맥이 무한정 길어지는 것을 막기 위해, `server.js`에서 히스토리를 주입하기 전 **최근 10턴(MAX_HISTORY_TURNS) 및 최대 8000자(MAX_HISTORY_CHARS)** 이내로 슬라이싱 및 압축하는 전처리 로직을 반드시 포함한다.

### 3.2. 데이터 흐름도 (Data Flow) - 대화형 브릿지 기준

**[수신 플로우]**
1.  **텔레그램 앱** ➔ 텔레그램 서버 ➔ **아리 엔진 (`server.js`)** (웹훅 또는 폴링 수신)
2.  **아리 엔진 ➔ 브릿지 큐 (`.agents/tasks/pending/`)**: 
    - 멘션된 에이전트 확인.
    - 해당 텔레그램 방의 이전 대화 내역(Context)을 합쳐 JSON 파일 생성.
3.  **아리 엔진 ➔ SQLite DB**: Task 기록 및 대시보드 동기화 (진행 상태 표시).

**[실행 및 반환 플로우]**
1.  **실행 어댑터 (BaseAdapter 상속 구현체)** ➔ `pending/` 폴더 감지 및 파일 읽기.
    - *핵심 원칙*: IDE 기반이든 CLI 왓처 기반이든, 모든 실행 어댑터는 반드시 `BaseAdapter` 인터페이스를 상속받아 구현해야 합니다. 이를 통해 추후 어댑터 교체 시에도 시스템 중단 없이 간편히 스왑(Swap)할 수 있습니다.
2.  **LLM 추론**: `conversation_history`를 컨텍스트로 삼아 스트리밍 혹은 일괄 추론 진행.
3.  **결과 기록**: ➔ **`.agents/tasks/completed/`** 에 결과 JSON(응답 텍스트 포함) 반환.
4.  **아리 엔진 (`server.js`)** ➔ `completed/` 폴더 감지.
5.  **아리 엔진 ➔ 텔레그램 서버** ➔ 텔레그램 앱 (해당 그룹 채팅방에 에이전트 이름으로 메시지 전송).

## 4. 해결해야 할 기술적 허들 및 리스크 (Hurdles)

| 리스크 요인 | 설명 | 해결 방안 |
| :--- | :--- | :--- |
| **스트리밍(Streaming) 응답 지연** | 파일 폴링 방식은 처리가 다 끝난 후 결과를 반환하므로, 텔레그램에서 "타이핑 중..." 이나 실시간 스트리밍 경험을 주기 어렵습니다. | **해결책**: `server.js`가 파일을 넘긴 직후 텔레그램에 ⏳(모래시계) 이모지와 함께 "생각 중..." 메시지를 먼저 보내고, 결과가 오면 메시지를 수정(`editMessageText`)하는 방식 채택. |
| **병렬 처리 충돌** | 동시에 여러 명이 지시하거나, 긴 작업 중 다른 질문을 할 경우 파일 I/O 충돌 우려. | **해결책**: 각 요청 파일에 고유 UUID를 부여하고, 에이전트별 Lock(상태) 관리 로직 강화. |
| **텔레그램 역전송 (Watcher) 중복 위험** | 결과물을 텔레그램으로 전송하기 위해 별도의 Watcher 데몬을 구동하면 기존 시스템의 `AdapterWatcher`와 파일 I/O 충돌(Race Condition)이 발생합니다. | **해결책**: 별도 데몬 프로세스를 만들지 않고, 기존 `AdapterWatcher.js`의 작업 완료 콜백에 텔레그램 역전송(`bot.sendMessage`) 훅(Hook)을 추가하는 형태로 병합 확장합니다. |

## 5. Phase A (PoC) 개발 마일스톤 (Next Steps)

1.  **기존 텔레그램 핸들러 리팩토링 및 분리**: `server.js`에 비대하게 하드코딩된 기존 텔레그램 봇 처리 로직(L542-740 부근)을 완전히 들어내어 독립된 `telegram/telegramRouter.js` 모듈로 분리한다. 이를 통해 API 대화 봇과 브릿지 라우팅 봇을 병합 확장한다.
2.  **명령어 수신 라우터 개조**: `/luca`, `/prime` 명령어 수신 시 대화 히스토리를 묶어 `FilePollingAdapter`로 넘기도록 수정.
3.  **`FilePollingAdapter` 페이로드 규격 확장**: `conversation_history` 필드 추가.
4.  **역전송 훅(Hook) 연동**: 별도 데몬 개발 대신 기존 `AdapterWatcher.js`에 파일 처리 완료 시 텔레그램 API(`bot.sendMessage`)를 역호출하는 훅 연결.
4.  **E2E 테스트**: 텔레그램에서 `/luca 안녕?` ➔ 대답 수신 ➔ `/luca 내 이름이 뭐야?` ➔ 문맥 유지하여 대답하는지 확인.
