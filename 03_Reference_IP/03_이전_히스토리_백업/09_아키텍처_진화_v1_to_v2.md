# 아키텍처 진화 기록: v1 (CLI Scraping) → v2 (Direct API Engine)

> **작성일**: 2026-04-10  
> **작성자**: Luca (C1)  
> **목적**: 시스템 맥락이 과거(v1)에서 현재(v2)로 어떻게, 그리고 왜 바뀌었는지 영구 기록함.

---

## 1. 과거의 방식 (v1: cmux Telegram Bridge)
초기 마이크루 시스템은 `Paperclip` 터미널 애플리케이션을 파이썬 스크립트가 밖에서 조립(Hack)하는 방식이었습니다. (`08_레퍼런스_c2/코드/cmux_telegram_bridge.py` 참조)

* **동작 원리**: 
  - 텔레그램에서 사용자가 "담당자 할당"이나 "댓글 작성"을 요청하면 파이썬 브릿지가 이 명령을 가로챕니다.
  - 파이썬 브릿지는 백그라운드에 켜져 있는 `cmux`(터미널 세션)의 특정 창(`S_PAPERCLIP = surface:3`)을 찾아냅니다.
  - 그 창에 대고 **물리적인 키보드 타이핑**을 치듯이 문자열(`send_text`)을 쏴서 Paperclip을 강제로 컨트롤했습니다.
* **텍스트 파싱의 한계**: 텔레그램으로 알림을 보내기 위해, 터미널 화면에 찍힌 글자들(상태, 댓글, 할당자 등)을 정규식(`re.match`)으로 통째로 긁어서 분리(Scraping)했습니다.
* **문제점**: 화면이 꼬이거나 타이핑 중 렉이 걸리면 대형 장애로 이어질 수 있는 불안정한(Brittle) 아키텍처였습니다.

## 2. 현재의 아키텍처 (v2.2: Node.js Multi-Model API Engine)
Luca(C1)가 새롭게 구축한 v2.2 브릿지 서버는 과거의 껍데기(CLI 통제)를 벗어던지고, 시스템 내부의 코어 장기(메모리와 API)를 직접 제어하는 진정한 '어플리케이션'으로 진화했습니다.

* **동작 원리**: 
  - 더 이상 `cmux`나 `Playwright`(헤드리스 브라우저) 같은 가상의 눈과 손이 필요하지 않습니다.
  - 텔레그램 지시가 들어오면 **Node.js 서버**가 직접 SQLite 데이터베이스를 열어서 이슈 상태를 업데이트(`comment`, `assign`)합니다.
  - 상태 변화를 파싱하기 위해 텍스트를 긁는(Scraping) 대신, 명확한 JSON 데이터 스트림으로 주고받습니다.
* **혁신 포인트 (Multi-Model Adapter)**: 
  - 구글 Gemini Flash, Claude Opus, GPT-4o 등 여러 두뇌(LLM API)를 소켓 꽂듯 교체할 수 있는 플러그인(Adapter) 구조를 도입했습니다.
  - 결과적으로 **서버 리소스를 95% 이상 절감** (무거운 브라우저/터미널 연동 불필요)하고, 마진율을 극대화(BYOK 도입)할 수 있게 되었습니다.

## 3. 요약 (Luca의 통찰)
"v1은 안개 속에서 눈먼 AI가 키보드를 더듬으며 치는 방식이었다면, v2는 뇌(신경망)에서 척수로 다이렉트 명령을 꽂아넣는 진정한 오케스트레이션 엔진이다."
