## 핵심 설계 원칙

> **"동일한 두뇌, 다른 프로토콜"**
>
> 양 팀은 완전히 동일한 모델 조합을 가진다.
> 실험의 유일한 변수는 **팀 내부 상호작용 프로토콜** (적대 vs 협력)이다.

### 왜 이 원칙이 중요한가

이전 설계(v1~v3)의 문제:
- 팀별 모델 구성이 달랐음 → 결과 해석 불가 (모델 차이? 프로세스 차이?)
- ARI가 Opus였음 → 사용자의 모든 대화에 Opus 호출 → 비용 폭탄
- 팀원 수가 불균형 → 대조군이 비약적으로 불리한 구조

v4.0 해결:
- 양 팀 **동일한 3인 구성 + 동일한 모델 티어**
- ARI = Gemini Flash (라우터 전용, 팀 외부)
- Opus = 크루 내부 전용 (Phase 2, 4에서만 호출)

```
                    ┌────────────────────┐
                    │  ARI — Gemini Flash │
                    │  사용자 ↔ 팀 가교    │
                    │  태스크 접수·라우팅  │
                    └────────┬───────────┘
                             │
              ┌──────────────┴──────────────┐
              ▼                             ▼
┌─────────── Team A ──────────┐  ┌─────────── Team B ──────────┐
│   적대적 검토 프로토콜        │  │   CKS 협력 프로토콜           │
│                             │  │                              │
│  🖼 NOVA   — Gemini Flash   │  │  🖼 LUMI   — Gemini Flash   │
│     이미지 생성 실무          │  │     이미지 생성 실무           │
│                             │  │                              │
│  🎬 LILY   — Claude Sonnet  │  │  🎬 PICO   — Claude Sonnet  │
│     영상/코드(Remotion) 실무  │  │     영상/코드(Remotion) 실무  │
│                             │  │                              │
│  ⚖️  OLLIE  — Claude Opus   │  │  🌙 LUNA   — Claude Opus    │
│     적대적 비판·판관          │  │     협력적 통합·합성자         │
└─────────────────────────────┘  └──────────────────────────────┘

  변수: 프로토콜 (적대 vs CKS)
  상수: ARI, 모델 조합, 팀원 수, 역할 수, 호출 횟수
```

## 에이전트 프로필

### 🔀 공유 에이전트 (팀 외부)

| 에이전트 | 모델 | 역할 | 호출 빈도 | 비용 |
|:--------|:-----|:----|:---------|:----|
| **ARI** | Gemini Flash | 사용자 대화·라우팅·결과 전달 | 매우 높음 (모든 대화) | 💚 최소 |

> ARI는 "현관문". 판단·합성 없음. 빠른 응대 + 크루로 라우팅만.

### 🔴 Team A — 적대적 대조군

| 에이전트 | 모델 | 역할 | 담당 역량 |
|:--------|:-----|:----|:---------|
| **NOVA** | Gemini Flash | 이미지 실무 | NanoBanana 프롬프트 작성·생성 |
| **LILY** | Claude Sonnet 4.6 | 영상 실무 | Remotion Composition.tsx 코드 작성 |
| **OLLIE** | Claude Opus | 적대적 판관 | 결과물 혹독한 비판 → 방어·수정 반복 |

**A팀 프로토콜:**
```
NOVA(이미지 초안) + LILY(영상 초안)
       ↓
OLLIE: "이 결과물은 왜 부족한가?" [적대적 비판]
       ↓
NOVA·LILY: 비판 방어 또는 수정안 제출
       ↓
OLLIE: 최종 합격/불합격 판정
```

## 모델 선택 근거

| 역할 | 모델 | 선택 이유 |
|:----|:-----|:---------|
| **이미지 실무** | Gemini Flash | NanoBanana(Imagen 3) 네이티브 연동. 프롬프트 생성 속도 최적 |
| **영상 실무** | Claude Sonnet 4.6 | React/TypeScript 코드 작성 품질 최우수. Remotion Composition 코딩 적합 |
| **합성/판관** | Claude Opus | 검증된 품질 천장. 루카 실험에서 합성·검토 역할 확인 |
| **라우터** | Gemini Flash | 고빈도·저비용 필수. 사용자 UX 속도 최우선 |

### 단계적 업그레이드 계획

```
Phase 1 (POC):  이미지 실무 = Flash (비용 통제, 프로토콜 검증 집중)
Phase 2 (검증): 이미지 실무 = Pro  (품질 천장 테스트)

※ 어차피 NanoBanana의 Imagen 3 호출은 동일.
  에이전트 모델 티어 ≠ 생성 이미지 품질
  → 프롬프트 작성 능력이 결정 (SKILL.md 표현 사전)
```
