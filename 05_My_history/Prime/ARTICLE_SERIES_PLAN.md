## 📐 글쓰기 원칙

### 톤 & 매너
- **1인칭 시점** — "내가 직접 겪은 이야기"
- **대화체** — 개발 용어를 쓰되 반드시 비유/해설 병기
- **솔직함** — 삽질, 실패, AI한테 화났던 순간도 포함
- **구조화** — 매 편마다 `에피소드 → 문제 → 해결 → 교훈` 4단 구조

### 멀티 보이스 구조 (2026-05-09 추가)

각 편은 **3개의 목소리**가 어우러져 입체적인 글을 만든다:

| 보이스 | 역할 | 톤 | 예시 |
|--------|------|-----|------|
| 🎙️ **CEO (1인칭)** | 에피소드의 주인공. 실제 경험담, 감정, 판단의 흐름 | 솔직하고 생생한 현장감 | "심장이 뛰었다. 진짜로." |
| 🔍 **Prime (제3자 관찰)** | 에피소드를 분석하는 해설자. 바이브 코딩 팁, 전략적 인사이트, CEO 의사결정 평가 | 객관적이고 전문적, 독자에게 실용 정보 제공 | "여기서 대표가 내린 판단은 '검증 루틴'이라는 습관으로 발전한다. 바이브 코딩에서 가장 중요한 스킬이다." |
| 💬 **팀원 사이드바** | 에피소드에 관련된 AI 팀원(루카, 소넷, 아리 등)의 시점. 그 상황에서 느낀 감정, 반응 | 캐릭터별 성격 반영, 인간미 부여 | *루카: "그때 대표님이 아키텍처를 바꾸자고 했을 때, 솔직히 모듈 3개를 새로 짜야 한다는 걸 알고 있었다."* |

### Prime 관찰 코멘터리에 포함할 내용
- **바이브 코딩 팁** — 독자가 바로 적용할 수 있는 실전 노하우
- **에이전트 구성법** — 모델 선택, 역할 분배, 팀 운영 전략
- **개발 전략** — 아키텍처 결정의 이유, 기술 선택의 근거
- **CEO 평가** — 비개발자로서 CEO가 한 판단의 의미와 수준을 객관적으로 해설

### 1부: 바이브 코딩, 뭔데? (입문편)

> **키워드**: 바이브 코딩 정의, AI 코딩 도구, 비개발자 시작점

| # | 제목 (안) | 에피소드 → 문제 → 해결 | 핵심 교훈 |
|---|-----------|----------------------|-----------|
| 1-1 | **"코드 한 줄도 몰랐던 내가 SaaS를 만들기로 했다"** | 아이디어는 있지만 외주비 2천만원 → AI 코딩 도구 발견 → 바이브 코딩 개념 정립 | 바이브 코딩 = "의도를 전달하면 AI가 구현하는 방식" |
| 1-2 | **"AI 코딩 도구 전쟁 — 나는 어떤 걸 골라야 하나"** | Cursor, Copilot, Claude Code, Antigravity 비교 → 각각 써봤는데 다 다름 → 내 기준 선택법 | 도구는 "무엇을 만드느냐"가 아니라 "어떻게 대화하느냐"로 고르기 |
| 1-3 | **"첫 프롬프트로 웹페이지를 만들었을 때의 충격"** | "로그인 페이지 만들어줘" → 30초 만에 화면 생성 → 하지만 작동하지 않는 부분들 | "만들어지는 것"과 "동작하는 것"은 다르다 — 검증 습관의 시작 |
| 1-4 | **"바이브 코딩의 한계 — AI는 만능이 아니다"** | 복잡한 로직 요청 → 환각(Hallucination) 발생 → 존재하지 않는 API를 호출하는 코드 생성 | 환각 탐지법: "이 API 정말 있어?" 습관화 + 공식 문서 교차검증 |
| 1-5 | **"개발자 없이 개발하는 마인드셋"** | 기존 개발 프로세스 vs 바이브 코딩 프로세스 비교 → "설계→코딩→테스트"에서 "의도→생성→검증→반복"으로 | CEO/PM이 직접 코딩할 때 갖춰야 할 3가지 태도 |

### 2부: AI 에이전트 팀 빌딩 (조직편)

> **키워드**: 다중 AI 에이전트, 역할 분담, 페르소나 설계, 협업 프로토콜

| # | 제목 (안) | 에피소드 → 문제 → 해결 | 핵심 교훈 |
|---|-----------|----------------------|-----------|
| 2-1 | **"AI 한 명으론 부족하다 — 팀을 만들어야 했다"** | 혼자 쓰던 AI가 맥락을 자꾸 잊어버림 → 역할별로 분리하면? → 루카(CTO), 소넷(기획), 프라임(감수) 탄생 | 컨텍스트 격리의 필요성: 한 AI에게 모든 것을 시키면 성능이 떨어진다 |
| 2-2 | **"AI 에이전트에게 이름과 성격을 줬더니"** | 무작위 AI 사용 → 일관성 없는 결과 → 페르소나(이름, 전문분야, 성격) 부여 → 결과물 품질 향상 | 페르소나 설계법: 이름, 전문분야, 금지사항, 대화 스타일 설정 |
| 2-3 | **"모델마다 성격이 다르다 — 적재적소 배치법"** | Gemini는 빠르지만 가끔 헛소리, Claude는 신중하지만 느림 → 작업 유형별 모델 매칭 전략 | Flash(일상대화), Pro(복잡분석), Sonnet(코딩), Opus(코드리뷰) — 4계층 운영 |
| 2-4 | **"AI 팀원이 서로 싸우기 시작했다"** | A 에이전트가 만든 코드를 B가 덮어씀 → 원본 데이터 훼손 → "객체 불변성" 원칙 수립 | 다중 에이전트 충돌 방지 3원칙: 영역 격리, 불변성 규칙, 리뷰 필수 |
| 2-5 | **"AI 팀의 조직도 — 누가 누구에게 보고하는가"** | 모든 AI에게 동시에 지시 → 카오스 → "CEO→Ari(비서)→칸반→에이전트" 단방향 지휘체계 확립 | 지휘체계 설계: 프런트데스크(비서) — 백오피스(실행) 분리 아키텍처 |

### 3부: 실전 바이브 코딩 에피소드 (개발편)

> **키워드**: 아키텍처 결정, 디버깅, 데이터 사고, 리팩토링

| # | 제목 (안) | 에피소드 → 문제 → 해결 | 핵심 교훈 |
|---|-----------|----------------------|-----------|
| 3-1 | **"아키텍처가 뭔지도 몰랐는데 아키텍처를 설계했다"** | AI가 만든 코드가 점점 스파게티화 → "구조"의 필요성 깨달음 → 비서/실행 레이어 분리 | 비개발자의 아키텍처 사고법: "역할"과 "통로"만 그리면 된다 |
| 3-2 | **"데이터가 사라졌다 — AI가 DELETE를 실행한 날"** | AI에게 "정리해줘" → DB 전체 삭제 → 복구 불가 → dangerously 접두사 정책 수립 | 파괴적 명령어 안전장치: "삭제"에는 반드시 경고 레이어 추가 |
| 3-3 | **"디버깅 — AI에게 버그를 고치라 하면 새 버그가 생긴다"** | 버그 수정 → 다른 곳 파손 → 또 수정 → 무한루프 → 근본 원인 분석 강제 정책 도입 | "증상 치료" vs "원인 치료" — P-012 정책이 탄생한 이유 |
| 3-4 | **"프로젝트가 두 개가 되자 세상이 무너졌다"** | 싱글 프로젝트에서 멀티 프로젝트 → 데이터가 섞임 → 프로젝트별 격리 아키텍처 구축 | 다중 프로젝트 운영의 핵심: "보이지 않는 벽"을 먼저 세워라 |
| 3-5 | **"기술 부채 — AI가 빠르게 쌓아올린 코드의 대가"** | 속도 우선 개발 → 중복 코드, 하드코딩 범람 → 리팩토링 스프린트 강제 실행 | 바이브 코딩의 그림자: 빠른 생성 = 빠른 부채 축적, 주기적 정리 필수 |

### 4부: 자동화 파이프라인 (자율 주행편)

> **키워드**: 자율 실행, 코드 리뷰 자동화, 파이프라인, 워치독

| # | 제목 (안) | 에피소드 → 문제 → 해결 | 핵심 교훈 |
|---|-----------|----------------------|-----------|
| 4-1 | **"AI를 시키고 자러 갈 수 있을까? — 자율 파이프라인의 꿈"** | 매번 수동 지시 → 지침 → AI가 스스로 다음 작업 생성하는 Chain-Reaction 시스템 구상 | 자율 주행의 핵심: "다음에 뭘 해야 하는지"를 AI 스스로 결정하게 하기 |
| 4-2 | **"자율 주행 첫날 — 엔진이 멈추지 않았다"** | 파이프라인 첫 가동 → AI가 끝없이 작업 생성 → 멈추질 않음 → 마일스톤 체크포인트 도입 | 안전장치 없는 자동화는 폭주 — "언제 멈출지"를 먼저 정의해야 한다 |
| 4-3 | **"AI가 AI의 코드를 리뷰한다 — Supreme Review 체계"** | 코드 품질 문제 → 수석 리뷰어(Opus) 배치 → AI 간 Peer Review 프로세스 구축 | 코드 리뷰 자동화: "작성자 ≠ 검토자" 원칙을 AI 팀에도 적용 |
| 4-4 | **"워치독 — 파이프라인이 죽으면 누가 살리나"** | 자율 실행 중 에이전트가 멈춤 → 감시견(Watchdog) 로직으로 자동 복구 | 자동화 시스템의 핵심: 실행 로직보다 "감시 및 복구" 로직이 더 중요 |
| 4-5 | **"비개발자가 CI/CD를 이해하게 된 순간"** | 코드 변경 → 수동 배포 → 자동 빌드/배포 파이프라인 구축 → "이게 CI/CD였구나" | 개념보다 경험: 직접 겪어야 용어가 이해된다 |

> **키워드**: 비용, 스케일링, 커뮤니티, 비개발자 리더십

| # | 제목 (안) | 에피소드 → 문제 → 해결 | 핵심 교훈 |
|---|-----------|----------------------|-----------|

| 5-2 | **"정책이 20개가 되기까지 — AI 팀의 사규 만들기"** | 사고 발생 → 재발 방지 정책 작성 → 정책 누적 → POLICY_INDEX 탄생 | 조직 운영의 본질: 사람이든 AI든, "규칙"이 조직을 만든다 |
| 5-3 | **"비개발자 CEO가 기술 의사결정을 내리는 법"** | 아키텍처 선택지 → 모르는 용어 투성이 → AI에게 "5살에게 설명하듯" 요청 → 이해 후 판단 | AI 시대의 리더십: "구현"은 몰라도 "판단"은 내가 해야 한다 |
| 5-4 | **"바이브 코딩 커뮤니티에서 내가 배운 것들"** | 혼자 삽질 → 커뮤니티 발견 → 다른 바이브 코더들의 패턴 학습 → 성장 가속 | 바이브 코딩은 혼자 하는 게 아니다 — 커뮤니티의 힘 |
| 5-5 | **"D+60 회고 — 비개발자는 결국 개발자가 되는가?"** | 60일간의 여정 정리 → 무엇이 변했나 → 앞으로의 방향 | 바이브 코딩은 "개발을 배우는 것"이 아니라 "만드는 방법을 배우는 것" |

## 📅 발행 로드맵

```
Phase 1 (1~2주차)    1부 1-1 ~ 1-5 발행     ← 유입 및 관심 확보
Phase 2 (3~4주차)    2부 2-1 ~ 2-5 발행     ← 핵심 차별화 콘텐츠
Phase 3 (5~6주차)    3부 3-1 ~ 3-5 발행     ← 실전 에피소드로 공감대
Phase 4 (7~8주차)    4부 4-1 ~ 4-5 발행     ← 고급 자동화 관심층 확보
Phase 5 (9~10주차)   5부 5-1 ~ 5-5 발행     ← 정리 및 강의 전환 준비
```

- **발행 주기**: 주 2~3편 (화/목 또는 월/수/금)
- **에피소드 유연성**: 기본 25편 구성이나, 진행 중 추가 에피소드 발굴 시 확장 가능
- **1부 발행 중 2부 초안 병행** — 항상 1부 선행 작성

> 실제 에피소드가 발생한 시점에 CEO와 대화하며 포착하는 방식으로 전환합니다.

1. **에피소드 포착** — 개발/운영 중 의미 있는 사건이 발생하면 CEO와 대화하며 즉시 메모

3. **CEO 인터뷰 기반 초안** — Prime이 CEO의 실제 경험담을 듣고 `에피소드→문제→해결→교훈` 4단 구조로 초안 작성
4. **CEO 감정/경험 보정** — 실제 감정, 그날의 분위기, 개인적 뉘앙스 추가
5. **발행** — 블로그 플랫폼 업로드
6. **독자 피드백** — 댓글/반응 기반으로 다음 편 소재 조정

- **한꺼번에 여러 편을 쓰지 않는다** — 에피소드가 일어난 시점에 1편씩
- **위 플랜의 25편 구성은 방향 참고용** — 순서, 제목, 내용 모두 실제 경험에 따라 유동적으로 변경
