# [06] 온보딩 보안 아키텍처 설계 (KeyProvider)

**작성:** Luca | **Phase:** 20 준비 | **Prime 참조 리뷰:** 6차

---

## 목표
로컬 개발 환경(`.env` 의존)에서 다수의 고객이 자신만의 API 키를 사용하는 **SaaS 멀티 테넌트 환경**으로 전환하기 위한 보안 기밀 관리 아키텍처를 수립한다.

---

## 핵심 설계

### 1. KeyProvider 3-Tier Bridge
```
Priority 1: 메모리 캐시 (Map)   → 성능 (서버 재시작 불필요)
Priority 2: DB (user_settings) → 온보딩 위저드로 저장한 키
Priority 3: .env               → 레거시/개발자 환경 폴백
```

### 2. /api/secrets 분리 엔드포인트
- 일반 `/api/settings`와 달리 **화이트리스트 키만 수락** (예: `GEMINI_API_KEY`, `ANTHROPIC_API_KEY`).
- GET 응답은 반드시 `getMaskedKey()` 마스킹 처리 (`AIza••••HYxY`).
- Socket.io `io.emit` 호출 전면 금지 (보안 방벽).

### 3. 어댑터 Lazy Initialization
- `GeminiAdapter`, `AnthropicAdapter` 생성자에서는 클라이언트 객체 미초기화.
- `_ensureClient()` 호출 시점에 `keyProvider.getKey()`를 통해 키를 불러와 즉석 초기화.
