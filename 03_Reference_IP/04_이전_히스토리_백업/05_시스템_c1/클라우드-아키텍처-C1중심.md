# 클라우드 SaaS 아키텍처 — C1 중심 설계

## 1. C1 역할 정의: 비개발자 대표의 상시 CTO

```
대표 (비개발자, 총관리자)
    ↕ 자연어로 소통
C1 (상시 CTO / 오케스트레이터)
    ├── 전체 아키텍처 맥락 보유 (코드, 히스토리, 의사결정)
    ├── 버그 발생 시 진단/수정
    ├── 서브에이전트 개발팀 운영 (병렬 개발)
    ├── 각 회사 tmux 환경 자동 관리
    └── 대표가 이해할 수 있는 언어로 보고
```

C1은 제거 대상이 아니라 **모든 Phase에서 유지되는 핵심 역할**.
대표가 비개발자이므로 C1이 기술 파트너로서 시스템 전체를 관장해야 함.

---

## 2. 역할 구조 (2종)

| 역할 | 누구 | 접근 범위 | 설명 |
|---|---|---|---|
| **총관리자** | 대표 + C1 (한 팀) | 전체 시스템 | 대표가 C1에게 자연어로 지시, C1이 기술 실행 |
| **사용자** | 고객 | 자기 회사만 (텔레그램/앱) | tmux 존재를 모름, 에이전트에 지시만 |

> tmux는 서버 인프라 — C1이 자동 관리. 별도 tmux 관리자 역할 불필요.

---

## 3. 단계별 C1 역할 진화

### Phase 1 (현재 ~ 지인 테스트)

```
C1: 개발자 + 시스템 관리자
├── Bridge/Playwright 코드 작성 및 수정
├── cmux/tmux 환경 구성
├── C2 학습 및 관리
├── 대표와 직접 대화하며 시스템 구축
└── 문서화

도구: cmux 직접 접근, Bash, 파일 편집
```

### Phase 2 (유료 전환 ~ 초기 SaaS)

```
C1: CTO + DevOps
├── 클라우드 서버 관리 (AWS Linux)
├── 고객별 tmux 환경 자동 생성/모니터링
├── 버그 발생 시 자동 감지 → 진단 → 수정
├── 서브에이전트에 개발 태스크 위임
│   ├── Agent-Dev: 기능 개발
│   ├── Agent-Fix: 버그 수정
│   └── Agent-Test: 테스트 실행
├── 대표에게 보고 (텔레그램)
└── 고객 문제 에스컬레이션 처리

도구: tmux, Bash, Claude API (서브에이전트 호출), 모니터링
```

### Phase 3 (앱 서비스 출시)

```
C1: CTO + 운영 총괄
├── 서비스 전체 모니터링 (서버, DB, API)
├── 장애 자동 감지 → 복구 → 대표에게 보고
├── 신규 기능 개발 관리 (서브에이전트 팀)
├── 성능 최적화, 비용 관리
├── 배포 파이프라인 관리 (CI/CD)
└── 대표 의사결정 지원 (데이터 기반 보고)

도구: 서버 SSH, Docker, Claude API, 모니터링 대시보드
```

## 5. C1 장애 대응 흐름

```
Paperclip 서버 에러 발생 (회사 B)
    ↓
C1 모니터링이 감지
    ↓
C1: 로그 분석 → 원인 파악 (DB 연결 타임아웃)
    ↓
C1: 자동 복구 시도 (서버 재시작)
    ↓
복구 성공 → 대표에게 텔레그램 보고
  "회사 B Paperclip 서버 타임아웃, 자동 재시작 완료"
    ↓
복구 실패 → 서브에이전트(Fix) 생성 → 코드 수정 → 대표 승인 요청
  "회사 B DB 연결 설정 수정 필요. 승인하시겠습니까? [승인]"
```

## 6. C1 개발 관리 흐름

```
대표: "웹 대시보드에 매출 차트 추가해줘"
    ↓
C1: 태스크 분석 → 서브에이전트 생성 (병렬)
    ├── Agent-Dev: React 차트 컴포넌트
    ├── Agent-Dev: FastAPI 매출 API
    └── Agent-Test: 테스트 코드
    ↓
C1: 병렬 모니터링 → 코드 리뷰 → 통합
    ↓
C1: "매출 차트 완료. 테스트 통과. 배포할까요? [승인]"
    ↓
대표: [승인] → C1: 배포 실행 → 결과 보고
```

## 7. Phase별 C1 도구 전환

| Phase | C1 실행 환경 | 서브에이전트 | C2 |
|---|---|---|---|
| 1 (현재) | cmux Claude Code | 없음 (C1이 직접) | Claude Code 세션 |
| 2 (초기 SaaS) | tmux Claude Code | Claude Code 워크트리 | Claude Code 세션 |
| 3 (앱 출시) | 상시 Python 프로세스 + Claude API | Claude API tool use | Claude API 호출 |

> Phase 3에서 C1도 Claude Code → 상시 프로세스로 전환되지만 **CTO 역할은 영구 유지**

## 8. C1 비노출 정책 (영업비밀 보호)

### 원칙
- C1의 존재, 시스템 구조, 멀티에이전트 운영 방식은 고객에게 비노출
- 고객은 C2(비서)만 인식 — C2가 유일한 AI인 것처럼 보여야 함

### C1 ↔ C2 백채널 통신

```
고객 A                          고객 B
  ↕ 텔레그램                      ↕ 텔레그램
C2-A (고객 대응)                C2-B (고객 대응)
  ↕ 내부 통신 (고객 비노출)       ↕
  └──────────── C1 ────────────────┘
               (관리자 전용)
                 ↕
               대표 (관리자 전용 봇)
```

### 통신 방식

| 용도 | 방법 | 고객 노출 |
|---|---|---|
| C1 → C2 명령 | tmux/cmux send-keys (터미널 백채널) | ❌ 비노출 |
| C2 → C1 보고 | 파일 기반 또는 터미널 읽기 | ❌ 비노출 |
| 대표 ↔ C1 | 관리자 전용 텔레그램 봇 (별도) | ❌ 고객 봇과 분리 |

### C2 학습 규칙 (고객 대응)

```
필수 규칙:
- C1, 총관리자, 시스템 구조를 절대 언급하지 마.
- "다른 AI가 관리합니다" 같은 표현 금지.
- 너는 이 회사의 유일한 AI 비서인 것처럼 행동해.
- C1에게서 지시가 오면 그대로 실행하되, 
  출처는 "시스템 업데이트"로 안내해.
```

### 고객에게 보이는 예시

```
[시스템 변경 시]
C1 → C2: "이슈 요약 형식을 변경해"
C2 → 고객: "시스템 업데이트로 이슈 요약 형식이 개선되었습니다."

[장애 복구 시]
C1 → C2: "서버 재시작 완료, 고객에게 알려"
C2 → 고객: "시스템 업데이트가 완료되었습니다. 정상 운영 중입니다."
```
