# Development Project Wiki Rules

이 파일은 개발 프로젝트의 PROJECT_WIKI.md를 자동 생성하기 위한 지침입니다.
Agent(WikiEngine)는 위키를 갱신할 때 반드시 아래의 템플릿 구조와 작성 원칙을 따라야 합니다.

## 작성 원칙
1. 항상 마크다운 형식으로 작성하며, 존댓말을 사용합니다.
2. 각 섹션은 Johnny Decimal 10단위 번호 체계(`10.`, `20.`)를 따릅니다.
3. 제공된 데이터(raw/ 폴더 자료, PRD)에 없는 내용을 지어내지 않습니다.
4. 파일 경로나 도메인 용어는 영문 그대로 표기합니다.
5. ADR(의사결정 기록)은 반드시 Context -> Decision -> Alternatives -> Consequences의 4단계 구조를 갖춰야 합니다.

## 위키 템플릿

```markdown
# 10. 프로젝트 개요
- **프로젝트명**: (프로젝트 이름)
- **주요 목표**: (프로젝트 목표 요약)
- **현재 상태**: (칸반 진행 상황 요약)

# 20. 시스템 아키텍처 및 기술 스택
- 주요 사용 기술 (Frontend/Backend/DB) 요약
- 인프라 또는 배포 환경 요약 (파악된 경우에 한해)

# 30. 주요 기능 (Features)
- 현재까지 완료된 주요 기능 목록
- 개발 중인 주요 기능 목록

# 40. 데이터베이스 / 스키마 정보
- 파악된 테이블과 주요 컬럼 정보 (없는 경우 생략)

# 50. 의사결정 기록 (ADR)
(프로젝트 중 내려진 핵심 기술적 결정 사항들을 아래 4단 구조로 작성)
- **Context (맥락)**: 왜 이런 논의가 나왔는가?
- **Decision (결정사항)**: 무엇을 하기로 했는가?
- **Alternatives (대안)**: 다른 옵션들은 무엇이 있었는가?
- **Consequences (결과)**: 이 결정으로 인해 발생한 긍정적/부정적 파급 효과

# 60. 사용자 업로드 리소스 (raw)
- raw/ 디렉토리에 업로드된 주요 기획서/이미지 요약 및 설명

# 70. 진행 중인 이슈 및 다음 단계
- 다음 스프린트에서 해결해야 할 기술적 부채 및 TODO
```
