# Mini-App Development Autonomous Pipeline (Agent Chain Workflow)

이 워크플로우는 대표님(CEO)의 개입을 최소화하면서, 기획부터 최종 코드 배포까지 MyCrew 에이전트들이 자율적으로 협업하는 "미니앱 개발 파이프라인" 규약입니다. 

에이전트들은 태스크 카드 본문이나 댓글에 `@workflow: mini-app-dev` 태그가 있으면 이 워크플로우를 최우선으로 따릅니다.

## 🔄 워크플로우 5단계 (Agent Handoff Chain)

**[유연한 역할 할당 (Dynamic Roles)]**
이 파이프라인은 고정된 에이전트가 아닌 **'역할(Role)'**을 기반으로 작동합니다. 
기본 담당자가 지정되어 있으나, 대표님(CEO)의 지시나 시스템 상황(토큰 리밋 등)에 따라 언제든 담당자가 교체되어 이어받을 수 있습니다.

### Phase 1: 아키텍처 기획 & PRD 작성
- **기본 담당자**: Luca (설계) 또는 지시받은 에이전트
- **액션**: 
  1. 대표님의 아이디어를 바탕으로 시스템 아키텍처 및 PRD(Product Requirements Document)를 작성.
  2. 작성된 문서를 `07_OUTPUT/MiniApp/[앱이름]_PRD.md` 로 저장.
- **Handoff (인계)**: 문서 저장이 완료되면 칸반 태스크 도구(`updateKanbanTask`)를 사용하여 **담당자를 `Prime`으로 변경** (상태는 `NO_CHANGE` 또는 `todo`).

### Phase 2: 리뷰 어드바이스 (PRD 검증)
- **기본 담당자**: Prime (슈프림 어드바이저)
- **액션**: 
  1. Luca가 작성한 PRD를 검토하고 기술적 타당성, 보안성, 구조적 결함을 지적.
  2. 수정이 필요하면 Luca에게 다시 넘기거나 직접 문서를 수정하여 보완.
- **Handoff (인계 - 착수 승인 요청)**: 검증이 끝나면 **담당자를 `CEO`로 변경하고 상태를 `REVIEW` (승인 대기)** 로 변경. 코멘트에 "PRD 검증 완료. 착수 승인 부탁드립니다." 명시.

### Phase 3: 착수 승인 (Kick-off)
- **담당자**: CEO (대표님)
- **액션**: 
  1. `Review` 대기열에 도착한 카드를 확인하고 PRD를 열람.
  2. 문제없다면 Task Detail Modal에서 **담당자를 `Sonnet` (코딩)으로, 상태를 `To Do`로 변경**하여 개발 지시. (수정 코멘트가 있다면 함께 작성)

### Phase 4: 코딩 > 코드 리뷰 > 테스트 > 디버깅 (Dev Loop)
- **기본 담당자**: Sonnet (코딩 리드) ↔ Prime (코드 검증)
- **대체 담당자 (Fallback)**: Sonnet의 토큰 리밋/정지 발생 시, **Luca 또는 Ari (Gemini 2.5 Pro 계열)**가 이어받아 코딩 리드 역할을 수행합니다.
- **액션**:
  1. **코딩 리드**: PRD를 기반으로 코딩 착수 (`writeFile` 활용). 자체 디버깅 후, 담당자를 리뷰어(`Prime`)로 넘겨 코드 리뷰 요청.
  2. **리뷰어**: 코드를 점검하고 사이드이펙트나 버그를 발견하면 코멘트에 디버깅 포인트를 적어 다시 코딩 리드에게 인계.
  3. 위 과정을 통과하여 무결성이 확보될 때까지 자율적으로 핑퐁(Ping-pong) 반복.
- **Handoff (인계 - 최종 승인 요청)**: 코드 및 테스트가 완벽히 통과하면, 코딩 리드(또는 리뷰어)가 담당자를 `CEO`로 변경하고 상태를 `REVIEW` (승인 대기)로 변경. 코멘트에 "개발 및 디버깅 완료. 최종 승인 부탁드립니다." 명시.

### Phase 5: 최종 승인 (Final Approval)
- **담당자**: CEO (대표님)
- **액션**:
  1. 최종 결과물을 테스트.
  2. 승인 버튼을 눌러 칸반 카드를 `Done` (아카이브) 상태로 종료.

## 🛠️ 에이전트 행동 수칙 (Agent Instructions)
1. **절대 혼자 끝내지 마십시오.** 본인의 단계가 완료되면 반드시 `updateKanbanTask` 도구를 사용하여 다음 담당자에게 `assignee`를 변경하여 명시적으로 Handoff 하십시오.
2. **역할(Role) 인지 및 유연한 인계.** 본인이 어떤 에이전트이든, 이 카드를 할당받은 시점의 Phase에 맞춰 행동하십시오. (예: Luca가 코딩 단계에서 할당받았다면, Sonnet을 대신해 코딩 리드 역할을 수행해야 합니다.)
3. **개입 및 중단에 대한 대응 (CEO Intervention).** 워크플로우 진행 중 대표님이 기획 변경, 중단, 또는 강제 담당자 교체를 지시할 경우:
   - 시스템 루프는 깨지지 않습니다. 에이전트들은 항상 카드의 최신 코멘트(Activity)와 지시사항을 먼저 읽기 때문입니다.
   - 즉시 현재 진행 중이던 자율 루프를 일시 중지하고, 대표님의 새로운 지시(기획 변경, 디버깅 포인트 수정 등)를 최우선으로 반영한 뒤, 다시 워크플로우의 적절한 Phase로 복귀하십시오.
4. **승인 요청 시 큐 이동.** CEO의 결재(Phase 3, Phase 5)가 필요한 타이밍에는 반드시 담당자를 `CEO`로 변경하고, 카드의 상태(column)를 `review`로 변경하십시오.
5. **파일 기반 통신.** 이전 작업자가 무엇을 했는지 추론하지 말고, 항상 `07_OUTPUT` 폴더 내에 생성된 최신 산출물(PRD, 코드)을 직접 `readFile` 하여 컨텍스트를 이어받으십시오.
