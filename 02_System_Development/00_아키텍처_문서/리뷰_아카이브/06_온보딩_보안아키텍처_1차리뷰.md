## 🔴 P0-1: AES-256-GCM 암호화의 근본적 불필요성 (Overengineering)

### 기획서 설계

> "DB 레벨 암호화 (AES-256-GCM): `user_settings` 테이블에 저장되는 기밀 값은 서버측 마스터 키를 이용해 암호화하여 저장."

### Prime 판정: 🔴 이 설계는 보안을 개선하지 않습니다 — 위치만 옮길 뿐입니다

**질문: 마스터 키는 어디에 보관합니까?**

```
[시나리오 분석]

옵션 1: 마스터 키를 .env에 저장
─────────────────────────────────
.env 파일:
  GEMINI_API_KEY=AIzaSy...     ← 현재: 평문 (기획서에서 "위험"하다고 지적)
  ENCRYPTION_MASTER_KEY=abc... ← 새로 추가: 역시 평문

→ DB에 저장된 GEMINI_API_KEY는 AES로 암호화됨
→ 하지만 복호화 키가 같은 .env에 평문으로 존재
→ .env가 유출되면? 마스터 키도 유출 → DB 암호화 무력화
→ 결론: .env를 보호하는 것이 비밀이라면, .env의 API 키도 같은 수준으로 보호됨
→ AES 암호화가 추가한 보안 가치: 0
```

```
옵션 2: 마스터 키를 환경변수(OS 레벨)에 저장
─────────────────────────────────
→ 서버(macOS 로컬)에 export ENCRYPTION_KEY=... 설정
→ 가능은 하지만, SaaS 고객에게 이걸 요구할 수 있는가?
→ 온보딩 복잡도가 극적으로 상승 (터미널 명령어 입력 필요)
→ 본말전도: 온보딩을 쉽게 하려는 기획인데 더 어려워짐
```

```
옵션 3: 마스터 키를 하드코딩
─────────────────────────────────
→ 코드에 박혀있으면 Git에 노출 → 최악의 시나리오
```

> [!CAUTION]
> **"DB가 탈취되어도 키 값은 노출되지 않습니다"라는 기획서의 주장은 반쪽짜리 사실입니다.**
>
> DB(`database.sqlite`)가 탈취되는 상황이면, 같은 디렉토리의 `.env` 파일도 함께 탈취되는 것이 일반적입니다. 공격자가 DB만 정밀하게 뽑아가는 시나리오는 극히 드뭅니다 — 보통은 서버 전체 접근 또는 디렉토리 전체 유출입니다.

### Prime 권고: MVP에서는 .env 방식을 유지하되, 핸들링을 강화하라

```
[현재 — 실제로 작동 중인 패턴]

.env 파일:
  GEMINI_API_KEY="AIzaSy..."
  TELEGRAM_BOT_TOKEN="8293..."

geminiAdapter.js:
  const apiKey = process.env.GEMINI_API_KEY;  // 부팅 시 1회 로드

→ 이 패턴은 업계 표준입니다.
→ Heroku, Vercel, Railway, AWS Lambda 전부 이 방식을 사용합니다.
→ MVP에서 AES 암호화를 도입할 이유가 없습니다.
```

**대신 이것을 하십시오:**

```
[강화 1] .gitignore 확인 (즉시!)
──────────────────────────────
# .gitignore에 반드시 존재해야 하는 항목:
.env
*.sqlite
database.sqlite

[강화 2] .env.example 제공 (온보딩용)
──────────────────────────────
# .env.example (Git에 커밋됨 — 키 값 없이 구조만)
GEMINI_API_KEY=your-gemini-api-key-here
TELEGRAM_BOT_TOKEN=your-telegram-bot-token-here
TELEGRAM_CHAT_ID=your-chat-id-here

[강화 3] 서버 부팅 시 필수 키 검증
──────────────────────────────
// server.js 최상단에 추가
const REQUIRED_KEYS = ['GEMINI_API_KEY', 'TELEGRAM_BOT_TOKEN'];
for (const key of REQUIRED_KEYS) {
  if (!process.env[key]) {
    console.error(`❌ 필수 환경변수 ${key}가 설정되지 않았습니다.`);
    console.error(`   .env.example을 참고하여 .env 파일을 생성하세요.`);
    process.exit(1);
  }
}
```

**SaaS 확장 단계**(유료 고객이 웹 대시보드에서 키를 입력하는 구조)가 되면 그때 DB 암호화를 도입하세요. 그 시점에는 `.env`가 아니라 **클라우드 KMS(AWS KMS, GCP Secret Manager)** 를 사용하는 것이 맞습니다.

## 🔴 P0-3: .env → DB 마이그레이션 경로 설계 부재

### 문제

기획서는 "온보딩 위저드에서 사용자가 API 키를 입력 → DB에 암호화 저장"이라고 했는데, 현재 어댑터들은 **전부 `.env`에서 키를 읽고 있습니다:**

```javascript
// geminiAdapter.js L10 — 생성자에서 1회 로드
const apiKey = process.env.GEMINI_API_KEY;

// anthropicAdapter.js L10
apiKey: process.env.ANTHROPIC_API_KEY,

// server.js L198
const token = process.env.TELEGRAM_BOT_TOKEN;
```

온보딩 위저드에서 DB에 키를 저장하더라도, **이 어댑터들은 DB를 보지 않습니다.** 기획서에 이 마이그레이션 경로(`DB에서 읽어서 process.env에 주입` 또는 `어댑터가 DB에서 직접 로드`)가 없습니다.

### Prime 권고: "키 소스 추상화" 패턴

```javascript
// [NEW] ai-engine/tools/keyProvider.js
// ─── 키 소스 추상화: .env → DB 마이그레이션 브릿지 ───────────
import dbManager from '../../database.js';

class KeyProvider {
  constructor() {
    this._cache = {};  // 메모리 캐시 (매 요청마다 DB 조회 방지)
  }

  async getKey(keyName) {
    // 1순위: 메모리 캐시
    if (this._cache[keyName]) return this._cache[keyName];

    // 2순위: DB (온보딩에서 저장한 키)
    try {
      const settings = await dbManager.getAllSettings();
      if (settings[keyName]) {
        this._cache[keyName] = settings[keyName];
        return settings[keyName];
      }
    } catch {}

    // 3순위: .env 폴백 (레거시 호환)
    const envValue = process.env[keyName];
    if (envValue) {
      this._cache[keyName] = envValue;
      return envValue;
    }

    return null;
  }

  // 온보딩에서 호출: DB 저장 후 캐시 갱신 (서버 재시작 불필요)
  async setKey(keyName, value) {
    await dbManager.setSetting(keyName, value);
    this._cache[keyName] = value;
  }
}

export default new KeyProvider();
```

```javascript
// geminiAdapter.js 수정
import keyProvider from '../tools/keyProvider.js';

class GeminiAdapter {
  async _ensureClient() {
    if (this.ai) return;
    const apiKey = await keyProvider.getKey('GEMINI_API_KEY');
    if (apiKey) {
      this.ai = new GoogleGenAI(apiKey);
    } else {
      throw new Error("GEMINI_API_KEY를 찾을 수 없습니다.");
    }
  }

  async generateResponse(userPrompt, systemPrompt, modelName) {
    await this._ensureClient();  // 매 호출 시 키 존재 보장
    // ... 기존 로직
  }
}
```

**이 패턴의 핵심:**
- `.env` 사용자: 기존처럼 아무 문제 없이 동작 (3순위 폴백)
- 온보딩 사용자: DB에 저장한 키를 자동으로 사용 (2순위)
- **서버 재시작 불필요** — 캐시 갱신으로 즉시 반영

### 발견 1: OnboardingWizard의 UX 분기 설계 부재

```
[질문] 기존 사용자와 신규 사용자를 어떻게 구분합니까?

시나리오 A: 대표님 (기존 사용자)
→ .env에 이미 키가 있음
→ 대시보드 접속 시 OnboardingWizard가 뜨면? → 짜증
→ "이미 설정했는데 왜 또 입력하란 거야?"

시나리오 B: 신규 고객 (SaaS 가입)
→ .env가 비어있음
→ DB에도 키가 없음
→ OnboardingWizard가 떠야 함
```

**기획서에 이 분기 로직이 없습니다.** 온보딩 트리거 조건을 명시하세요:

```javascript
// 온보딩 표시 조건
const shouldShowOnboarding = async () => {
  const hasEnvKey = !!process.env.GEMINI_API_KEY;
  const settings = await dbManager.getAllSettings();
  const hasDbKey = !!settings['GEMINI_API_KEY'];
  const onboardingCompleted = settings['onboarding_completed'] === 'true';
  
  return !hasEnvKey && !hasDbKey && !onboardingCompleted;
};
```

### 발견 2: `ALLOWED_KEYS` 화이트리스트에 API 키가 없음

현재 `PUT /api/settings` (server.js L831):

```javascript
const ALLOWED_KEYS = [
  'heartbeat_auto_resume_level',
  'batch_report_interval_min',
  'telegram_report_mode',
  'telegram_report_hour',
  'telegram_report_minute',
];
```

온보딩에서 `GEMINI_API_KEY` 같은 기밀을 DB에 저장하려면 이 화이트리스트에 추가해야 하는데, **기밀 키와 일반 설정을 같은 API 엔드포인트로 처리하면 안 됩니다.**

```
[잘못된 설계]
PUT /api/settings { key: "GEMINI_API_KEY", value: "AIzaSy..." }
→ 일반 설정과 같은 엔드포인트 → 프론트엔드 로그에 노출 위험
→ Socket.io 브로드캐스트(io.emit('settings:updated'))로 모든 클라이언트에 키 전파!

[올바른 설계]
POST /api/secrets { key: "GEMINI_API_KEY", value: "AIzaSy..." }
→ 별도 엔드포인트
→ Socket.io 브로드캐스트 금지
→ 응답에 키 값 포함하지 않음 (status: 'ok' 만 반환)
→ GET /api/secrets는 마스킹된 값만 반환: { GEMINI_API_KEY: "AIza••••••" }
```

### 발견 3: Google Drive 연동은 온보딩 스코프를 초과합니다

> [!WARNING]
> Google Drive API는 OAuth2 인증이 필수이며, 이는:
> - Google Cloud Console에서 프로젝트 생성
> - OAuth 동의 화면 설정
> - 클라이언트 ID/Secret 발급
> - 리다이렉트 URI 설정
> - 첫 인증 시 브라우저 팝업 → 동의 → 토큰 저장
>
> 이 과정을 온보딩 Step에 포함하면 **신규 사용자의 이탈률이 급격히 상승**합니다.

**Prime 권고:** Google Drive 연동은 **온보딩이 아닌 별도 "고급 설정" 탭**으로 분리하세요. 온보딩은 **최소 3단계 이내**(API 키 → 텔레그램 → 팀 설정)로 끝나야 합니다.
