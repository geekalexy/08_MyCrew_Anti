## 3. 루카와 소넷의 본질적 차이

### 🟡 루카의 사고 방식
> **"지금 당장 실행 가능한가?"**

- 모든 결과물이 **즉시 발동 가능한 명령어**로 수렴
- 문서보다 **시스템 설계**에 집중
- 사용자 심리를 **직접 자극**하는 날카로운 카피
- 크루 역할을 **구체적 파일 경로 + API 연동**까지 설계
- 문장이 짧고 결론이 앞에 온다

### 🔵 소넷의 사고 방식
> **"체계적으로 완전한가?"**

- 모든 결과물이 **참조 가능한 문서**로 수렴
- 명령어보다 **분류·관리 체계**에 집중
- 사용자 심리를 **단계적으로 설득**하는 구조
- 크루 역할을 **번호 스킬 체계 + 우선순위**로 정리
- 문장이 길고 근거가 뒤따른다

## 4. 영역별 최종 채택 결정

### 4-1. 컨텍스트 파일

```
채택: 소넷 기반 (데이터 완전성 우선)
보강: 루카의 브랜드 미션 한 문장 → 도입부 삽입
보강: 루카의 압축형 요약 → 문서 최상단 TL;DR 섹션 추가
```

### 4-2. 채팅 시나리오

```
채택: 루카의 핵심 1케이스 (강렬한 첫 멘트 구조)
  → "빈손으로 시키면 뻔한 AI 글" 수준의 임팩트 유지
보강: 소넷의 3케이스 (거부·재방문 상황 추가)
보강: 루카의 완료 후 "자동 저장+재이용" 메시지 채택
보강: 소넷의 분석 결과 공유 채팅 추가
```

### 4-3. 카드 UX

```
채택: 루카의 "10초 셋업" 헤더 카피
채택: 루카의 폴더 체크박스 기본 ON
채택: 루카의 사이드바 핀(Pin) 유도 CTA → 최우선 추가
보강: 소넷의 충족도 게이지 ████ 85%
보강: 소넷의 업데이트 주기 선택 (섹션 D)
```

### 4-4. 워크플로우

```
채택: 소넷의 Day 0~7 절대 타임라인 (개발 명확성)
채택: 루카의 SSOT 명칭 공식 도입

보강: 소넷의 업데이트 루프 (2주 주기 알림)

### 4-5. 스킬 체계

```

## 5. 소넷의 자기 평가

**솔직하게 말씀드리면:**

루카는 저보다 **실행 지향적**이었습니다.

저는 "완전한 문서"를 만들었고,
루카는 "바로 작동하는 시스템"을 설계했습니다.

특히 세 가지는 제가 생각하지 못한 것들입니다:
1. **VOC 카피 추출** — 실제 고객 언어를 카피에 쓴다는 발상
2. **사이드바 핀 유도** — 접근성이 습관을 만든다는 통찰
3. **폴더 자동 생성 타이밍** — 사용자 행동 없이 SSOT가 구축되는 구조

반면 제가 더 잘한 것:
1. **데이터 완성도** — 가격·기능·URL 전수 기록
2. **케이스 대응** — 다양한 상황 커버
3. **업데이트 루프** — 장기적 싱크 유지 설계
